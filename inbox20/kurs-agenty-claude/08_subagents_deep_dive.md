# Глубокое погружение в субагенты

> Из курса по работе с Claude Code. Дополнения от Claude помечены значком 💡.

---

## Что такое субагенты и зачем они нужны

**Субагент** — это отдельный «дочерний» Claude Code, которого основной агент запускает в изолированной среде. Каждый субагент:
- Имеет **собственный чистый контекст** (не загружен историей основного диалога).
- Получает **только свою инструкцию** (свой `markdown`-файл).
- Выполняет **одну конкретную задачу**.
- Возвращает результат основному агенту.

> 💡 **Дополнение от Claude:** представьте, что главный агент — это руководитель, а субагенты — его сотрудники. Руководитель не делает задачу сам, он раздаёт её специалистам, они работают параллельно (или последовательно), каждый в своём «офисе» (контексте), а потом приносят результат. Руководитель собирает результаты в единый отчёт. Так вы не сжигаете контекст главного диалога на детали — там остаётся только финальная сводка.

---

## Какую проблему решают субагенты

Раньше, когда субагентов не было, единственный способ работать над сложной задачей — открывать **много параллельных диалогов**:

- Один диалог на промпт-дизайнера.
- Второй на копирайтера.
- Третий на ресёрчера.
- И так далее.

Каждый диалог нужно было запускать руками, копировать результат из одного, вставлять в другой — много времени и ручной работы.

Если попытаться сделать всё в **одном** диалоге — контекст переполняется, агент тупит, качество ответа падает.

**Субагенты решают эту проблему** — главный агент сам, без вашего участия, открывает «дочерние диалоги» и распределяет задачи.

## Где живут субагенты

Файлы субагентов лежат в папке:
```
.claude/agents/         (локально для проекта)
~/.claude/agents/       (глобально)
```

Каждый субагент — это **markdown-файл** с инструкцией. Структура: точно такой же `frontmatter` (YAML в начале), как у скилов, плюс инструкция в теле файла.

### Пример структуры файла субагента

```markdown
---
name: scraper
description: Парсит указанные сайты и возвращает структурированные данные.
tools: WebFetch, Read, Write, Bash
model: claude-sonnet-4-6
color: yellow
---

# Инструкция парсера

Ты — парсер сайтов. Твоя задача:
1. Получить URL от главного агента.
2. Открыть сайт, извлечь нужные данные.
3. Сохранить результат в /agent-runtime/shared/scraped.json
4. Вернуть путь к файлу.
```

Поля frontmatter:

| Поле | Зачем |
|---|---|
| `name` | Имя субагента (обязательно) |
| `description` | По нему главный агент понимает, когда нужен этот субагент |
| `tools` | Список разрешённых инструментов |
| `model` | Какую модель использовать (например, `claude-opus-4-6` или `claude-sonnet-4-6`) |
| `color` | Цвет в терминале для удобной навигации между субагентами |

> 💡 **Дополнение от Claude:** правильно указывать `model` под характер задачи. Аналитический ресёрч → Opus. Простой парсинг или написание шаблонного кода → Sonnet (быстрее и дешевле). Очень простые проверки → Haiku. Если у всех субагентов стоит Opus — будете жечь лимиты впустую.

## Как работают субагенты на практике

### Сценарий 1. Большая разработка, разбитая на чанки

Допустим, у вас огромный проект — 12 этапов («чанков») разработки. Каждый чанк — это десятки задач: создание структуры папок, бэкенд, авторизация, фронтенд, БД, тесты, деплой и так далее.

**Если делать всё в одном диалоге:**
- Чтобы загрузить план разработки, агент сожжёт огромный объём токенов только на чтение.
- После пары чанков контекст переполнится.
- Агент начнёт «забывать» предыдущие задачи и тупить.
- Качество кода резко упадёт.

**Если использовать субагентов:**
1. Главный агент читает план.
2. На первый чанк он спавнит одного или нескольких субагентов параллельно.
3. Каждый субагент выполняет свою часть с **чистым контекстом**.
4. Возвращает результат.
5. Главный агент создаёт `/clear`, переходит ко второму чанку.
6. Так до 12-го чанка.

> 💡 **Дополнение от Claude:** на больших проектах субагенты — это не «модно», это необходимость. Без них вы упрётесь в потолок контекста за пару часов работы. С ними можно вести разработку днями, потому что главный диалог хранит только сводки.

### Сценарий 2. Агентная команда (например, мониторинг новостей)

Пример из курса — пайплайн из четырёх агентов:

| Субагент | Задача |
|---|---|
| **Coordinator** | Тимлид, раздаёт задачи остальным, проверяет результаты |
| **Scraper** | Парсит новостные сайты, собирает сырые данные |
| **Analyst** | Анализирует собранные данные, выделяет важное |
| **Reporter** | Формирует итоговый отчёт (в PDF, Excel и т. д.) |

Когда вы пишете главному агенту: «запусти пайплайн новостей», он:
1. Запускает **Scraper** в отдельной сессии. Тот парсит сайты, сохраняет результат.
2. Запускает **Analyst**. Тот читает результат скрейпера, анализирует.
3. Запускает **Reporter**. Тот создаёт PDF-отчёт.
4. Передаёт всё **Coordinator**, который проверяет качество и завершает.

Каждый субагент работает в своей изолированной среде — никаких лишних токенов.

## Как главный агент знает, кого вызывать (оркестрация)

Над субагентами **обязательно** должна быть оркестрация — описание, **кто и в каком порядке** запускается. Есть два уровня:

### Уровень 1. CLAUDE.md

В корневом `CLAUDE.md` проекта пишется:
- Какие субагенты есть в проекте.
- Какой агент за что отвечает.
- В каком порядке их запускать.
- Какие данные между ними передаются.

### Уровень 2. Отдельный файл pipeline.md (опционально)

Если оркестрация сложная, можно вынести её в отдельный файл (например, `pipeline.md`). В нём описывается полный workflow, обмен данными, последовательность вызовов. `CLAUDE.md` тогда просто ссылается: «смотри детали в pipeline.md».

### Как выглядит оркестрация в CLAUDE.md (пример)

```markdown
## Pipeline

1. coordinator — принимает запрос от пользователя
2. scraper — парсит сайты из списка urls.txt
3. analyst — анализирует данные scraper'а
4. reporter — формирует PDF/Excel
5. coordinator — финальная проверка и завершение

## Маршрутизация запросов

- "сделай ресёрч по теме X" → запустить deep-researcher
- "спарси сайт Y" → запустить scraper
- "новости за неделю" → запустить news-digest
- "проанализируй документ" → запустить doc-analyzer
- "сделай отчёт" → запустить report-generator
```

> 💡 **Дополнение от Claude:** инструкции для главного агента и субагентов пишите **по принципу Chain of Thought**: пошагово, как алгоритм. «Сначала классифицируй запрос → определи, нужна ли цепочка → передай агенту такие-то входные данные → получи такой-то результат → передай следующему». Без этого Claude может «креативить» и менять порядок, что плохо для воспроизводимости.

## Как переключаться между субагентами в терминале

Когда главный агент запустил несколько субагентов, в терминальном Claude Code они отображаются как параллельные «вкладки».

**Управление:**
- **Стрелка вниз** — выбор субагента.
- **Enter** — переключиться в его сессию.
- **Стрелка вверх** + Enter, чтобы вернуться к main.

В каждом субагенте видна вся его «переписка» — что он делает, какие инструменты вызывает, какие данные у него внутри.

> 💡 **Дополнение от Claude:** **этой возможности нет в расширении Claude Code для VS Code** — там вы видите только то, что главный агент запустил субагента, и финальный результат. Под капотом ничего не видно. Это одна из главных причин, почему опытные пользователи работают именно в **терминале**, а не в расширении: видимость работы субагентов критична для отладки.

## Как ограничить субагент (поле `tools`)

В frontmatter можно перечислить, какие инструменты разрешены конкретному субагенту:

```yaml
tools: WebFetch, Read, Write, Bash
```

Например, **парсеру** не нужен инструмент `WebSearch` — у него и так есть конкретные URL. Аналитику не нужен `WebFetch` — он работает с уже скачанными данными.

Зачем ограничивать:
1. **Безопасность.** Субагент не сможет случайно сделать что-то вне своей зоны.
2. **Экономия токенов.** Меньше инструментов в контексте — меньше токенов на их описания.
3. **Предсказуемость.** Не будет «творчества» там, где не надо.

> 💡 **Дополнение от Claude:** есть смысл начать со списка `tools: Read, Write` (минимум) и добавлять по мере необходимости. Ставить «всё подряд» — антипаттерн.

## Когда субагенты НЕ нужны

Не для каждой задачи нужны субагенты. Не плодите их без необходимости — это лишняя сложность.

| Задача | Субагенты? |
|---|---|
| Простой вопрос, ответ за один промпт | Не нужны |
| Маленький скрипт, одна функция | Не нужны |
| Создание лендинга, простой сайт | Не нужны |
| Большая разработка, 10+ этапов | **Нужны** |
| Регулярный мониторинг (новости, сайты, отчёты) | **Нужны** |
| Многоэтапная аналитика разных типов данных | **Нужны** |
| Нужна параллельная работа над разными частями | **Нужны** |

> 💡 **Дополнение от Claude:** правило большого пальца: **если задача укладывается в один логичный промпт и один диалог — субагенты не нужны**. Создавайте их, когда понимаете, что без них контекст переполнится или нужна параллельная работа.

## Чек-лист создания субагента

1. Создайте файл в `.claude/agents/имя-субагента.md` (для проекта) или `~/.claude/agents/` (глобально).
2. Напишите frontmatter:
   ```yaml
   ---
   name: <имя>
   description: <чёткое описание, когда вызывать>
   tools: <минимально необходимый набор>
   model: <claude-opus-4-6 или claude-sonnet-4-6>
   color: <цвет для навигации>
   ---
   ```
3. В теле файла — пошаговая инструкция (Chain of Thought).
4. Чётко опишите **входные данные** (что передаёт главный агент) и **выходные** (что субагент возвращает).
5. Укажите **папки для обмена данными** (например, `agent-runtime/shared/`).
6. Не дублируйте функциональность — каждый субагент должен делать что-то одно.
7. Обновите главный `CLAUDE.md` — добавьте новый субагент в список и в правила маршрутизации.

> 💡 **Дополнение от Claude:** субагент работает по тем же принципам, что и обычный Claude Code — поэтому промпт-инжиниринг важен. Чем точнее описана задача в его инструкции, тем меньше «творчества» и тем лучше результат. Если субагент работает нестабильно — обычно проблема не в Claude, а в нечёткой инструкции.
