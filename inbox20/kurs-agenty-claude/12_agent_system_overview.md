# Готовая агентная система на базе Claude Code

> Из курса по работе с Claude Code. Дополнения от Claude помечены значком 💡.

---

## О чём эта инструкция

Это разбор реальной агентной системы, которую автор курса использует и раздаёт бесплатно через свой Telegram-канал. Это **не «hello world»**, а полнофункциональная система из 7+ субагентов, способная делать ресёрч, парсить сайты, анализировать документы, генерировать отчёты в PDF/Excel/HTML и многое другое.

Цель этой инструкции — показать, как устроена грамотная агентная система, чтобы вы могли:
1. Использовать готовую систему автора (если получите её).
2. Построить **свою** систему по тем же принципам.

> 💡 **Дополнение от Claude:** обратите внимание на архитектурные решения здесь. Это не «прикрутил одного агента — работает». Это продуманная архитектура с разделением ответственности, маршрутизатором, общей папкой обмена данными и предсказуемым поведением. Именно так должна выглядеть production-ready система.

## Список агентов и их роли

### Router (главный агент)

- **Роль:** тимлид. Не выполняет задачи сам, а **маршрутизирует** запросы пользователя на нужного субагента.
- **Использует:** `CLAUDE.md` с правилами маршрутизации.
- **Выбор модели:** Opus 4.6 — он принимает решения, ему нужна способность к рассуждению.

> 💡 **Дополнение от Claude:** концепция «router-агент» — стандартный паттерн в production-системах. Главный агент должен заниматься только координацией. Если он и парсит, и анализирует, и пишет отчёты — это плохо проектированная система. Чёткое разделение ролей делает систему предсказуемой и тестируемой.

### Deep Researcher

- **Роль:** глубокий ресёрч по теме.
- **Триггеры:** «исследуй», «найди информацию по», «сравни X и Y», «плюсы и минусы».
- **Что делает:** ищет в интернете, собирает данные из множества источников, синтезирует их в подробный отчёт.

### Parser

- **Роль:** парсинг конкретных сайтов.
- **Триггеры:** «спарси сайт», «собери данные с».
- **Использует:** Chrome DevTools MCP или скил `agent-browser`.
- **Возвращает:** структурированные данные в `agent-runtime/shared/scraped_data.json`.

### News Digest

- **Роль:** регулярный мониторинг новостей по заданным темам.
- **Триггеры:** «новости за», «что нового по теме», «дайджест».
- **Использует:** RSS-MCP, поиск в интернете, парсер.

### Doc Analyzer

- **Роль:** анализ документов любого формата.
- **Триггеры:** «проанализируй документ», «выжимка из», ссылка на файл.
- **Поддерживает:** Word, PDF, CSV, Excel.
- **Если документ огромный:** разбивает анализ на под-субагентов.

### Report Generator

- **Роль:** формирование финальных отчётов.
- **Форматы:** Markdown, HTML, PDF, Excel, Google Sheets, Word, PowerPoint.
- **Использует:** скилы `pdf`, `docx`, `xlsx`, `pptx`, MCP Google Sheets.

### Meeting Notes

- **Роль:** конспект записи встречи (Zoom, Google Meet, Yandex Telemost и т. д.).
- **Триггеры:** загружена аудио/видеозапись.
- **Использует:** Whisper (локально или Supadata API) для транскрипции.

## Как работает маршрутизация

Когда пользователь пишет в Claude Code, **сначала Router читает CLAUDE.md**. В нём прописана таблица маршрутизации с ключевыми словами и соответствующими субагентами.

Пример раздела маршрутизации в CLAUDE.md:

```markdown
## Маршрутизация запросов

### Прямой вызов (если пользователь явно указал агента)

Если пользователь пишет «запусти агента parser» — Router передаёт задачу напрямую parser'у.

### Через ключевые слова

| Если в запросе есть | Запустить |
|---|---|
| "ресёрч", "найди", "сравни", "следуй" | deep-researcher |
| "собери с сайта", "спарси", "проанализируй сайт" | parser |
| "новости", "что нового", "дайджест" | news-digest |
| "проанализируй документ", "выжимка из" | doc-analyzer |
| "сделай отчёт", "PDF с", "Excel с" | report-generator |
| "запись встречи", "конспект zoom" | meeting-notes |

### Цепочки (если задача требует нескольких агентов)

| Запрос | Цепочка |
|---|---|
| "спарси и сделай отчёт" | parser → report-generator |
| "найди новости и сделай PDF" | news-digest → report-generator |
| "проанализируй документ и презентацию из него" | doc-analyzer → report-generator |
```

> 💡 **Дополнение от Claude:** ключевая мысль: **Router не «думает» сам**. Он **сверяется с таблицей** в CLAUDE.md. Это делает поведение предсказуемым. Если хотите изменить маршрутизацию — правьте таблицу, не пытайтесь переучить агента «здравым смыслом».

## Папка agent-runtime — обмен данными

Это критически важный архитектурный элемент. Все промежуточные и финальные данные кладутся в две папки:

### outputs/ — финальные отчёты

Здесь финальные результаты, которые увидит пользователь:
- `report.md` — Markdown-версия.
- `report.html` — для браузера.
- `report.pdf` — для печати/отправки.
- `data.xlsx` — таблицы.
- `presentation.pptx` — презентации.

### shared/ — данные между агентами

Промежуточные JSON-файлы для передачи данных между субагентами:

```json
{
  "scraped_at": "2026-04-29T10:00:00Z",
  "source": "banki.ru",
  "data": {
    "USD": 79.83,
    "EUR": 84.12,
    "GBP": 96.45
  }
}
```

Один субагент сохраняет JSON, второй его читает. **Никаких передач через копирование текста в чате** — всё через файлы.

> 💡 **Дополнение от Claude:** обмен через файлы — пуленепробиваемый. Никаких потерь данных, никаких ограничений по размеру, всё прозрачно (можно открыть и проверить руками). Если бы агенты обменивались через текст в диалоге — данные бы съедались контекстом и обрезались. Через файлы — нет ограничений.

## Пример сценария работы

### Промпт пользователя

В Claude Code пользователь пишет:
```
Нас интересует парсинг курсов валют по доллару, евро и британскому фунту, а также интересуют котировки по основным акциям на московской бирже, по основным российским компаниям и индексам.

https://banki.ru
https://www.moex.com

Запусти специального агента, который сделает ресёрч по этим сайтам и сделает нам отчёт.
```

### Что происходит

1. **Router** читает запрос.
2. Видит триггеры «парсинг», «сайты» и слово «отчёт».
3. Понимает, что нужна **цепочка**: `parser → report-generator`.
4. **Запускает Parser** с задачей: спарси указанные URL по этим темам.
5. Parser идёт на сайты, собирает данные, кладёт результат в `agent-runtime/shared/scraped_data.json`.
6. **Запускает Report Generator** с задачей: возьми данные из shared, сделай отчёт в Markdown + HTML.
7. Report Generator читает JSON, создаёт `outputs/report.md` и `outputs/report.html`.
8. Router передаёт пользователю: «Готово, отчёт здесь: [пути к файлам]».

Время работы: примерно 30 секунд.

### Дальнейший диалог

```
Теперь сделай отчёт ещё в PDF и в Google-таблице.
```

Router запускает **два параллельных субагента** Report Generator:
1. Один с задачей PDF (через скил `pdf`).
2. Второй с задачей Google Sheets (через MCP Google Sheets).

Они работают независимо, оба возвращают результат в `outputs/`.

> 💡 **Дополнение от Claude:** **главное в архитектуре** — сначала продумываются «прямые вызовы» (один агент = одна задача), потом «цепочки» (последовательные), потом «параллельные» (одновременные). Не начинайте с параллельных — они вкусные, но их сложно отлаживать.

## Как создавать ваших собственных субагентов

### Метод 1. Попросить Claude Code

Самый простой способ:
1. Запустить новый чистый диалог в Claude Code.
2. Описать задачу, которую должен решать новый субагент.
3. Указать, какие инструменты ему доступны (или сказать «сам реши»).
4. Указать, как он должен встраиваться в существующий пайплайн.

**Пример промпта:**
```
Создай нам ещё одного субагента. Его задача — скачивать видео с YouTube, транскрибировать его в текст и делать краткую выжимку.

Должно быть несколько важных инсайтов из видео, ключевые выводы, краткий итог.

Также этот агент должен быть встроен в нашу агентную систему так, чтобы после него можно было запускать report-generator (для отчётов по этому видео).
```

Claude Code:
1. Создаст файл `.claude/agents/youtube-analyzer.md`.
2. Установит нужные CLI-инструменты (например, `yt-dlp`, `whisper`, `ffmpeg`).
3. Обновит `CLAUDE.md` — добавит маршрутизацию.
4. Объяснит, что добавил.

### Метод 2. Написать вручную

Создаёте файл `.claude/agents/имя-агента.md` с frontmatter и инструкцией. Подробнее — в инструкции «Глубокое погружение в субагенты».

> 💡 **Дополнение от Claude:** метод 1 быстрее, но **обязательно проверяйте**, что Claude Code сделал. Откройте созданный `.md`-файл, прочитайте инструкцию, поправьте если что-то не так. Иногда Claude добавляет лишние шаги или, наоборот, недопонимает задачу.

## Принципы хорошей агентной системы

### 1. Один субагент = одна роль

Не делайте «универсального помощника». Чем уже специализация, тем стабильнее работа.

### 2. Чем проще инструкция, тем лучше

В курсе подчёркнуто: **инструкции должны быть простыми, но конкретными**. Сложные → больше точек отказа. Слишком общие → агент «творит» вместо работы.

### 3. Обмен данными через файлы

Не пытайтесь передавать данные через текст между субагентами. Только через файлы (JSON, CSV, Markdown).

### 4. Используйте Chain of Thought

Пишите инструкции как алгоритм:
```
1. Получи URL от Router.
2. Открой сайт через Chrome MCP.
3. Извлеки нужные блоки.
4. Сохрани JSON в shared/.
5. Верни Router путь к файлу.
```

### 5. Подбирайте модель под роль

- Решает, что делать → Opus.
- Просто исполняет → Sonnet.
- Простые проверки → Haiku.

### 6. Логируйте

В каждом субагенте можно прописать «оставь короткий лог в `agent-runtime/logs/имя-агента.log`». Это бесценно для отладки.

> 💡 **Дополнение от Claude:** к рекомендациям курса добавлю ещё одну: **тестируйте каждого субагента отдельно**, прежде чем встраивать его в пайплайн. Если в изоляции работает плохо — в системе он будет работать ещё хуже. Не пытайтесь отлаживать всю систему сразу.
