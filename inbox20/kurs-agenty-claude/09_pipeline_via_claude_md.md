# Запуск пайплайна агентов через CLAUDE.md

> Из курса по работе с Claude Code. Дополнения от Claude помечены значком 💡.

---

## О чём эта инструкция

Это **практическая часть** работы с субагентами: как настроить главную инструкцию `CLAUDE.md` так, чтобы можно было одним промптом запустить целый пайплайн (цепочку) субагентов, и как переключаться между ними в процессе работы.

В качестве примера — пайплайн мониторинга новостей из четырёх агентов: координатор → парсер → аналитик → репортёр.

---

## Структура `CLAUDE.md` для агентного проекта

В файле `CLAUDE.md` для агентного проекта обязательно должны быть следующие блоки:

### 1. Описание проекта

Краткое — 1–2 предложения. Например:
```markdown
# News Intelligence Pipeline

Мультиагентная система для мониторинга и анализа новостей открытого интернета.
```

### 2. Список агентов

Перечень всех субагентов в проекте с указанием их роли:

```markdown
## Агенты в проекте

- **coordinator** — тимлид, оркестрирует пайплайн, проверяет результаты
- **scraper** — парсит указанные сайты, собирает сырые данные
- **analyst** — анализирует данные, выделяет важное
- **reporter** — формирует итоговый отчёт (PDF, Excel, Markdown)
```

### 3. Pipeline (последовательность)

Главное — **порядок запуска**:

```markdown
## Pipeline

1. coordinator (приём задачи)
2. scraper → собирает данные с сайтов
3. analyst → анализирует данные scraper'а
4. reporter → формирует отчёт
5. coordinator (финальная проверка, завершение)
```

### 4. Где обмениваются данными

Папки для передачи данных между агентами:

```markdown
## Папки

- `agent-runtime/shared/` — промежуточные данные между агентами (JSON)
- `agent-runtime/outputs/` — финальные отчёты для пользователя
```

### 5. Правила маршрутизации (если есть router-агент)

Если есть агент-маршрутизатор, который сам решает, кого вызывать:

```markdown
## Маршрутизация запросов

| Триггер в запросе пользователя | Какого агента запускать |
|---|---|
| "ресёрч", "найди", "сравни", "исследуй" | deep-researcher |
| "спарси сайт", "собери данные с" | scraper |
| "новости за", "что нового" | news-digest |
| "проанализируй документ", "выжимка из" | doc-analyzer |
| "сделай отчёт", "PDF с" | report-generator |
| "записи встречи", "конспект zoom" | meeting-notes |
```

> 💡 **Дополнение от Claude:** `CLAUDE.md` — это вход для главного агента в любой новый диалог. Чем чётче в нём описан пайплайн и правила маршрутизации, тем меньше Claude будет ошибаться в выборе субагента. Если видите, что Claude часто запускает не того агента — ужесточите правила маршрутизации с конкретными ключевыми словами.

---

## Дополнительный файл pipeline.md (опционально)

Если оркестрация сложная, можно вынести её в отдельный файл `pipeline.md`. Например, если у вас 8+ агентов, ветвления (если данные пустые → один путь, если есть → другой), сложные условия передачи данных.

В таком случае `CLAUDE.md` остаётся коротким, а `pipeline.md` содержит:
- Полный workflow с диаграммой.
- Все правила обмена данными.
- Условия и ветвления.
- Описание форматов данных в `shared/`.

> 💡 **Дополнение от Claude:** этот файл **необязательный**. Если у вас 3–5 агентов с линейным пайплайном — обходитесь одним `CLAUDE.md`. Дополнительные файлы тратят токены, потому что главный агент будет их подгружать. Создавайте `pipeline.md` только когда `CLAUDE.md` стал больше 200–300 строк.

## Запуск пайплайна — практический пример

### Шаг 1. Подготовка данных

В курсе автор использовал заранее спарсенные новости с Hacker News. Они уже лежали в Excel-файле в проекте, потому что были собраны до этого скилом `agent-browser`.

### Шаг 2. Промпт для главного агента

```
Запусти цепочку агентов на основе информации в Hacker News, которая уже скачана. И выдай отчёт.
```

Что важно в промпте:
1. **«Запусти цепочку агентов»** — главный агент сразу понимает, что нужен пайплайн, а не одиночный субагент.
2. **«На основе информации в Hacker News, которая уже скачана»** — указано, где входные данные.
3. **«И выдай отчёт»** — задана цель.

### Шаг 3. Выбор модели

Для запуска пайплайна автор использует **Opus 4.6 с max effort**. Это оправданно для координирующего агента — он принимает решения, что и кому делегировать. У самих субагентов в их frontmatter могут стоять более слабые модели.

### Шаг 4. Наблюдение за работой пайплайна

Главный агент:
1. Прочитал `CLAUDE.md`.
2. Понял пайплайн.
3. Запустил **первого субагента** (например, `reporter` сразу — если данные уже спарсены).
4. В терминале появилась новая «вкладка» — субагент Reporter работает.

В Claude Code снизу появятся имена всех активных субагентов. Можно стрелкой вниз переключиться на каждого и увидеть его внутренний процесс:
- Какие инструменты он вызывает.
- Какие файлы создаёт.
- Какие данные передаёт.

### Шаг 5. Передача результата дальше

Когда `Reporter` закончил, он передаёт результат **обратно в главного агента**. Главный агент видит отчёт, оценивает, и:
- Либо запускает следующего субагента (например, `coordinator` для финальной проверки).
- Либо финализирует и выдаёт результат пользователю.

### Шаг 6. Финал

В нашем примере главный агент после Reporter запустил Coordinator. Координатор проверил, что всё готово, и закрыл всех субагентов с сообщением «оба агента завершили работу, корректно выключились».

В папке `agent-runtime/outputs/` появился готовый PDF-отчёт «AI News Intelligence Report» с executive summary, ключевыми трендами, таблицей статей и рекомендациями.

## Переключение между субагентами в терминале

Когда несколько субагентов работают параллельно, **в терминальном Claude Code** появляется навигация:

| Действие | Что делает |
|---|---|
| **Стрелка вниз** | Перемещение по списку субагентов |
| **Enter** | Войти в выбранного субагента |
| **Цветовая подсветка** | Каждый субагент окрашен в свой цвет (зависит от поля `color` в frontmatter) |

Внутри каждого субагента вы видите его собственную «переписку» — как будто это отдельный диалог.

**В расширении Claude Code для VS Code** этой возможности нет. Вы увидите только финальный результат — вся внутренняя работа субагентов скрыта.

> 💡 **Дополнение от Claude:** **именно поэтому опытные пользователи работают в терминале**. Когда субагенты ошибаются, это критично — иметь возможность зайти к каждому и посмотреть, что у него «под капотом». В расширении вы видите только финальный результат и не понимаете, на каком шаге проблема.

## Использование скилов внутри субагентов

Субагенты могут использовать любые скилы, установленные в проекте. Например:

- **Reporter** использует скил `pdf` для создания PDF-отчётов.
- **Scraper** может использовать скил `agent-browser` для парсинга сайтов.
- **Analyst** может использовать скил `excel` для создания таблиц.

В описании субагента (`description` в frontmatter) можно явно указать, какие скилы он использует. Тогда главный агент при выборе агента учитывает доступные у него инструменты.

> 💡 **Дополнение от Claude:** **скилы имеют огромное преимущество в субагентах**: каждый субагент видит весь набор скилов, но в свой контекст подгружает только те, которые ему реально нужны. Это значит, что иметь 50 установленных скилов глобально — нормально и не «утяжеляет» каждого субагента.

## Минимальный шаблон агентного проекта

Структура папок для проекта с пайплайном:

```
my-agent-project/
├── CLAUDE.md                      ← главная инструкция, оркестрация
├── pipeline.md                    ← (опционально) детали workflow
├── .claude/
│   ├── agents/                    ← субагенты
│   │   ├── coordinator.md
│   │   ├── scraper.md
│   │   ├── analyst.md
│   │   └── reporter.md
│   ├── settings.json              ← настройки (deny-список, modes)
│   └── skills/                    ← локальные скилы (если есть)
└── agent-runtime/
    ├── shared/                    ← промежуточные JSON между агентами
    └── outputs/                   ← финальные отчёты для пользователя
```

> 💡 **Дополнение от Claude:** папку `agent-runtime/` обязательно добавьте в `.gitignore`, если используете git — там накапливаются временные данные и отчёты, которые незачем тащить в репозиторий. То же касается `node_modules/` (если есть JS) и любых файлов с секретами.

## Чек-лист запуска пайплайна

- [ ] В `CLAUDE.md` описаны все субагенты с их ролями.
- [ ] Указан порядок Pipeline.
- [ ] Описаны папки для обмена данными.
- [ ] Если есть router — описаны правила маршрутизации.
- [ ] Каждый субагент в `.claude/agents/` имеет frontmatter с `name`, `description`, `tools`, `model`.
- [ ] Subagent.md написан как Chain of Thought (пошагово).
- [ ] Запускаете пайплайн на Opus 4.6 (для главного агента).
- [ ] Работаете в **терминальном** Claude Code, а не в расширении (видимость субагентов).
- [ ] В `settings.json` правильно настроены permissions (особенно если Bypass Permissions).

## Запуск пайплайна без Agent Teams

Важный момент: **обычные субагенты работают без режима Agent Teams**. Они изолированы друг от друга и общаются только через главного агента.

Это означает, что **этой инструкции достаточно для большинства задач**. Полноценные Agent Teams (где субагенты обмениваются информацией напрямую друг с другом) нужны только для сложных кейсов — об этом отдельная инструкция.
