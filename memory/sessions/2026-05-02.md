## Сессия 6 (поздневечерняя — Anthropic Agent Skills, начали с Discovery)

Инна вернулась к решению из сессии 4 «Agent Skills пусть пока лежат» и предложила переосмыслить — установить эти скиллы так, чтобы команда брала их **для усиления**, не работала по ним напрямую. Скилл = дополнительный инструмент, наша С.С.С.Р. = основа.

### Терминологическое разъяснение

Прояснили на старте — два разных понятия путаются:
- **«Скиллы» в нашей команде** = `/brief`, `/discovery`, `/deploy`, `/vk-pack` и т.д. — 12 наших, в `.claude/skills/`, написаны под С.С.С.Р.
- **«Anthropic Agent Skills»** — сторонние из Marketplace (`skillsmp.com`). Три ключевых из обучения Дмитрия Ледовских: Discovery Interview (parcadei), Full-Stack Developer (shubhamsaboo), Front-End Design (Anthropic).

Технически файлы те же (`SKILL.md` в `.claude/skills/`), но смысл разный: внешние — чужая методология. Claude автоактивирует их по description, и наши агенты могут начать работать не по С.С.С.Р., а по чужому флоу.

### Архитектурный риск автоактивации

Anthropic Agent Skills работают по автоактивации. Если поставить Discovery Interview — он будет конкурировать за триггер с нашим `/discovery`. Гипотеза «скилл = инструмент усиления» технически не гарантируется механизмом автоактивации — её надо отдельно прописывать в CLAUDE.md и агент-файлах. Это меняет рамку обсуждения: «усиление» не достаётся бесплатно установкой.

### Решение Инны: сравнить и усилить, не подгружать

Вместо «установить параллельно» Инна предложила сравнить наш и внешний Discovery, найти сильные/слабые стороны каждого, и **усилить наш**. Если усиленный наш покрывает те же сценарии — внешний не нужен. Подход вписывается в наше «правило независимости агентов» из `learnings/README.md`.

### Сделано — детальное сравнение

- Прочитан полный текст внешнего скилла Discovery Interview (parcadei/Continuous-Claude-v3, 443 строки SKILL.md, скачан с raw.githubusercontent.com)
- Прочитан наш `/discovery` (147 строк, `.claude/skills/discovery/SKILL.md`)
- Сравнение по 4 осям: сильные/слабые каждого, что взять, что не брать.

**У нас сильнее:** адаптация под наш мир (Product Builder/Systems handoff, клиентский/наш проект), русский без жаргона, дружеский тон, размежевание `/brief` vs `/discovery`, минимум-защита (Блок В+Г обязательны).

**У внешнего сильнее:** Phase 1 — тип продукта; Knowledge Gap Detection (таблица сигналов); Conflict Resolution; Completeness Check; Iteration Rules (минимум 10-15 вопросов); адаптация под technical/non-technical/in-a-hurry; Research Loops; P0/P1/P2 приоритизация.

**Не подходит у внешнего:** английский + жаргон (Executive Summary, P0-P1-P2 — нарушает наше правило); категории Security/Deployment в полной форме (избыточно для наших ниш); технологический фокус (cloud/on-prem, WebSockets vs SSE — не наш стек); 8 категорий = 16-32 вопроса минимум (длинно); сохранение в `thoughts/shared/specs/` (чужой фреймворк); Phase 7 на drift prevention.

### План усиления — согласован Инной

**5 «зелёных» элементов (точно встраиваем):**
1. **Phase 1 — тип продукта** с привязкой к агентам команды для handoff: сайт→`websites`, бот→`systems`, AI-консультант→`ai-builder`, Mini App→`systems+websites`. Под каждый тип — свой набор обязательных блоков (визитка ≠ Mini App ≠ AI). Гибриды разрешены отдельным типом.
2. **Knowledge Gap Detection** — таблица сигналов под лексикон Инниного клиента («как у [конкурент]», «что-то модное», «ну такое») + готовые реплики. Лимит ловли ≤3 за интервью, чтобы не превратилось в допрос.
3. **Conflict Resolution** — список конфликтов под наши ниши («уникальный И как у бренда», «премиум-вид И эконом-цена», «быстро И с упаковкой»). Всегда 3 опции (X / Y / гибрид). Внутренняя проверка «реальный ли конфликт» перед вскрытием — чтобы не выдумывать дилеммы.
4. **Completeness Check** — проверка качества (размытое/декларативное/противоречивое подсвечивается), не только галочки. Лимит ≤2 уточнения на одно место — против бесконечного цикла.
5. **Iteration Rules** — адаптивный минимум по типу: визитка 8-12, лендинг 12-18, бот/Mini App/AI 18-25, SaaS 15-20. Считаем **закрытые** ответы, не заданные вопросы. Режим «турбо» при дефиците времени.

**2 «жёлтых» (открытый вопрос Инны):**
6. **Research Loops** через делегирование — 5 триггеров (нет конкурентов → `/competitor-research`, не уверен в рынке → `/market-research`, не знает ЦА → `audience-researcher`, ссылается на «как у X» без описания → ресёрч X, технологический выбор без понимания → `Systems`). Маркеры собираются **в конце Шага 2**, не прерывают поток. Если ресёрчей много — выдаём промежуточный результат вместо ТЗ.
7. **P0/P1/P2 → Обязательно/Желательно/Если останутся ресурсы** — только для MVP-функций (Блок В), не для всей спеки. Лимиты: ≤3 в «Желательно», ≤2 в «Если останутся». «Что НЕ входит в MVP» **остаётся** как отдельная сильная секция, третий уровень не сглаживает её.

### Архитектурное решение — Progressive Disclosure

Стандартный механизм Anthropic Agent Skills. Чтобы SKILL.md не разросся до 400+ строк (риск производительности кэша), выносим справочные таблицы в отдельные файлы рядом:

```
.claude/skills/discovery/
├── SKILL.md           ← инструкции, ~200 строк
├── product-types.md   ← типы + блоки под каждый
├── gap-signals.md     ← сигналы + готовые реплики
├── conflicts.md       ← конфликты под наши ниши
└── completeness.md    ← чек-лист + правила качества
```

SKILL.md ссылается на них, Claude подгружает по необходимости.
