#### 3. Архитектурный разговор с Инной — три ключевых развилки

3.1. **`/brief` vs `/discovery` — по типу задачи или по глубине?** Прочитали `/brief` SKILL.md и обнаружили внутреннее противоречие: антитриггер «не для цифровых продуктов» vs Шаг 1 предлагает тип «Система (бот, CRM, автоматизация)». Инна выбрала **разделение по типу задачи**: `/brief` — маркетинг/контент/бренд, `/discovery` — новый цифровой продукт. **`/brief` сейчас не трогаем** (там есть противоречие в антитриггере, но это отдельная задача когда вернёмся к воронке).

3.2. **Бизнес-процесс — Инна (консультирует) + Ирина (внедряет).** Разложили полный пайплайн от контакта до сопровождения: совместная встреча → бриф/discovery → 2 КП (раздельные, юр.фасад) → 2 договора → параллельная работа → MVP → запуск → сопровождение. Бриф один общий, КП и договоры раздельные. Скиллы /brief и /discovery работают только на этапе сбора брифа — дальше пайплайн ведут другие агенты команды.

3.3. **Фильтр-воронка перед моим пайплайном.** Инна развернула: перед «моим» пайплайном есть AI-диагностика на сайте. Они с Ириной не всех берут. Кто не подходит — отправляется на продуктовую «греющую» лестницу (продукты для самопокупки), либо рекомендует знакомым более крупным клиентам. Только прошедшие фильтр доходят до совместной встречи.

**Важно:** AI-диагностика и греющая лестница — это **два сайта (Инны и Ирины)** + **две AI-диагностики**. Архитектура этих воронок ещё «слабенькая» в существующих документах — отдельная большая задача на потом. Сейчас завершаем `/discovery` v2.
