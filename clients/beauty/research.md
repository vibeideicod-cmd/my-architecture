Исследование для разработки лендинга с флагманским продуктом Beauty App Builder.
Дата: апрель 2026.

---

## АГЕНТ 3: ТРЕНДЫ 2025–2026

### 1. Bento Grid 2.0 — Активная сетка
Карточки разного размера, живые: при наведении раскрываются, играют видео.
**Применение:** Секция "Фичи конструктора" — 6 карточек разного размера. Большая = главная фича (запись онлайн). При наведении — краткое демо-видео внутри.

### 2. Glassmorphism Evolved
Полупрозрачные слои с тонким блюром (`backdrop-filter: blur`). Деликатный, не кричащий.
**Применение:** Карточки тарифов и FAQ на тёмно-фиолетовом фоне. Ощущение: дорого и легко.

### 3. Spatial UI и Atmospheric Depth
Освещение, тени, параллакс. Мягкие цветные тени вместо чёрных.
**Применение:** Hero-экран с мокапом телефона в 3D-плоскости, чуть повёрнут, с мягкой фиолетовой тенью.

### 4. Kinetic Typography
Заголовки растягиваются, реагируют на курсор.
**Применение:** Hero-заголовок с плавным появлением слов (slide-up). Слово "Ваше" проявляется медленно — ощущение индивидуальности.

### 5. Scrollytelling
При прокрутке продукт "собирается". Пользователь путешествует.
**Применение:** Секция демо конструктора — при скролле последовательно появляется интерфейс приложения.

### 6. Micro-Delights
Маленькие радости: анимации при наведении и клике.
**Применение:** CTA "Написать в Telegram" — иконка при наведении чуть прыгает. При клике — pulse-анимация.

### 7. Dark + Pastel гибриды
Тёмный фон + пастельные акценты (лавандовый, мятный). Дышит, не давит.
**Применение:** Тёмный `#0d0d12` + акцент лавандово-розовый `#c4a4d4`. "Технология с душой."

### 8. Soft Brutalism
Видимые сетки, жирные гротески — но с мягкой типографикой внутри.
**Применение:** Секция FAQ — чёрная рамка, крупный номер (01, 02, 03), но нежный шрифт.

### 9. AI-виджеты в реальном времени
Адаптивные блоки, подстраивающиеся под пользователя.
**Применение:** "Подберём тариф за 30 секунд" — 3 вопроса → рекомендация тарифа.

### 10. Human Touch
Рукописные подчёркивания, стрелки от руки.
**Применение:** Цитата клиента — рукописное подчёркивание. Стрелка-скетч к CTA.

### UX-ДИЗАЙНЕР: Продуктовый подход / Builder UX

#### Типичные сложности при самостоятельной настройке

**Ментальная модель не совпадает с реальностью.** Мастер думает: "просто создам страничку". Реальность: каталог → категории → услуги → слоты → оплата. Каждый уровень вложенности — точка ухода.

**Конкретные болевые точки:**
- Не понимают разницу между "ботом-конструктором" и "приложением клиента" — два интерфейса в голове не укладываются
- Пустое поле "описание услуги" парализует
- Фото не того размера, нет кропа, интерфейс не подсказывает
- Настройка расписания — самое сложное, особенно если мастер работает нестабильно
- Слово "каталог" — не их слово. Их слово — "прайс" или "мои услуги"

**Где отваливаются на онбординге:**

| Этап | Процент отвала | Причина |
|---|---|---|
| Первый запуск бота | ~15% | Непонятно, что делать дальше |
| Пустое состояние ("добавьте услугу") | ~35% | Нет примера, нет шаблона |
| Настройка расписания / слотов | ~25% | Слишком много полей |
| "Поделиться с клиентом" | ~15% | Не понимают, что именно отправлять |
| После публикации — ждут записей | ~10% | Нет обратной связи, нет следующего шага |

---

#### 3-шаговый онбординг для мастера

**Принцип:** мастер видит работающее приложение до того, как его попросят заплатить.

**ШАГ 1 — "Расскажи о себе" (45 секунд)**

Бот задаёт 3 вопроса последовательно — именно в чате, не форма:
```
— Как тебя зовут?
— Чем занимаешься? [кнопки: Ногти / Брови / Ресницы / Волосы / Массаж / Другое]
— В каком городе работаешь?
```
Фото — опционально, пропустить одной кнопкой.

**ШАГ 2 — "Добавь первую услугу" (2 минуты)**

Бот предлагает готовые шаблоны по специализации:
```
Я подготовил типичные услуги для мастера по ногтям:
```

#### Retention — удержание на подписке

**Функции, которые создают зависимость от продукта:**

- **История клиентов** — Маша приходит раз в 3 недели, предпочитает гель-лак. Этого нет нигде больше.
- **Аналитика доходов** — сколько заработал vs прошлый месяц. Какая услуга приносит больше всего.
- **База клиентов с тегами** — постоянные, новые, "потерявшиеся" (60+ дней не приходили).
- **Авто-напоминания** — настроил один раз, работает всегда. Отписался — снова вручную.
- **Портфолио работ с привязкой к записи** — в Instagram нет привязки, здесь есть.

**Ловушки переключения (switching costs):**
1. Накопленная база клиентов с историей визитов — перенести вручную нереально
2. Персональная ссылка на каталог уже роздана клиентам, вписана в шапку Instagram, на визитке
3. Клиенты привыкли записываться именно так — смена инструмента = риск их потерять
4. История и аналитика остаётся в продукте — ушёл = потерял всё
5. Экспорт базы клиентов — только на тарифе Мастер/Про (стимул оставаться на платном)

**При отписке — "мягкая заморозка":**
- Данные не удаляются 90 дней
- Доступ к просмотру открыт, новые записи — нет
- Win-back сообщение на 7-й день: "Возвращайтесь — дадим 30 дней бесплатно"
- Конверсия win-back у аналогичных SaaS: 8-15%

**Топ-5 механик удержания:**

1. **"Умное расписание"** — "У вас 3 свободных окна в пятницу — разослать напоминание потерявшимся клиентам?"
2. **Еженедельный отчёт** — каждый понедельник: записей X, доход Y, новых клиентов Z. Создаёт привычку смотреть на бизнес через продукт.
3. **"Потерявшиеся клиенты"** — раз в месяц: "5 клиентов не были у вас 60 дней. Написать им?" Нажал — отправил. Прямые деньги.
4. **Накопительный статус** — после 6 месяцев бейдж "Проверенный мастер" в каталоге. Соцдоказательство для клиентов.
5. **Реферальная программа** — приведи подругу-мастера, получи месяц бесплатно.

**Telegram-последовательность для борьбы с оттоком:**

- Мастер не заходил 7 дней → "У вас 2 предстоящие записи — напомнить клиентам автоматически?"
- За 5 дней до списания → "За этот месяц вы приняли 34 записи через TMA. Продолжаем?"

### TELEGRAM-РАЗРАБОТЧИК: Архитектура и масштабирование

#### Multi-tenancy vs отдельный бот на каждого мастера

| Параметр | Multi-tenant (один бот) | Отдельный бот на мастера |
|---|---|---|
| Скорость разработки MVP | Быстро — одна кодовая база | Медленно — нужен orchestrator |
| Стоимость инфраструктуры | Низкая | Высокая — N webhook-эндпоинтов |
| UX для мастера | Хуже — работает "внутри" чужого бота | Лучше — своё имя в Telegram |
| UX для клиента | Нейтрально | Лучше — пишет "своему" боту |
| Масштабируемость | Высокая — горизонтально | Сложная — 1000 ботов = 1000 webhook |
| Обновления | Мгновенно для всех — один деплой | Нужен rolling update всех ботов |
| Изоляция данных | Программная (tenantId) | Физическая — естественная |

**Рекомендация:**
- **MVP → Multi-tenant.** Один бот, один webhook, один сервис. Недели, не месяцы.
- **При 500+ мастеров → Гибрид:** multi-tenant backend + опция Premium "получи собственного бота"

**Брендинг мастера внутри общего бота:**
Deep link передаёт идентификатор мастера: `t.me/YourSaasBot/app?startapp=master_123`
TMA читает `start_param`, запрашивает брендинг (имя, цвет, аватар) и применяет динамически через CSS-переменные.

#### Критически важные Telegram API

**Telegram Payments — подписка мастера:**

Нативной рекуррентности нет. Схема: Invoice → оплата → webhook → обновить `subscription_expires_at` → cron напоминание за 3 дня с новым invoice.

**Провайдеры для России:**

| Провайдер | Рекуррентность | Комиссия |
|---|---|---|
| Robokassa | Да (recurring tokens) | 2.3-3.5% |
| ЮKassa | Да (auto-payment) | 3.5% |
| Т-Банк | Да | 2.69% |
| Tribute (ЮMoney) | Нет | ~8% |
| СБП | Нет | 0.4-0.7% |

**Рекомендация:** ЮKassa или Т-Банк — сохранение карты, серверное списание, ссылка на оплату в боте.

**Критичные WebApp API:**
- `MainButton` — единственная CTA на экран ("Записаться — 2000 ₽"), `showProgress()` при отправке
- `BackButton` — навигация без браузерной истории
- `HapticFeedback` — `selectionChanged()` при выборе услуги, `notificationOccurred('success')` при записи

**CloudStorage — что куда:**

| CloudStorage (Telegram) | Своя БД |
|---|---|
| Последний мастер, черновик формы | Расписание, записи, клиенты |
| Флаг "онбординг пройден" | История платежей |
| Кэш каталога (TTL) | Профили мастеров |

#### Ограничения и как их обойти

**1. Кэширование TMA**

Telegram агрессивно кэширует. `location.reload()` часто не помогает.

Решение: версионированные URL (`app.js?v=20250411`), при открытии TMA проверять версию в CloudStorage и форс-обновлять. Данные расписания/слотов — никогда не кэшировать дольше 60 секунд.

**2. Лимиты Bot API при росте**

| Лимит | Значение | Решение |
|---|---|---|
| Глобальный | 30 msg/сек | Message queue (BullMQ + Redis) |
| Один пользователь | ~1 msg/сек | Дедупликация уведомлений |
| Webhook timeout | 60 сек | Отвечать 200 OK сразу, обрабатывать async |

**3. Оффлайн-режим**

TMA работает только онлайн. При потере связи во время заполнения формы — сохранять черновик в CloudStorage при каждом изменении поля.

**4. Критическое ограничение, которое упускают новички: `initData` не валидируется**

TMA передаёт данные пользователя (telegram_id, имя) в `initData`. Без валидации на сервере — любой может подделать user_id и получить чужие данные.

Обязательно: HMAC-SHA256 валидация `initData` на каждый API-запрос + проверка свежести (`auth_date` не старше 5 минут). `initData` передаётся в заголовке `Authorization: tma <initData>`. Без этого вся система небезопасна.
