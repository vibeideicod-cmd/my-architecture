> **Назначение:** для каждого элемента визитки — что есть сейчас, куда должно вести, что нужно создать. Дизайн заморожен — меняем только логику переходов и новые продукты-получатели.

> **Юр.правила:** `clients/irina/CLAUDE.md` (изготавливаю/разрабатываю/реализую — нет «провожу аудит/консультирую»)

---

## Принципы

1. Каждый интерактив имеет смысл и ведёт куда-то конкретно — никаких декоративных кнопок без действия
2. Каждый клик — шаг по воронке: ознакомление → интерес → квалификация → покупка → повторная покупка
3. Где возможно — даём «полную картину» в карточке/модалке, а уже из неё CTA «купить» или «обсудить»
4. Стек реализации: vibe coding — TG Mini App, VK Mini App, чистый HTML+CSS+JS лендинги, бот на Node.js, Supabase
5. Юр.форматировки везде на её словаре: «изготавливаю», «разрабатываю», «реализую», «упаковываю» — не «консультирую»
6. Логотипы внешних брендов (TG, VK) — фирменные цвета, не подменяем монохромом
7. Все заявки уходят в её собственную CRM (`my-bz-klient` aliased), не в Google-таблицы и не в Calendly

## Архитектурные развилки (что Инна должна решить)

### Развилка 1. TG-кнопка — куда ведёт первый клик?

**Варианты:**
- **A.** Прямая личка (как сейчас) — быстро, но Ирина тонет в нерелевантных запросах
- **B.** Бот квалификации → личка только если клиент квалифицирован — фильтр работает, но 1 лишний шаг
- **C.** Бот → Mini App каталог (видишь продукты с ценами) → личка только для крупных заказов — лучшая воронка, но дольше делать

**Моё мнение:** B + C параллельно. Один бот (П1) с двумя ветками: «нужна услуга — иди в каталог (П2)» и «есть конкретный вопрос — пиши лично». Бот квалифицирует за 4-5 вопросов и в 80% случаев закрывает на каталог без участия Ирины.

### Развилка 2. Календарь записи — как делаем?

**Варианты:**
- **A.** Свой Mini App в TG — слоты в Supabase, Ирина админит через бот — **Инна выбрала «не Google»**
- **B.** Встроенный виджет на сайте (форма-заявка, Ирина потом подтверждает время вручную) — проще, но создаёт ручную работу
- **C.** Кнопка «свяжись со мной — выбрали удобное время» — без календаря вообще

**Моё мнение:** A. Раз бейдж на сайте кричит «Открыт 1 слот · май» — клиент должен **сразу** видеть и брать этот слот, не писать в чат. Это премиум-UX. **П3** делает Systems за 3-4 часа.

### Развилка 3. Карточки услуг — лендинг или модалка?

**Варианты:**
- **A.** Каждая услуга = отдельный лендинг по URL (`/brendbuk/`, `/logo/`) — лучше для SEO и для отправки одной ссылкой клиенту
- **B.** Каждая услуга = модалка fullscreen на главной — быстрее, единая визитка
- **C.** Каждая услуга = пункт в TG Mini App каталоге (П2) — всё в одном месте, но требует переход в TG

**Моё мнение:** A. Раз `stydiyatsi.ru` зарегистрирован, грех не использовать subpages. Ирина может отправить ссылку «вот мой брендбук — `stydiyatsi.ru/brendbuk`» — это сильнее, чем «откройте мою визитку и пролистайте». Каждый лендинг = 1-3 часа Кодыча.

### Развилка 4. Voice-инструмент — реальный или симуляция?

**Варианты:**
- **A.** Реальная транскрипция через Whisper + генерация через Claude (П18, 6-8 часов) — даёт «вау», но требует расходов на API (~$0.01 за демо, при 100 кликах/день = $30/мес)
- **B.** Оставить симуляцию + добавить честный мелкий текст «это демо — в боте работает по-настоящему» — 0 часов, 0 расходов
- **C.** Реальная транскрипция, но **только в её TG-боте** (на сайте симуляция со ссылкой «попробовать по-настоящему в боте») — лучшая воронка: демо ловит → бот закрывает на лиде

**Моё мнение:** C. Симуляция на сайте честно подписана как демо. CTA «попробовать по-настоящему» ведёт в **П1 → П2**, где **внутри Mini App** работает реальный voice-to-image. Это превращает игрушку в инструмент-продажник, и API-расходы только на квалифицированных лидах.

### Развилка 5. Палитра-генератор — расширять или оставить?

**Варианты:**
- **A.** Оставить демку как есть — 0 часов
- **B.** Добавить генератор PDF палитры (П16) — клиент скачивает свою палитру, попадает в базу, получает рассылку
- **C.** Превратить в отдельный микро-продукт «Конструктор палитры» (~M14 в её линейке) — продаём за 199-499₽ как лид-магнит

**Моё мнение:** B как минимум. PDF палитры на лету = клиент уносит **результат** домой → есть повод вернуться. Для C — отдельный заход после запуска визитки. **П16** = 2-3 часа.

## Что закрыли этой картой

- **27 «мёртвых» элементов** превращены в осмысленные ссылки
- **Один шлюз П1 (TG-бот)** — все основные CTA сходятся в нём, не дублируем коммуникацию
- **8 кейсов в галерее** становятся продающим инструментом, а не декорацией
- **11 строк Capabilities** — каждая = отдельный лендинг = отдельный URL для шеринга
- **Демки палитры и voice** конвертируются в лидов через П1 + забирают подписку через PDF (П16) и счётчик (П19)

## Что осталось открыто (требует Discovery до старта)

1. **Контент 8 кейсов** для П15 — нужны исходники работ от Ирины + текст «что было / что разработала / результат» × 8. Без этого модалка-кейс пустая.
2. **Тексты лендингов** П4-П14 — это работа Copywriter в её голосе. До текстов лендинги верстать рано.
3. **П12 «AI-помощник» лендинг с встроенным демо** — нужна предварительно обученная база знаний под Ирину (FAQ + услуги + цены). Это полу-Discovery, полу-Build.
4. **Цены в каталоге П2** — Ирина должна зафиксировать «от X₽» по каждому продукту. Без цен Mini App-каталог не запустить.
