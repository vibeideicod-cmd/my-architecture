**Почему:**
1. **Юридическая чистота.** Каждый домен принадлежит одному юр.лицу. Оферта одна, чеки одной стороне, никаких пересечений с налоговой.

4. **Между сайтами — кросс-ссылки** на главных страницах и в кейсах.

### Поэтапный план

- **MVP-этап (сейчас):** одностраничник Ирины — простая визитка с описанием, портфолио, кнопкой связи. Можно на бесплатном поддомене (Vercel / GitHub Pages) — не тратим бюджет на хостинг до того, как пойдут продажи.

Поддомен на старте (вариант B), потом перенос на свой домен (C). Технически: 301-редирект, контент мигрирует. Это путь «MVP → рост» без больших начальных затрат.

## Решение 3. База клиентов — общая или раздельные?

### Юридическая правда (важная!)

База клиентов = **персональные данные** (ФИО, телефон, email, история взаимодействия). По ФЗ-152 у каждой записи в базе должен быть **оператор персональных данных**. Это ИП или самозанятый, который собирает и обрабатывает данные.

### Варианты

| Вариант | Юр.раскладка | Сложность | Чистота |
|---|---|---|---|
| **A. Раздельные базы** | Каждая ведёт своих клиентов. При партнёрстве — обмен карточками с согласия клиента | Простая | ✅ Полная чистота |
| **B. Общая база, оба оператора** | Обе зарегистрированы в РКН как операторы, общая политика называет обеих | Сложная | ⚠️ Возможно, но громоздко |
| **C. Общая база, Инна = оператор, Ирина = обработчик** | Инна владеет базой, Ирина работает с данными по договору поручения (ст.6 ФЗ-152) | Средняя | ✅ Юридически чисто |

### Моя рекомендация: 🟢 Вариант C — общая база, Инна оператор, Ирина обработчик

**Почему:**
1. **Инна ты хотела общую базу** — оставляем общую (одна «Моя база клиентов» на VPS).
2. **Юридически чисто** — есть прямой механизм в ФЗ-152: договор поручения на обработку. Не нужно регистрировать обе стороны как операторов.
3. **Клиенту понятно:** в политике конфиденциальности написано «Оператор данных — Инна Андрейченко (ИП), обработчик — Ирина Цепаева по договору № X».
4. **При спорах по ПД** ответственность делится по договору — оба защищены.

### Что нужно сделать

1. Yuridika составляет **договор поручения на обработку ПД** между ИП Инны и самозанятой Ириной
2. В политике конфиденциальности на обоих сайтах указываются обе стороны
3. На каждом сайте — согласие на обработку ПД (галочка в форме / у AI-помощника)
4. Регистрация Инны в РКН как оператора (если ещё не сделана)

### 📋 Задачи для Yuridika + PD-Lawyer
- Договор поручения на обработку ПД между Инной и Ириной
- Шаблон политики конфиденциальности (отдельный для каждого сайта, с указанием обеих сторон)

## Решение 4. «Партнёрские бандлы» — что это и как работает

### Простое объяснение

**Бандл = комплект.** Это когда мы продаём не один продукт, а несколько продуктов вместе, как пакет. Клиент платит общую сумму, получает всё сразу. Удобно для продажи комплексных решений.

**Пример бандла Инна + Ирина:**

> «Упаковка эксперта под ключ — 50к» = брендбук Ирины (изготовление, 30к) + аудит позиционирования Инны (консультация, 20к). Клиент видит один продукт «Упаковка эксперта», платит 50к.

### Как это работает юридически

Бандл — это **маркетинговая упаковка**, а не юридическая. Юридически — это всегда **два отдельных контракта**:

```
Клиент покупает «Упаковку эксперта» — 50к
   ↓
Подписывает 2 договора:
├── С Иной (ИП): «Экспертная консультация по позиционированию» — 20к → чек ИП
└── С Ириной (самозанятая): «Изготовление брендбука» — 30к → чек НПД
   ↓
Деньги идут раздельно — каждой на свой счёт
```

### Какие бандлы у нас могут быть

| Бандл | Инна (консультация) | Ирина (реализация) | Подход |
|---|---|---|---|
| **Упаковка эксперта** | Аудит позиционирования | Брендбук + визитка + соцсети-визуал | «Меня не видят» |
| **Старт онлайн** | Стратегия воронки | Лендинг + квиз + бот-запись | «Нет клиентов» |
| **Автосистема** | Архитектура автоматизации | AI-помощник + воронка-вёрстка | «Нет времени» |
| **Нейро-архитектура под ключ** | Сборка 7-слойной архитектуры | Реализация всех артефактов (сайт, платформа, AI) | Флагман «Хочу всё» |

- Шаблон оферты на сайте, объясняющей бандловую схему

## Решение 6. Методология Ирины — С.С.С.Р. с её расшифровкой

### Принято Иной: общая платформа С.С.С.Р., буквы расшифровываются под Ирину

### Расшифровка С.С.С.Р. под Ирину (на основе её транскрипции)

| Буква | Расшифровка Инны (для справки) | **Расшифровка Ирины** | Источник из транскрипции |
|---|---|---|---|
| **С** | Структура (бизнеса) | **Структура (визуальная)** — ясная, понятная композиция | «Чёткие, красивые, ясные картинки. Чтобы глядя на них можно было понять, о чём идёт речь.» |
| **С** | Стратегия | **Стиль** — индивидуальный, не шаблонный | «Каждая идея — новый визуал. Нельзя повторять один дизайн на другого, просто поменять цвет.» |
| **С** | Свобода | **Свобода** — творческая для дизайнера + видение клиента | «Мне нравится, когда у меня есть свобода творчества. Я хочу увидеть, создать дизайн глазами клиента, но в то же время хочу свободу творчества.» |
| **Р** | Результат | **Результат** — дизайн как решение бизнес-задачи | «Идеальный клиент понимает, что дизайн — это решение бизнес-задач. Не просто логотип побольше, а решение, чтобы пользователь быстрее купил.» |

### Почему такая расшифровка работает

1. **Совпадает с её философией** — каждая буква подкреплена её собственными словами из транскрипции
2. **Зеркальна Инной по ритму** (С.С.С.Р.), но раскрыта через визуал
3. **Создаёт партнёрский продукт** — методология одна, расшифровка по специализации:
   - Инна = С.С.С.Р. (бизнес-архитектура)
   - Ирина = С.С.С.Р. (визуальная архитектура)

### Рабочее название методологии Ирины

**«Визуальная С.С.С.Р.»** — методология фирменного стиля и дизайна по принципу Структура · Стиль · Свобода · Результат.

Или короче: **«С.С.С.Р. в визуале»** (повторяет публичный голос Инны).

## Решение 7. Договорная схема партнёрства

### Чистый юридический ответ

> ⚠️ **Один контракт от ИП Инны на бандл — невозможен.** Часть услуг оказывает Ирина (самозанятая), и она должна выписать **свой чек** через приложение «Мой налог». Иначе деньги, которые Ирина получит через ИП Инны, для Ирины станут «доходом от посредничества» — а это запрещено для самозанятой (ст.4.4 ФЗ-422).

### Рекомендуемая схема: 🟢 Два независимых договора на каждый бандл

```
Клиент покупает «Упаковка эксперта — 50к»
   ↓
Подписывает 2 договора в один день:
├── Договор № 2026-001-А с ИП Инны: «Экспертная консультация» — 20к → чек ИП
└── Договор № 2026-001-Б с самозанятой Ириной: «Изготовление брендбука» — 30к → чек НПД
   ↓
Каждая получает деньги напрямую от клиента, на свой счёт
```

### Преимущества

- ✅ Полная юридическая чистота
- ✅ Ни одна не зависит финансово от другой
- ✅ Каждая выписывает свои чеки и платит свои налоги
- ✅ При расторжении договоров можно прервать одну часть, продолжить другую

### Партнёрский % (если применимо)

Если Инна привела клиента Ирине — Ирина платит Инне 10% от своего гонорара по партнёрской программе (`inna/architecture.md` Слой 4, п.4.6). Это оформляется **отдельным договором партнёрства** между ИП Инны и самозанятой Ириной (не зависит от договоров с конечным клиентом).

### 📋 Задачи для Yuridika + Contract-Lawyer
- **Шаблон 1:** Договор Инна-клиент (экспертная консультация)
- **Шаблон 2:** Договор Ирина-клиент (изготовление дизайна / разработка)
- **Шаблон 3:** Договор партнёрства Инна-Ирина (партнёрский %)
- **Памятка для клиента:** «Почему два договора» — простое объяснение для лендинга бандлов
