# Архитектура МГ-платформы: полный план развития

> Этот документ описывает текущее состояние, видение развития и фазы реализации МГ-платформы — от конструктора визиток (v2.0) до полноценной платформы управления мастер-группами (v3).

---

## Текущее состояние (v2.0 — апрель 2026)

### Что работает прямо сейчас

**Полный flow:** мастер подаёт анкету → админ одобряет → мастер проходит конструктор (8 вопросов) → публикует визитку → участник записывается через публичную страницу.

**Точки входа:**
- Mini App: `t.me/mastergruppa_bot/mg_platform`
- Web: `demo.ideidlyabiznesa1913.ru/mg/join`
- Админка: `demo.ideidlyabiznesa1913.ru/mg/admin`

### Схема БД — v2 таблицы (миграции 200-205)

| Таблица | Назначение | Ключевые поля |
|---|---|---|
| `mg_applications` | Анкеты кандидатов в мастера | full_name, contact, niche, status (submitted/approved/rejected), master_number, application_token |
| `mg_master_pages` | Страницы-визитки мастеров | slug, page_config (JSONB), token_hash, token_plain, published, version |
| `mg_config_questions` | Справочник 8 вопросов конструктора | block (hero/about/offer/cta), label, hint, type, required |
| `mg_leads` | Заявки участников через форму на публичных страницах | master_slug, visitor_name, visitor_contact, message |

### RPC-функции (11 штук, security definer)

| Функция | Для кого | Что делает |
|---|---|---|
| `mg_submit_application` | anon | Подать анкету → получить application_token |
| `mg_get_application_by_token` | anon | Проверить статус анкеты |
| `mg_moderate` | admin | Одобрить/отклонить → создать mg_master_pages |
| `mg_save_page` | master | Сохранить изменения в конструкторе (JSONB merge) |
| `mg_publish_page` | master | Опубликовать страницу (валидация обязательных полей) |
| `mg_submit_lead` | anon | Отправить заявку участника через форму на странице |
| `mg_admin_list_applications` | admin | Все анкеты для админки |
| `mg_admin_list_pages` | admin | Все страницы мастеров |
| `mg_admin_list_leads` | admin | Все заявки участников |
| `mg_admin_get_master_link` | admin | Получить ссылку конструктора (восстановление) |
| `mg_admin_regenerate_token` | admin | Перевыпустить токен мастера |

### Схема БД — v1 таблицы (миграции 100-102, legacy)

Эти таблицы создавались для первой версии МГ — «ведение 5-недельных мастер-групп с участниками, материалами, заданиями и чатом». **Не удалены, не используются в v2 UI, но важны для v3.**

| Таблица | Назначение | Ключевые поля |
|---|---|---|
| `mg_masters` | Ручные записи мастеров (только Инна-демо) | id (text), name, niche, bio, program_title, weeks, meetings_per_week, token_hash |
| `mg_participants` | Участники мастер-группы конкретного мастера | name, contact, master_id (FK→mg_masters) |
| `mg_materials` | Учебные материалы по неделям | master_id, week, title, url |
| `mg_tasks` | Задания по неделям | master_id, week, title, deadline |
| `mg_messages` | Чат мастер ↔ участник (realtime) | master_id, author_name, author_kind, body |
| `mg_cases` | Кейсы / результаты участников | master_id, participant_name, result, contact |

**Ключевая проблема:** v1 привязана к `mg_masters.id` (text, например `'inna'`), а v2 — к `mg_master_pages.slug` (text, например `'master-1'`). Нужен bridge.

## Видение v3: полноценная платформа

### Что хочет Инна

1. **Кабинет мастера** — каждый одобренный мастер видит свою страницу, управляет участниками, назначает задания, отмечает посещаемость
2. **Участники мастера** — записываются через публичную страницу или по прямой ссылке, проходят программу мастера
3. **Задания** — мастер создаёт задания (текст, дедлайн), участник отмечает «сделано» или загружает результат
4. **Посещаемость** — мастер создаёт встречи (дата, тема), отмечает кто пришёл
5. **Админская панель** — Инна видит ВСЕХ мастеров → их участников → прогресс. Дашборд с метриками, drilldown, экспорт

### Три роли

| Роль | Что видит | Точка входа |
|---|---|---|
| **Админ** | Всех мастеров, всех участников, все заявки, метрики, экспорт | `/mg/admin` |
| **Мастер** | Свою страницу, своих участников, свои задания, посещаемость | `/mg/cabinet?m=<slug>&t=<token>` |
| **Участник** | Публичную страницу мастера, свои задания, свой прогресс | `/mg/m/<slug>` → после записи: `/mg/my?p=<token>` |

## Маппинг v1 → v3

| Нужная функция | v1 таблица | Статус | Что доработать |
|---|---|---|---|
| Участники мастера | `mg_participants` | Есть, но FK на `mg_masters` | Создать bridge или новую `mg_v2_participants` с FK на `mg_master_pages.slug` |
| Материалы | `mg_materials` | Есть | Привязать к slug вместо master_id |
| Задания | `mg_tasks` | Есть, но без submissions | Добавить `mg_task_submissions` |
| Чат | `mg_messages` | Есть, realtime | Привязать к slug |
| Кейсы | `mg_cases` | Есть | Привязать к slug |
| Посещаемость | — | Нет | Новые таблицы `mg_meetings` + `mg_attendance` |
| Кабинет мастера (v2) | — | Нет | Новая страница `cabinet-v2.html` |
| Портал участника | — | Нет | Новая страница `my.html` или расширение `master-page.html` |
| Админский дашборд | — | Частично (3 вкладки) | Добавить метрики и drilldown |

**Решение по bridge:** создаём новые таблицы `mg_v2_*` с FK на `mg_master_pages.slug`, НЕ модифицируем v1 таблицы. V1 остаётся как есть для обратной совместимости.

## Фазы реализации

### Фаза v2.1 — Кабинет мастера + участники

**Цель:** мастер видит кто записался, может управлять списком участников.

**Новые таблицы:**

```
mg_v2_participants
├── id                bigserial PK
├── master_slug       text FK → mg_master_pages(slug)
├── full_name         text NOT NULL
├── contact           text NOT NULL
├── source            text (lead / direct / import)
├── lead_id           bigint FK → mg_leads(id) — если пришёл из формы
├── status            text (active / paused / completed / dropped)
├── joined_at         timestamptz default now()
└── notes             text
```

**Новые RPC:**
- `mg_get_my_participants(slug, token)` — мастер видит своих участников
- `mg_convert_lead_to_participant(lead_id, slug, token)` — конвертировать заявку в участника
- `mg_add_participant(slug, token, name, contact)` — добавить вручную
- `mg_update_participant_status(slug, token, participant_id, status)` — изменить статус

**Новые страницы:**
- `cabinet-v2.html?m=<slug>&t=<token>` → `/mg/cabinet` — приватный кабинет мастера
  - **Таб 1 «Моя страница»:** ссылка + кнопка «Редактировать» (открывает build.html)
  - **Таб 2 «Мои участники»:** таблица участников, конвертация из лидов, добавление вручную
  - **Таб 3 «Мои заявки»:** лиды с публичной страницы (из mg_leads) — кнопка «Принять» → становится участником

**Расширение админки:**
- Вкладка «Страницы» → у каждого мастера кнопка «→ Кабинет» (для проверки)
- Новый счётчик в статах: «Участников всего»

**Миграции:** серия 300 (300_v2_participants_schema, 301_v2_participants_rpc)

### Фаза v2.2 — Задания и прогресс

**Цель:** мастер создаёт задания, участник отмечает выполнение, мастер проверяет.

**Новые таблицы:**

```
mg_assignments
├── id                bigserial PK
├── master_slug       text FK → mg_master_pages(slug)
├── title             text NOT NULL
├── description       text
├── week              int — номер недели программы (опционально)
├── deadline          timestamptz
├── created_at        timestamptz default now()
└── is_active         boolean default true

mg_submissions
├── id                bigserial PK
├── assignment_id     bigint FK → mg_assignments(id)
├── participant_id    bigint FK → mg_v2_participants(id)
├── status            text (pending / submitted / approved / revision)
├── content           text — текст ответа участника
├── file_url          text — ссылка на файл (Яндекс.Диск и т.п.)
├── submitted_at      timestamptz
├── reviewed_at       timestamptz
├── review_note       text — комментарий мастера
└── created_at        timestamptz default now()
```

**Расширение кабинета мастера:**
- **Таб 4 «Задания»:** создание заданий, просмотр submissions, оценивание
- Карточка задания: название, дедлайн, сколько участников сдало из скольких

**Портал участника:** (`my.html?p=<participant_token>` → `/mg/my`)
- Мои задания (список с дедлайнами)
- Отправить выполнение (текст + ссылка на файл)
- Мой прогресс (сколько сделано / сколько всего)

**Миграции:** серия 400

### Фаза v2.3 — Встречи и посещаемость

**Цель:** мастер создаёт встречи, отмечает кто пришёл, статистика за период.

**Новые таблицы:**

```
mg_meetings
├── id                bigserial PK
├── master_slug       text FK → mg_master_pages(slug)
├── title             text NOT NULL — тема встречи
├── meeting_date      date NOT NULL
├── meeting_time      time — время начала (опционально)
├── notes             text
└── created_at        timestamptz default now()

mg_attendance
├── id                bigserial PK
├── meeting_id        bigint FK → mg_meetings(id)
├── participant_id    bigint FK → mg_v2_participants(id)
├── present           boolean NOT NULL
├── note              text
└── marked_at         timestamptz default now()
```

**Расширение кабинета мастера:**
- **Таб 5 «Встречи»:** календарь встреч, быстрая отметка «был/не был»
- Статистика: кто ходит регулярно, кто пропускает

**Расширение портала участника:**
- Расписание встреч, мои пропуски

**Миграции:** серия 500

---

### Фаза v2.4 — Админский дашборд с метриками

**Цель:** Инна видит полную картину по всем мастерам в одном месте.

**Расширение админки:**

- **Новая вкладка «📊 Обзор»** (дашборд):
  - Карточки: всего мастеров / всего участников / заявки за 7 дней / задания активных
  - Таблица мастеров: имя → кол-во участников → кол-во заявок → последняя активность
  - Drilldown: клик на мастера → его участники → их прогресс

- **Расширение вкладки «Страницы»:**
  - Колонка «Участники» (число)
  - Колонка «Заявки» (число)
  - Кнопка «→ Кабинет мастера» (открывает от имени админа)

- **Экспорт:**
  - CSV по мастерам
  - CSV по участникам (все или фильтр по мастеру)
  - CSV по заданиям + submissions

**Новые RPC:**
- `mg_admin_dashboard_stats(admin_secret)` — агрегированные метрики
- `mg_admin_master_detail(admin_secret, slug)` — детали по мастеру (участники, задания, посещаемость)
- `mg_admin_export_participants(admin_secret, format)` — CSV-экспорт

**Миграции:** серия 600 (в основном RPC, без новых таблиц)

## Принципы архитектуры

1. **Все таблицы с префиксом `mg_`** — не конфликтуют с beauty и другими проектами в том же Supabase
2. **Все операции через RPC (security definer)** — RLS заблокирована, прямой INSERT/UPDATE невозможен
3. **Каждая фаза — отдельный набор миграций** (300s, 400s, 500s, 600s) — можно остановиться на любой фазе
4. **Обратная совместимость** — v1 таблицы не удаляются, v2 таблицы не ломаются
5. **Авторизация через токены** — SHA-256 hash в БД, raw token в URL. Для внутренней платформы приемлемо
6. **Нет фреймворков** — чистый HTML + CSS + vanilla JS. Каждая страница — один самодостаточный файл
7. **JSONB для гибкости** — page_config позволяет добавлять новые блоки без миграций
8. **Терминология:** участник (не клиент), команда МГ (не Инна), нейтральный род

## Рекомендованный порядок

| Фаза | Зависит от | Примерный объём | Что получает Инна |
|---|---|---|---|
| **v2.1** | — | 2-3 дня | Мастера видят своих участников, конвертация лидов, базовый кабинет |
| **v2.2** | v2.1 | 2-3 дня | Задания и проверка, прогресс участников, портал участника |
| **v2.3** | v2.1 | 1-2 дня | Встречи и посещаемость, статистика посещений |
| **v2.4** | v2.1-v2.3 | 1-2 дня | Полный дашборд для админа, drilldown, экспорт |

**Начинать с v2.1** — это фундамент для всего остального. Без участников нет заданий и посещаемости.

## Открытые вопросы (для обсуждения с Инной)

1. **Авторизация участника:** сейчас у участника нет личного кабинета. Как он будет заходить? Варианты:
   - По одноразовой ссылке с токеном (как мастер в конструкторе)
   - Через Telegram Mini App с `tg.initDataUnsafe.user.id`
   - Через email-код (magic link)

2. **Нужен ли чат?** v1 имеет mg_messages с realtime. Переносить в v3 или оставить общение в Telegram?

3. **Материалы:** мастер загружает материалы (видео, PDF) — хранить ссылки или загружать в Supabase Storage?

4. **Интеграция с Zoom-архивом:** 100 GB записей встреч МГ в `~/Documents/Zoom` — нужно ли привязывать к встречам в системе?

5. **Multi-admin:** сейчас один пароль на всех. В v2.4 сделать таблицу `mg_admins` с отдельными паролями и `reviewed_by`?
