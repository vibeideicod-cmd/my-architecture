# Research: МГ-платформа — конструктор бизнес-визиток

**Фаза:** 2 — Research.
**Playbook:** [product-creation.md](../../knowledge/playbooks/product-creation.md) раздел 2/Фаза 2 (пятинаправленный промпт).
**Темп:** минимальная программа (3-4 конкурента, ~2 часа). Дедлайн плана — сегодня/ближайшие дни.

---

## 1. Конкуренты

Анализируются по 4 параметра: **что умеет**, **как устроен онбординг**, **что берём**, **чего избегаем**.

### 1.1 Taplink (ru) — `taplink.ru`

Самый массовый русскоязычный конструктор «визитка-ссылка» для Instagram-эпохи. Блоки: фото, заголовок, ссылки, кнопки, форма записи, галерея, текст, отзывы, соцсети.

- **Что умеет:** drag-and-drop редактор, десятки шаблонов, платные фичи (премиум от 299₽/мес), своё имя домена, аналитика кликов.
- **Онбординг:** регистрация по email/телефон → выбор шаблона → попадаешь в редактор с пустой страницей → ковыряешь блоки.
- **Что берём:** идею «одна ссылка = вся суть», форматы блоков (hero, offer, CTA, contacts).
- **Чего избегаем:** пустой редактор пугает не-технаря. Нет «опросника, который собрал бы всё за тебя». Всё собирается руками.

**Главное отличие нас:** у нас **опросник вместо редактора**. Не «собери из блоков», а «ответь на 8 вопросов и получи готовое».

### 1.2 Beacons.ai (en) — `beacons.ai`

Англоязычный creator-конструктор «link in bio» с акцентом на инфлюенсеров.

- **Что умеет:** около 30 типов блоков, shop, tip-jar, integrations с TikTok/YouTube/Spotify, email-capture, аналитика, A/B-тесты.
- **Онбординг:** регистрация → короткий onboarding-опрос (5-7 вопросов: о себе, о нише, цели) → автоматически создаётся стартовая страница по ответам → дальше редактируешь.
- **Что берём:** **онбординг-опрос как способ собрать стартовую страницу**. Именно это нам нужно — но без дальнейшего редактора, MVP заканчивается на «получил результат».
- **Чего избегаем:** маркетплейс-фичи (shop, tip-jar) — слишком сложно, не наша ниша сейчас.

**Ключевой референс:** Beacons делает именно то, что мы хотим — опрос → авто-сборка. Но мы останавливаемся на MVP «опрос → страница» без редактора.

### 1.3 Stan.store (en) — `stan.store`

Creator-economy конструктор «одна ссылка = один эксперт» с упором на монетизацию.

- **Что умеет:** ссылки, цифровые товары, коучинг-сессии, курсы, приём платежей, booking-календарь.
- **Онбординг:** регистрация → выбор «I'm a coach / creator / educator» → auto-setup страницы на основе выбора → дальше кастомизация.
- **Что берём:** минималистичную дефолтную страницу, которая уже выглядит ок сразу после регистрации (без «пустого редактора»).
- **Чего избегаем:** платёжка, booking, продукты — это всё в «чего НЕ будет в MVP».

### 1.4 Skillbox / Setters — резидентские анкеты (ru)

Не конструктор страниц, а **референс по модерации анкет**. Как образовательные платформы принимают резидентов/экспертов/менторов:

- **Что умеет:** форма-анкета (ФИО, опыт, кейсы, мотивация, согласие) → попадает в CRM → модератор видит список → решение «принято / отклонено / на доработку» с комментарием → ответ кандидату.
- **Что берём:** структуру анкеты (5-10 полей, не больше), механику статусов (`submitted → under_review → approved / rejected`), обязательный consent на обработку ПД.
- **Чего избегаем:** сложная multi-step анкета с автосохранением, интеграция с CRM, автоматические проверки по базе.

**Главное отличие нас:** наш admin не модерирует через CRM — у Инны простая админка, одна страница с очередью и двумя кнопками.

## 3. UX и сценарии

Три типичных сценария пути пользователя.

### 3.1 Сценарий «Алиса подаёт анкету»

**Кто:** Алиса, 38, школа гитары, участница МГ УБ, не технарь.
**Точка входа:** Инна дала ссылку `demo.ideidlyabiznesa1913.ru/mg/` на встрече.

1. Открывает `index.html` → видит лендинг с CTA «Стать мастером» (новая секция)
2. Нажимает CTA → попадает на `apply.html` (форма анкеты, 8-9 полей)
3. Заполняет ФИО, контакт (Telegram @alisa_guitar), город (Москва), нишу (музыка), о бизнесе (300 знаков «школа гитары для взрослых...»), опыт ведения групп, что хочет получить, согласие на ПД
4. Нажимает «Отправить» → получает экран «Спасибо! Твой номер — Мастер 7. Статус: на рассмотрении. Вот твоя ссылка: `/mg/status.html?a=<token>` — сохрани её»
5. **Что удобно:** короткая анкета, понятно «о чём спрашивают». **Что бесит:** если потерять ссылку статуса — потеряешь связь с анкетой (нет email-подтверждения).

**Вывод:** после отправки анкеты — показать ссылку статуса **крупно**, предложить **скопировать**, **отправить себе в Telegram** (через tg://), возможно **показать QR-код**. Критично.

### 3.2 Сценарий «Инна модерирует анкеты»

**Кто:** Инна, админ платформы.

1. Открывает `admin.html` → вводит пароль `nbcccp-2026`
2. Видит таб «Анкеты» со счётчиком новых (`submitted`)
3. Кликает на анкету Алисы → развёрнутая карточка с полями
4. Нажимает «Одобрить» → вводит короткий комментарий (опционально) → нажимает Submit
5. **Под капотом:** RPC `mg_moderate` → создаётся запись в `mg_master_pages`, генерится `slug` (например, `master-7`) и секретный токен → возвращается Инне один раз
6. Инна **копирует ссылку на конструктор** (`/mg/build.html?m=master-7&t=<token>`) и отправляет Алисе в Telegram

**Что удобно:** realtime (новые анкеты всплывают без F5). **Что бесит:** ручная отправка ссылки мастеру — Инне надо в Telegram написать сообщение.

**Вывод:** минимизировать ручные действия. Если Алиса указала Telegram @username в анкете, кнопка «Одобрить и отправить» должна копировать в буфер готовый текст: «Привет, Алиса! Твоя анкета одобрена. Вот ссылка на конструктор: ...». Инна вставляет в Telegram Алисы и отправляет одним жестом.

### 3.3 Сценарий «Алиса проходит конструктор»

1. Открывает ссылку из сообщения Инны → попадает на `build.html` с токен-гейтом (SHA-256 матч)
2. Видит заголовок «Собери свою страницу — 8 вопросов» и прогресс-бар
3. Проходит вопросы по блокам (hero → about → offer → CTA)
4. На каждом шаге — можно нажать «Посмотреть, как выглядит» → открывается `master-page.html?m=master-7` в новой вкладке
5. Дошла до конца → нажимает «Опубликовать» → RPC `mg_publish_page` → redirect на её публичную страницу
6. Копирует ссылку → отправляет первому клиенту

**Что удобно:** линейный путь, нельзя «заблудиться в редакторе». **Что бесит:** нет live preview — надо открывать новое окно каждый раз. Для MVP это ок, но в критику надо записать.

**Вывод:** в Фазе 4 Build — если останется время, сделать sticky preview справа. Если нет — кнопка «Посмотреть» рядом с каждым полем.

## 4. Технологии

Сравнение трёх подходов к рендеру страницы мастера.

| Подход | Как устроено | Плюсы | Минусы |
|---|---|---|---|
| **(a) CSR из JSON** (рекомендую) | `master-page.html` читает JSON `page_config` из Supabase через anon key + view → рендерит секции | Простота, 0 серверной логики, переиспользуем паттерны v1, мгновенный reload после edit | JS обязателен (не для SEO критично, для нас — не страшно) |
| **(b) SSR через Edge Function** | Supabase Edge Function возвращает готовый HTML | SEO, нулевой JS на клиенте | Сложнее (Deno runtime), дольше Build, не нужно в MVP |
| **(c) Pre-build статический** | При approve генерируется HTML-файл и коммитится в репо / scp на Beget | Мгновенная загрузка, zero cost | Инфраструктура сложная, редактирование = re-build |

**Рекомендация: (a) CSR из JSON.** Это путь v1, мы его уже освоили, он даёт нужный результат и не требует новых инструментов.

## 5. Монетизация

Как подобные продукты зарабатывают (для будущего roadmap, не для MVP).

| Модель | Пример | Применимо нам? |
|---|---|---|
| **Freemium** (базовая бесплатно, премиум за деньги) | Taplink | **Да, на будущее.** Бесплатно: 8 вопросов, одна тема. Премиум: больше вопросов, разные темы, свой домен, аналитика. |
| **Transactional** (комиссия с платежа клиенту) | Stan.store | Не сейчас — у нас нет платёжки. На будущее — ок, если добавим приём платежей. |
| **Subscription** (подписка за доступ) | Beacons.ai Pro | **На будущее.** 499-990₽/мес, разблокирует расширенные фичи. |
| **White-label** (для бизнеса/агентств) | — | **На будущее.** Инна может продавать «платформа под ваш бренд» другим школам/сообществам. |

**Для MVP:** всё бесплатно. Финансовая модель формируется после первой обкатки и обратной связи от 10+ мастеров.

## Ключевые выводы

1. **Прямого конкурента с анкетой-опросником вместо редактора — нет** в русскоязычном сегменте. Beacons делает нечто похожее, но он англоязычный и более сложный. У нас есть ниша «для не-технарей на русском, которые боятся редакторов».
2. **8 вопросов достаточно** для минимально жизнеспособной страницы (hero + about + offer + CTA). Это подтверждает анализ Taplink/Beacons — их базовые шаблоны содержат те же 4 секции.
3. **Live preview не обязателен в MVP.** Beacons и Stan.store делают preview, но Taplink в базовом редакторе — нет. Мы можем сделать кнопку «Посмотреть» без live — это приемлемый компромисс для дедлайна.
4. **Фото через URL — риск для не-технарей.** Ни одна из изученных платформ не заставляет пользователя «вставить URL публичной картинки». Все дают загрузить файл. Это слабое место нашего MVP — надо в критике решить.
5. **Модерация через пароль (как v1)** — нормальный подход для одного админа. Skillbox/Setters — это другие масштабы, у нас одна Инна и ей не нужен enterprise-auth.
