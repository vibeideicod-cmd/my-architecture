# Идея: МГ-платформа — конструктор бизнес-визиток для участниц УБ

**Статус:** draft, ждёт подтверждения Инны.
**Фаза:** 1 — Discovery.
**Playbook:** [product-creation.md](../../knowledge/playbooks/product-creation.md).

---

## Формулировка

Я создаю **веб-конструктор мини-приложения «визитка + функционал под нишу»** для **кураторов кураторской системы Университета Бизнеса — экспертов из разных ниш, которые хотят создать или протестировать свои продукты** (Алиса — школа гитары, Светлана — стилист, Вера — психолог, и так далее), чтобы **за 20 минут каждый эксперт собрал своё приложение, которое можно отправить клиентам одной ссылкой и не бояться блокировок мессенджеров**.

> **Важно:** это НЕ платформа для ведения мастер-групп. Мастер-группа УБ — среда, где кураторы обкатывают свои продукты. Каждый куратор — эксперт со своим бизнесом в своей нише. Приложение — их личный инструмент, который они забирают себе. В проекте кураторы называются **мастерами** — это их роль на платформе.

---

## Для кого (ЦА)

**Первый круг (сейчас):** 10+ **кураторов кураторской системы УБ** — это эксперты из разных ниш, которые уже ведут свои бизнесы и используют мастер-группу как полигон для обкатки и тестирования своих продуктов. Женщины 35–60:

- Алиса — школа гитары для взрослых
- Светлана Байдак — стилист
- Вера — психолог (индивидуальные консультации)
- Другие кураторы-эксперты — разные ниши (эзотерика, тело, творчество, коучинг, продажи)

На платформе они заходят в роли **мастера** — это их системное название.

**Общие черты:**
- НЕ технари. Умеют пользоваться браузером и Telegram, но «сайт» = «страшно и дорого».
- Уже пытались или пытаются вести клиентов через Telegram — и сталкиваются с блокировками / неудобствами.
- Хотят, но боятся Taplink/Tilda («надо уметь / надо платить / надо дизайн»).
- В мастер-группе обкатывают новые продуктовые фишки — полигон без риска.

**Второй круг (позже):** внешние эксперты из других ниш, которых Инна принимает через анкету-модерацию. Те же профили: эксперт со своим бизнесом или наставник, который уже работает с клиентами.

---

## Какую проблему решаем

1. **Блокировки мессенджеров.** У участницы нет места, куда позвать клиента одной ссылкой, чтобы не зависеть от того, работает ли сегодня Telegram / WhatsApp.
2. **Страх «сайта».** Taplink/Tilda требуют разобраться в редакторе, выбрать шаблон, скомпоновать блоки. Для не-технаря это отдельный проект.
3. **Нет единой точки входа** для клиента: «Вот я, вот что я делаю, вот как записаться» — одна ссылка, за 20 секунд понятно.
4. **Для Инны:** нет механики, как **принимать** в систему новых мастеров, просматривать их, одобрять, выдавать им готовую площадку — без ручного копирования HTML и правки полей.

## Главный результат для пользователя

**Для участницы (мастера):**
1. Подала анкету → получила номер «Мастер N» и ссылку на статус
2. Инна одобрила → пришла ссылка на конструктор
3. Ответила на 8 вопросов → увидела собранную страницу
4. Нажала «Опубликовать» → получила рабочую ссылку вида `/mg/m/<slug>`
5. Отправила ссылку клиенту → клиент видит чистую визитку + кнопку «Записаться»

**Для Инны (админа):**
1. Открыла админку → ввела пароль
2. Увидела очередь новых анкет (realtime, без F5)
3. Прочитала анкету → нажала Approve или Reject с комментарием
4. Увидела собранные страницы всех одобренных мастеров

**Для клиента мастера (участник):**
1. Перешёл по ссылке → увидел чистую страницу-визитку
2. Не видит никаких админок, номеров «Мастер N», токенов в URL
3. Нажал кнопку «Записаться» → либо перешёл на контакт мастера, либо оставил заявку через форму (мастер выбирает в конструкторе)

## Главный продуктовый эффект

**ВАУ-момент на ближайшей встрече УБ:** участница за одну встречу проходит анкету → одобрение → конструктор → публикацию → **получает живую ссылку на свою страницу, которую может прямо здесь и сейчас отправить клиенту**.

Это должно вызвать у остальных участниц реакцию «я тоже так хочу» и послужить триггером задуматься о цифровизации собственного бизнеса.

## Тип продукта и стек

- **Тип:** веб-приложение с базой (из матрицы Playbook, раздел 3.2)
- **Стек:** HTML + CSS + JavaScript, без фреймворков, без CDN. Supabase JS v2 через CDN
- **Бэкенд:** тот же проект Supabase `my-architecture` (eu-west-2, Transaction pooler), новые таблицы `mg_applications`, `mg_master_pages`, `mg_config_questions`, `mg_leads` — рядом с существующими `mg_*`
- **Мигратор:** `node scripts/supabase-migrate.mjs --client mastergroup`
- **Брендбук:** светлая нейтральная палитра Инны (белый фон, горчичный/золотой акценты, зелёный support, серый текст) — см. [design-system.md](../../knowledge/standards/design-system.md)

- `admin.html` получает новые табы «Анкеты» и «Собранные страницы»

## Чего НЕ будет в MVP (осознанно отложено)

Этот раздел — контракт. После подтверждения Инной — ничего отсюда не добавляем без отдельного разговора.

- ❌ **Платёжка, тарифы, подписки, freemium-ограничения.** Всё бесплатно в MVP.
- ❌ **Выбор темы оформления.** Одна светлая тема по брендбуку Инны. Мастер не выбирает.
- ❌ **Загрузка фото в Supabase Storage.** Только URL внешнего изображения (Яндекс.Диск, Google Photos, любой публичный JPG/PNG).
- ❌ **Telegram Mini App** — отложено на «после первой обкатки». Инна подтвердила: «на перспективу, сейчас один продукт».
- ❌ **Собственный Telegram-бот мастера 24/7.** Отложено.
- ❌ **Сайт-витрина без бэкенда** (как альтернативный формат). Отложено.
- ❌ **Supabase Auth для мастеров** (email/пароль, magic link). Доступ — по токену в URL, как v1.
- ❌ **Supabase Auth для админа.** Инна подтвердила: «как в v1 — пароль в окошке». SHA-256 хэш пароля в коде `admin.html`.
- ❌ **Редактирование/удаление старых материалов и заданий** из v1. Старые табы кабинета не трогаем.
- ❌ **Автопубликация одобренных страниц в каталог.** Пока только по прямой ссылке, без публичного списка «все мастера».
- ❌ **E-mail/Telegram-нотификации** Инне о новой анкете. Только realtime в админке — Инна заходит сама.
- ❌ **Аналитика просмотров** страницы мастера.
- ❌ **Импорт из Taplink / Tilda / других** конструкторов.
- ❌ **12 вопросов конструктора.** Пока только 8 — минимальный набор для «живой страницы». Остальное — в v2.1.
- ❌ **Live preview в реальном времени** при заполнении конструктора. В MVP — кнопка «Посмотреть» открывает страницу в новом окне. Возможно, добавим live preview, если времени хватит — но это не обязательно.
- ❌ **Мульти-фото галерея / видео-блок / FAQ-секция.** Только hero + about + offer + CTA.
- ❌ **Редактирование номера мастера после присвоения.** Номер фиксированный. Мастер может только заменить его на ФИО в UI (через галочку «использовать настоящее имя»), но системный номер остаётся в админке.
