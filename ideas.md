# ideas.md — копилка идей агентства

Сюда падают три типа записей:
1. **Продуктовые идеи** — что можно построить и продать (MINI-N)
2. **Потенциальные клиенты** — конкретные люди/компании, кому предложить продукт
3. **Архитектурные идеи** — изменения в самом агентстве (новые агенты, процессы, инструменты)

Этот файл читается Аналитиком и Директором перед планированием. Когда идея созревает для конкретного клиента — она переезжает в его `clients/{имя}/PLAN.md` и помечается здесь как перенесённая.

---

## 🔁 Жизненный цикл идеи (pipeline)

```
💡 СЫРАЯ        →  📝 К РАССМОТРЕНИЮ   →  🎯 СОЗРЕЛА ДЛЯ КЛИЕНТА  →  ✅ В РЕАЛИЗАЦИИ
(набросок)          (детали продуманы,     (есть конкретный клиент,    (переехала в
                     ждёт своего часа)      можно предложить)            clients/{}/PLAN.md)
```

**Правило переноса:** как только идея привязывается к конкретному клиенту, она:
1. Добавляется в `clients/{имя}/PLAN.md` как задача
2. Здесь в ideas.md помечается статусом `✅ → clients/{имя}/ (дата)`
3. Не удаляется — остаётся как трейл, чтобы было видно откуда идея пришла

**Правило апгрейда:** когда собирается 8-10 сырых идей — Аналитик группирует их в линейку → становится основой структуры HUB сайта или новой продуктовой категории.

---

## 💡 Паттерн работы: Клиент → Шаблон → Продукт

Строим для одного клиента → шаблонизируем → продаём следующему.
Это главный способ как идеи становятся продуктами: сначала решаем чью-то конкретную боль, потом упаковываем в повторяемый шаблон.

---

## 📝 Шаблон новой идеи

```markdown
### MINI-N: Название
- **Статус:** 💡 Сырая | 📝 К рассмотрению | 🎯 Созрела | ✅ → clients/{имя}/ (дата)
- **Пример:** [Конкретный первый случай использования]
- **Шаблон:** [В какой повторяемый продукт это превращается]
- **Кому ещё:** [Список ниш/типов бизнеса кому пригодится]
- **Почему сейчас / почему потом:** [Что решает внедрение этой идеи]
- **Связано с:** [Ссылки на другие идеи, клиентов, документы]
```

- **Применять для:** Все клиенты которые общались через WhatsApp и чьи клиенты (B2C) — российская аудитория. Особенно пожилые, детские, социальные направления.

## 🏗 Архитектурные идеи (изменения в самом агентстве)

Идеи из мега-промпта от коллеги Инны (апрель 2026). Отложены, потому что текущая архитектура работает и ломать её ради этого нет смысла. Вернёмся, когда появится реальная потребность.

### АРХ-1: Отдельный агент Copywriter
- **Статус:** 💡 Сырая
- **Что:** Отделить продающий копирайтинг (лендинг-копирайтинг, email-воронки, sales-страницы) от контент-маркетинга (посты, SMM)
- **Почему отложено:** Сейчас объёмы не требуют разделения. `agents/content.md` справляется.
- **Когда внедрять:** Когда появится хотя бы 2-3 клиента с выраженной потребностью в продающем копирайтинге или когда мы возьмём клиента с воронкой продаж
- **Как внедрять:** Создать `agents/copywriter.md` по образцу existing агентов, добавить в Директор

### АРХ-2: Frontend Developer агент (React/Vue)
- **Статус:** 💡 Сырая
- **Что:** Отдельный агент для сложных веб-приложений на React/Vue/Next.js — не для простых HTML-страниц
- **Почему отложено:** Текущее правило «один самодостаточный HTML» перекрывает 80% задач. React избыточен.
- **Когда внедрять:** Когда возьмём клиента с веб-приложением где реально нужен фреймворк (например, админка с кучей состояний)
- **Конфликт:** Противоречит корневому правилу «никаких фреймворков» — придётся обновить CLAUDE.md

### АРХ-3: Backend Developer агент (Node/Python + БД + API)
- **Статус:** 💡 Сырая
- **Что:** Отдельный агент для серверных задач: API, БД, аутентификация, интеграции
- **Почему отложено:** Первый реальный бэкенд — Beauty. Когда дойдём до его реализации, поймём нужен ли отдельный агент или справляется Systems.
- **Когда внедрять:** Во время Backend фазы Beauty Визитки (см. [clients/beauty/BACKEND-PLAN.md](clients/beauty/BACKEND-PLAN.md))

### АРХ-4: Общие компоненты (/shared/components/)
- **Статус:** ❌ Отклонено (на данный момент)
- **Что:** Вынести повторяющиеся HTML-куски (header, footer, формы) в общие переиспользуемые компоненты
- **Почему отклонено:** Прямо противоречит корневому правилу «каждая страница — один самодостаточный HTML». Введение общих компонентов требует или препроцессора (не вяжется с ванильным HTML), или Jinja/EJS-подобного шаблонизатора (сложность).
- **Альтернатива:** Копировать между страницами вручную пока их мало. Когда будет 20+ похожих страниц — пересмотреть.

## Как этот файл обновляется

- **Новая идея:** добавить в соответствующий раздел по шаблону выше, начальный статус `💡 Сырая`
- **Идея созрела:** поменять статус на `📝 К рассмотрению` или `🎯 Созрела для клиента`, дописать детали
- **Идея уходит к клиенту:** поменять статус на `✅ → clients/{имя}/ (дата)`, добавить задачу в `clients/{имя}/PLAN.md`
- **Идея отклонена:** поменять статус на `❌ Отклонено`, дописать причину
- **Раз в месяц:** Аналитик просматривает весь файл и предлагает Инне решения по сырым идеям
