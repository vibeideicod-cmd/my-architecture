> Бренд формирует продукты → опираются на команду → выходят во внешний мир → упакованы на сайте → клиент покупает и попадает в Платформу → всё работает на инфраструктуре

> Инфраструктура держит платформу → платформа ведёт клиента → клиент пришёл с сайта → AI квалифицировал → данные усиливают продукт и бренд

### 1.3. Целевая аудитория

> *«Для тех, кто живёт в бизнесе 24/7 и тянет его на себе»*

**Ниша — вторична. Первичен фильтр по состоянию клиента.** Один человек тянет всё на себе, соцсети заброшены, автоматизации нет, времени разбираться — нет.

**Приоритетные ниши (8):**
1. Инфобизнес и онлайн-образование
2. Финансовые консультанты и брокеры
3. Бьюти (салоны, барбершопы, косметология, СПА)
4. Коучи, психологи, наставники
5. Фитнес-клубы и тренеры
6. Торговля — офлайн и онлайн (магазины, интернет-магазины)
7. Риэлторы и агентства недвижимости
8. Санатории и отели

**Фильтр по клиенту (важнее ниши):**
- Платёжеспособный — не ищет «самое дешёвое»
- Готов вкладываться в развитие — не ждёт «волшебную таблетку»
- Возраст обычно 30–60 лет
- Ценит включённость и долгосрочные отношения

Детальные портреты (сегменты А1, А2, B1, B2, C по готовности к AI; разрезы по 8 нишам): `inna/audience.md`.

### 1.8. Принципы выручки и масштабирования

**Цель по выручке:** наращивать долю рекуррента. Много подписок + много решений с сопровождением — «курочка по зёрнышку». Меньше зависимости от потока новых клиентов, больше стабильного дохода от действующих. Главный рекуррентный продукт — Платформа С.С.С.Р. (Слой 6).

**Принцип масштабирования:** один раз делаю решение для себя → тиражирую клиентам с адаптацией. Не продаю время — продаю сопровождение внедрения готового решения.

### 3.1. Нейро-команда (30 агентов)

**Как работает команда:** Директор управляет только 10 руководителями департаментов. Руководитель департамента знает своих специалистов и скиллы. Директор + руководитель + специалист закрывают 95% задач.

**В роадмапе:** Client Success · Accountant · Partner.

### 3.2. Принципы работы команды

**Директор — маршрутизатор верхнего уровня.** Все задачи входят через Директора. Он управляет только **10 руководителями департаментов** — не каждым специалистом и не каждым скиллом напрямую. Руководитель департамента знает своих специалистов и их инструменты — и принимает решения внутри своего домена.

**Принцип иерархии:**

```
Директор
   ↓ управляет только 10 руководителями
Руководитель департамента
   ↓ управляет своими специалистами и скиллами
Специалист (агент) со своими скиллами
   ↓ выполняет конкретную задачу
Результат
```

**Скилл = инструмент агента.** Скилл живёт отдельным файлом (чтобы можно было переиспользовать), но **принадлежит конкретному агенту**. В каждом агент-файле есть раздел `### Скиллы` со ссылками на его инструменты. Директор не держит каталог скиллов в голове — он знает только руководителей департаментов.

**Два режима работы:**

| Режим | Когда | Как |
|---|---|---|
| **Линейный** | Стандартная задача | Директор → руководитель департамента → специалист + скилл → результат |
| **Параллельный** | Сложная задача, несколько доменов | Директор → несколько руководителей одновременно → Директор собирает результат |

**Типичный рабочий цикл:**

```
Задача от Инны
   ↓ Директор читается первым (обязательно)
   ↓ Определяет домен → передаёт руководителю департамента
   ↓ Руководитель выбирает нужного специалиста и/или скилл
   ↓ Специалист выполняет через свой скилл
   ↓ Директор контролирует качество (QA-чек)
   ↓ Результат Инне
```

**Правила взаимодействия:**
- Директор читается **до** любого ответа на задачу — не после
- Директор знает только руководителей департаментов; руководитель знает своих специалистов и скиллы
- Агент работает только в своём домене; выход за рамки — через Директора
- При конфликте приоритетов (контент vs продажи, бренд vs юридика) — Директор принимает решение
- Финансовые и юридические решения — Financial + консультация Инны, не автономно

- **Конвертирует** — записывает на консультацию / отдаёт лид-магнит / ведёт на покупку

**Сценарий работы:**

```
AI-помощник Инны (на сайте HUB)
   ↓ задаёт 4-6 вопросов
   ↓ определяет: боль + ниша + сегмент A/B/C
   ↓ запускает нужный сценарий:
       → По боли → лендинг этой боли → продукт по этой боли (Слой 2)
       → Эксперт + готов платить → Платформа С.С.С.Р. (Слой 6)
       → Крупный заказ → Нейро-архитектура под ключ (флагман)
   ↓ финал: запись на встречу / лид-магнит / покупка / подписка
   ↓ контакт → Моя база клиентов (Слой 7)
```

**Тексты диалога** (первое сообщение, тон, сценарий вопросов): `inna/website.md`
**Каталог продуктов:** `inna/product-line.md`
**Портреты ЦА (для тонкой настройки AI):** `inna/audience.md`

> 🌱 **ГЛАВНАЯ СИЛА AI-ПОМОЩНИКА ИННЫ**
>
> Он не просто помощник на сайте. Он — **живая витрина продукта №4** (AI-консультант под клиента). Клиент общается, видит как это работает, и думает *«мне такого же надо для своего бизнеса»*.

### 5.4. Лендинги по болям + кейсы по нишам — принцип

**Лендинги (5 шт. = 5 болей из Слоя 2):**
Одна страница = одна боль = одно обещание. Боль — основная ось навигации.

**Кейсы (фильтр по 8 нишам — см. Слой 1, п.1.3):**
1. Инфобизнес и онлайн-образование
2. Финансовые консультанты и брокеры
3. Бьюти
4. Коучи, психологи, наставники
5. Фитнес
6. Торговля офлайн и онлайн
7. Риэлторы
8. Санатории и отели

Человек заходит → читает лендинг по своей боли → переходит в кейсы → фильтрует по своей нише → видит «такие как я тоже работали с Инной». Социальное доказательство.

**Почему именно так:**
- Не делать 20 нишевых лендингов (адская поддержка)
- Не делать только один универсальный (низкая конверсия)
- Боль = ось продажи (5 шт.). Ниша = ось доверия (8 шт.).

**Связь с продуктовой линейкой:** каждый лендинг боли ведёт к продуктам этой боли из `inna/product-line.md` (вход → дёшево → основное → подписка → флагман).

**Источники для текстов лендингов:**
- `inna/audience.md` — портреты ЦА, боли в словах клиента, язык
- `inna/positioning.md` — голос бренда, антипримеры, тексты позиционирования
- `inna/website.md` — структура страниц, готовые блоки, диалоги AI-помощника

**Источник вдохновения:** Stripe / Notion / Webflow делают страницы под use case — у них разрез по применению, у нас — по боли.

## Слой 6: Платформа С.С.С.Р.

> **Закрытый подписной продукт · 4 модуля · AI в контексте клиента**

То, что покупает клиент сегмента A. Сайт продал — клиент попадает сюда. Здесь Инна ведёт его по методологии С.С.С.Р. через систему, а не вручную.

### 6.1. Что это и для кого

Закрытая подписная платформа для **сегмента A** — экспертов, у которых уже есть продукт и клиенты, но нет системы. Они покупают подписку → получают личный кабинет → работают с поддержкой AI и еженедельных разборов.

Детальные портреты сегмента A (А1 «Архитектор системы», А2): `inna/audience.md`.
Платформа в каталоге продуктов (как подписной флагман): `inna/product-line.md` → раздел «Хочу всё» / «Нет структуры».

**Принцип:** Инна не ведёт каждый кабинет вручную. Автоматизации (N8N) делают рутинное — онбординг, напоминания, AI-отчёты. Инна включается точечно (разборы для всего потока, VIP-сессии).

> ⚖️ **ЮРИДИЧЕСКИЙ СТАТУС**
>
> Платформа С.С.С.Р. = **абонентское консультационное обслуживание по ОКВЭД 70.22**.
>
> AI-помощник, автоматизации, личный кабинет, модули — это **инструменты**, через которые оказывается услуга. Сама услуга = провожу аудит, даю рекомендации, сопровождаю внедрение клиентом своих решений с использованием AI-инструментов.
>
> Нельзя продавать «доступ к AI» как самостоятельный товар. Можно продавать «консультацию с использованием AI-инструментов». См. также `inna/positioning.md` и Слой 1, п.1.6.

### 6.2. 4 модуля

| Модуль | Что внутри | Боль |
|---|---|---|
| **Структура** | Аудит бизнеса, схема процессов, план внедрения | Нет структуры |
| **Стратегия** | Стратегия роста, позиционирование, продуктовая линейка | Не вижу картину |
| **Свобода** | AI-инструменты, автоматизации, делегирование | Нет жизни вне бизнеса |
| **Результат** | Трекинг прогресса, AI-отчёты, разборы | Работаю много — системы нет |

### 6.3. Тарифы (черновик)

| Тариф | Что входит | Юридическая формулировка (ОКВЭД 70.22) |
|---|---|---|

| **VIP** | Работа + индивидуальные сессии с Инной + приоритет | Абонентское консультационное обслуживание + индивидуальное экспертное сопровождение |

### 6.4. Путь клиента

```
ВХОД          → покупка подписки → профиль создаётся в Моей базе клиентов
ОНБОРДИНГ     → анкета → AI знает контекст этого клиента
РАБОТА        → модули + AI отвечает в контексте клиента
РАЗБОРЫ       → еженедельный Zoom-разбор + AI-отчёт по прогрессу
ПРОЛОНГАЦИЯ   → клиент видит результат → продлевает или апгрейдится
```

> **VPS · N8N · Моя база клиентов · Автоматизации**

Техническая база, которая обслуживает все остальные слои. Клиент её не видит — она просто работает.

### 7.1. Серверная инфраструктура

| Компонент | Где | Роль |
|---|---|---|

### 7.2. Моя база клиентов

Минимальная **CRM на нашем сервере** рядом с N8N на VPS:
- Своя таблица клиентов, выглядит и работает как Google Таблицы — но данные хранятся у нас на сервере
- Имя проекта: **Моя база клиентов**, технический алиас для папок/репо: `my-bz-klient`
- Контакты из AI-помощника Инны и форм сайта → N8N → Моя база клиентов
- ФЗ-152 соблюдён (российский сервер)
- Строим через vibe coding, без сторонних платных сервисов
- Конкретный движок выбирает Systems-агент в момент реализации (есть готовые open-source решения)

### 7.3. N8N — роль связующего звена

```
AI-помощник Инны     → webhook → N8N → Моя база клиентов
Сайт-форма           → webhook → N8N → Моя база клиентов + уведомление в TG
TG-бот               → N8N → Моя база клиентов
Платформа С.С.С.Р.   → N8N → онбординг, напоминания, AI-отчёты
Моя база клиентов    → N8N → рассылка / follow-up (в перспективе)
```
