## 3. Воронка — как клиент попадает в платный продукт

### 3.1. Главная модель воронки на старте — «Сарафан → автоматизация»

Это **главный сдвиг** по сравнению с инфоцыганской воронкой («холодный таргет → лид-магнит → email → tripwire»). У Инны на старте:
- Тёплый круг ≈7000 в VK + мастера МГ + ученики мастеров + активные Zoom-контакты + сарафан + ЛС VK.
- Холодной email-базы и текущего контент-ритма в VK на старте **нет**.
- Сарафан = клиенты приходят по рекомендациям и личным касаниям.

**Главное действие пользователя на HUB:** запросить **обратную связь / разбор** от Инны (не «купить курс»).

### 3.2. Каналы входа

| Канал | Что от него ожидаем |
|---|---|
| **Канал входа №1 — личное общение Инны** (Zoom, ЛС VK, мероприятия) | Контакт сразу попадает в `crm.*` через n8n, помечается источник «личное» |
| **Канал входа №2 — посты VK (реанимация) + TG-канал (запуск с тёплого круга)** | Точка сборки — HUB (через ссылку в посте / закреп / меню группы / TG-канал → бот → магнит → форма) |
| **Канал входа №3 — рекомендации** (мастера МГ, ученики, существующие клиенты) | Ссылка на HUB / визитку → AI-помощник → форма обратной связи |

### 3.3. Точка сборки и продуктовая лестница

```
Канал входа (1, 2 или 3)
     ↓
HUB (главная или лендинг боли)
     ↓
AI-помощник Инны (5 вопросов: ниша → боль → уточнение → готовность A/B/C → предложение)
     ↓
Маршрутизация:
   ├─ В лид-магнит /podarok/<имя>/    ← email-цепочка после магнита
   ├─ На лендинг боли + продукт боли  ← из inna/product-line.md
   ├─ Запись на бесплатный разбор     ← Zoom-консультация Инны (мост к платному)
   ├─ Платформа С.С.С.Р.              ← если эксперт A1-A2
   └─ Нейро-архитектура под ключ      ← если крупный заказ
     ↓
Контакт → «Моя база клиентов» (crm.*) ← фиксирует источник
```

### 3.4. Лестница C→B→A как механизм продукта (новое осознание 09.05)

## 4. Юр.контур и 3 ступени обкатки

### 4.1. ОКВЭД 70.22 — железное правило

Все публичные тексты, оферта, чеки, лендинги, AI-помощник — **только** через формулу «Аудит → Рекомендации → Сопровождение внедрения». Подробно — `inna/website.md` раздел «Юридическая лексика», `inna/positioning.md` Часть IV.

### 4.2. Юр.переезд через ~месяц

```
Сейчас (тестовая фаза, ~до конца мая 2026)
- Beget-аккаунт `icepaeqw_hub@icepaeqw.beget.tech` Иринин (самозанятая)
- Все ресурсы (shared, оба VPS, домены) на её аккаунте
- Домен `ideidlyabiznesa1913.ru` зарегистрирован на физическое лицо
- Архитектура разрабатывается в `/preview/` на текущем аккаунте

Через ~месяц
- Инна заводит свой Beget-аккаунт под ИП
- Переоформляет домен на ИП
- Переоформляет VPS (или закрывает Model Zarek)
- Запускается публично уже на своём аккаунте
- Stydiyatsi.ru остаётся за Ириной полностью
```

DevOps определяет **окно переезда** (до запуска или в паузу между запусками поддоменов, не середина активной работы).

### 4.3. 3 ступени обкатки (с уточнением 09.05)

```
1. ИННА — делаю себе → тестирую → дорабатываю
              ↓
   Параллельно (не блокирует):
              ↓
2. ИРИНА — зеркальная сборка под её нишу (бренд-дизайн)
              ↓
3. КЛИЕНТЫ — продаём как флагман «Нейро-архитектура под ключ»
   с двумя кейсами «Инна» и «Ирина»
```

**Самозапуск Инны = первый публичный кейс продукта** (не просто обкатка). Каждый шаг — будущий маркетинговый материал: скриншоты «было/стало», цифры (рост охвата, лидов, обращений), сами процессы.

**Уточнение 09.05:** 2-я ступень (Ирина) идёт **параллельно**, не **последовательно** перед клиентами. Когда Инна готова — выходим на платных клиентов из тёплого круга, не ждём Ирину.

## 7. Принципы (must-keep, не урезаем)

1. **AI-помощник на корне HUB = главная фишка.** Виджетом на всех страницах, юр.чисто, диагностика по С.С.С.Р. + маршрутизация по 5 болям × 8 нишам × A/B/C сегменту. См. `key-decisions.md`.
2. **Один поддомен = одна функция.** Не дублируем (грабли Гандапаса из Research v1).
3. **Свой VPS, не GetCourse / Bizon365.** Соответствие позиционированию и ОКВЭД 70.22.
4. **Поэтапный запуск экосистемы**, не одновременный. По этап-карте п.2.5.
5. **`mini.*` под ОКВЭД 70.22** — обязательная плашка «демо, не продукт» + noindex.
6. **`bz.*` = один источник, два интерфейса** (read-only зеркало `knowledge/`).
7. **Vultr London VPN изолирован** от продуктовой инфры (ФЗ-152).
8. **Принцип живой витрины** — Слой 1 п.1.7. Сначала строю себе → потом предлагаю клиентам.
9. **Технологии показываем цветом, не словами** — главное правило бренда (Слой 1 п.1.6).
10. **Расширяем, не урезаем** — все идеи сохраняются в архитектуре, переносятся в очередь, не отрезаются.
11. **Фокусные документы обновляем одновременно** с обсуждением (новое правило 09.05).
12. **Темп — дни и часы, не недели** через vibe coding и параллельную работу агентов.
13. **«Начинаем с этапа клиента»** через лестницу C→B→A — каждое предложение под этап.
14. **3 ступени обкатки параллельно** (Ирина не блокирует выход Инны на клиентов).
