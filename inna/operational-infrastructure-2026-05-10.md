## 5. Почему такое распределение — три причины

### Причина 1 — техника

Cheerful Marik **физически не вытянет** ещё 6 поддоменов поверх n8n+Baserow:

| Параметр | Cheerful Marik сейчас | Model Zarek сейчас |
|---|---|---|
| Диск | 64% занят (11 ГБ свободно из 29) | 26% занят (21 ГБ свободно из 29) |
| RAM | 1.5 ГБ из 1.9 ГБ занято | 0.4 ГБ из 1.9 ГБ занято |
| Запущенные сервисы | n8n + Baserow + traefik + nginx | nginx и certbot не установлены, чисто |

PostgreSQL для CRM + Платформа + AI-движок на Cheerful Marik = OOM-killer (out-of-memory) убьёт боевой n8n клиента Ирины.

Model Zarek чистый, специально под нашу задачу.

### Причина 2 — изоляция рисков

Cheerful Marik = **боевой клиентский сервер**. Если при разработке наших поддоменов что-то сломаем — упадёт **бот клиента Ирины**.

На Model Zarek можно ломать сколько угодно — там только наше.

### Причина 3 — нельзя повышать тариф Beget

По решению `feedback_no_beget_tariff_upgrade.md` — тариф Beget shared не повышаем. Поэтому статичные поддомены идут не на shared, а на VPS Model Zarek.

## 7. Лимит 5 = сайты, не DNS-записи

На тарифе Beget shared лимит **«5 сайтов»**. Это количество **физических папок с файлами** под отдельные домены/поддомены. Папка = сайт = слот.

DNS-запись — это **строка в таблице DNS-зоны**: «поддомен `inna` → IP `31.128.41.17`». Никакой папки на Beget shared при этом не создаётся. Файлы лежат на Model Zarek.

```
Beget shared (тариф «5 сайтов»)              DNS-зона ideidlyabiznesa1913.ru
─────────────────────────────────             ────────────────────────────────
СЛОТ 1:  ideidlyabiznesa1913.ru ◄─── @  ──► IP Beget shared        (1 слот)
СЛОТ 2:  stydiyatsi.ru          ◄─── @  ──► IP Beget shared        (1 слот)
СЛОТ 3:  demo.* (icepaeqw_demo) ◄─── demo  ─► IP Beget shared      (1 слот)
СЛОТ 4:  свободно                                                            
СЛОТ 5:  свободно                                                            

(не на Beget — DNS-записи на VPS, файлы НЕ на Beget)
                                       inna       ──► 31.128.41.17  ← 0 слотов
                                       mini       ──► 31.128.41.17  ← 0 слотов
                                       ai         ──► 31.128.41.17  ← 0 слотов
                                       crm        ──► 31.128.41.17  ← 0 слотов
                                       bz         ──► 31.128.41.17  ← 0 слотов
                                       platforma  ──► 31.128.41.17  ← 0 слотов
                                       ain8n      ──► 45.9.41.80    ← 0 слотов
```

**Итог: добавление 6 A-записей на Model Zarek не съедает ни одного слота на Beget shared.**

## 8. Способ создания DNS-записей — A правильный, B нет

Beget даёт два способа создать поддомен. **Они РАЗНЫЕ.**

### ✅ Способ A — правильный (для нас)

```
cp.beget.com
   → Домены и поддомены
   → ideidlyabiznesa1913.ru
   → Управление DNS-записями
   → Добавить запись
   → Тип: A, Имя: <поддомен>, Значение: <IP>
```

- Создаётся **только строка в DNS-таблице**
- Файлов и папок не создаётся
- **Слот не съедается**
- Нужно для всех 6 наших поддоменов на Model Zarek

### ❌ Способ B — неправильный (НЕ ДЕЛАТЬ)

```
cp.beget.com
   → Сайты
   → Создать сайт
   → Поддомен
```

- Создаётся **папка `~/<sub>/`** на Beget shared
- Создаётся DNS-запись на IP Beget shared (не на Model Zarek)
- **Съедает слот** из 5
- **Не работает** для нас — нам нужен IP Model Zarek, а Beget автоматически подставит свой IP

**Правило:** для всех 6 поддоменов используем **только Способ A**. Способ B Beget может предлагать как «удобный» — кликать его не нужно.

### 10.3. Что НЕ трогаем в панели

| Запись | Куда смотрит | Почему не трогаем |
|---|---|---|
| `@` (корень) | IP Beget shared | HUB корень, не переезжает |
| `demo` | IP Beget shared | Остаётся как сейчас, переезд отложен |
| `ain8n` | `45.9.41.80` | Уже работает, n8n живёт там |
| `www` (если есть) | IP Beget shared | Алиас на корень, не трогаем |
| MX-, TXT-, NS-записи | (как есть) | Это почта и техническое — не наша задача |

### 10.4. Проверка после добавления записей

После сохранения 6 A-записей — на маке Инны проверить распространение DNS:

```bash
dig +short inna.ideidlyabiznesa1913.ru
dig +short mini.ideidlyabiznesa1913.ru
dig +short ai.ideidlyabiznesa1913.ru
dig +short crm.ideidlyabiznesa1913.ru
dig +short bz.ideidlyabiznesa1913.ru
dig +short platforma.ideidlyabiznesa1913.ru
```

Каждая команда должна вернуть `31.128.41.17`. Распространение DNS занимает обычно 5-30 минут (TTL 3600 = 1 час).

Только после этого можно выписывать SSL-сертификаты на Model Zarek (иначе Let's Encrypt не получит подтверждения по HTTP-01 challenge).

## 18. Технические решения для Этапа 1 (закрыто командой 10.05)

### 18.1. SSL — wildcard через Cloudflare DNS

**Решение:** `*.ideidlyabiznesa1913.ru` — один wildcard-сертификат на все поддомены через Cloudflare DNS-01 challenge.

**Что делаем:**
1. Регистрируем бесплатный аккаунт Cloudflare
2. Добавляем домен `ideidlyabiznesa1913.ru` в Cloudflare
3. В панели Beget shared меняем NS-записи домена на NS Cloudflare (2 строки)
4. Cloudflare автоматически переносит все существующие DNS-записи (`@`, `demo`, `ain8n`, `stydiyatsi.ru`)
5. Управление DNS теперь в панели Cloudflare (а не Beget)
6. На Model Zarek настраиваем `certbot-dns-cloudflare` плагин с API-токеном Cloudflare
7. Получаем wildcard-сертификат `*.ideidlyabiznesa1913.ru` + базовый `ideidlyabiznesa1913.ru`
8. Сертификат автоматически обновляется каждые 90 дней без участия Инны

**Почему так:**
- Один сертификат покрывает все поддомены (текущие 7 + любые будущие). Добавление нового поддомена = одна A-запись, без возни с SSL.
- Cloudflare DNS — индустриальный стандарт. Бесплатный, uptime 99.99%+.
- ФЗ-152 не нарушается: через Cloudflare DNS не идут персональные данные. Cloudflare DNS = справочник «домен → IP», там нет содержимого, нет форм, нет переписки. Сами сайты лежат на наших серверах в России.
- Beget DNS API ненадёжен (нет официального плагина certbot, ручной hook ломкий) — отвергнут.

**Что Инна делает руками:** регистрация Cloudflare-аккаунта + смена NS в панели Beget (5 минут).

### 18.2. Пользователь на Model Zarek — `deploy` без sudo

**Решение:** создать пользователя `deploy` (login через ssh-ключ Инны), без sudo. Владелец папок `/var/www/<sub>/` — `www-data` (стандарт). `deploy` имеет права записи в эти папки через группу `www-data`. Root-доступ остаётся только для админских операций (Director через ssh-алиас `model-zarek`).

**Почему так:**
- Стандарт безопасности: ssh-ключ для деплоя статики не должен иметь админских прав.
- Если ключ когда-нибудь утечёт — атакующий не может удалить сервер, перезаписать systemd-юниты, поломать nginx. Только подменить файлы статики (восстанавливается из репо за 30 секунд).
- Изоляция: будущие deploy-скрипты для статики `inna.*`, `mini.*`, `bz.*` работают под `deploy`, не имеют прав трогать `crm.*`, `ai.*`, `platforma.*` (там нужен root).

**Что Инна делает руками:** ничего, deploy-скрипты настраиваются автоматически.

### 18.3. Nginx-конфиги в репо — `deploy/zarek-nginx/`

**Решение:** все nginx-server-блоки и базовые конфиги хранятся в репо в папке `deploy/zarek-nginx/`. На сервере это копии — источник правды в репо, разворачиваются по deploy-скрипту.

```
deploy/
├── zarek-nginx/
│   ├── nginx.conf                  ← базовый конфиг (если правится)
│   ├── inna.conf                   ← server-блок для inna.*
│   ├── mini.conf                   ← server-блок для mini.*
│   ├── ai.conf                     ← server-блок для ai.*
│   ├── crm.conf                    ← server-блок для crm.*
│   ├── bz.conf                     ← server-блок для bz.*
│   ├── platforma.conf              ← server-блок для platforma.*
│   └── demo.conf                   ← готовый шаблон для demo.* (НЕ активируется на Этапе 1)
├── zarek-setup.sh                  ← скрипт первичной настройки сервера
└── zarek-deploy.sh                 ← скрипт раскатки конфигов с мака на сервер
```

**Почему так:**
- Единый стиль с уже существующими 9 deploy-скриптами в корне репо
- Все инфраструктурные файлы в одном месте — легко найти, легко аудировать
- Версионирование: изменение конфига = коммит в репо. Видна история всех правок nginx
- Откат любых изменений на сервере — через git checkout + редеплой

**Что Инна делает руками:** ничего, это техническая структура.
