# Ключевые решения архитектуры Инны — must-keep

> **Назначение:** короткий список решений, которые нельзя растерять при разработке. Когда команда (Websites, Systems, AI Builder, Copywriter, Branding) садится делать конкретный кусок — обязательно сверяется с этим файлом.

## 🟢 2026-05-09 — Никакой платной рекламы, бюджет на трафик = 0₽

**Решение:** Инна не использует платную рекламу ни в каком виде. Не VK Таргет, не Яндекс.Директ, не Telegram Ads, не закупка у блогеров и пабликов. Даже в собственных постах не делаем рекламные вставки.

**Почему критично:**
- Стратегия эксперта-консультанта под ОКВЭД 70.22 строится на **системе и доверии**, не на покупке трафика.
- У Инны уже есть тёплый круг (≈7000 VK + МГ + ученики + сарафан) — ему не нужны рекламные пуши.
- Реклама = деньги вперёд за непроверенную гипотезу. Контент + AI-помощник + автоматизация = строительство актива.
- Без рекламы tripwire-лестница (3 уровня платных мостов) **не нужна** на старте — она для холодного трафика, который у Инны не появляется через рекламу.

**Что НЕ делать:**
- ❌ Не предлагать «запустим рекламу на лендинг», «таргет на лид-магнит», «реклама в нишевых пабликах».
- ❌ Не закладывать бюджет на рекламу в финансовых расчётах (строка = 0 ₽).
- ❌ Не делать «обходные пути для VK через платный таргет» — обходим алгоритмы только бесплатными методами.
- ❌ Не делать рекламные вставки в собственных постах («купи / акция / скидка / успей»).
- ❌ Не строить полную tripwire-лестницу (990 → 3990 → 9990 → mid → флагман) на старте.

**Что делать:**
- ✅ Органический контент VK / TG / своих каналов.
- ✅ Сарафанное радио и рекомендации.
- ✅ Автоматизация прогрева (email-цепочка, AI-помощник, n8n).
- ✅ Личные продажи через тёплый круг.
- ✅ Лид-магниты как полезный материал за подписку (не «успей купить»).
- ✅ SEO и поисковая органика.
- ✅ Кейсы, отзывы, экспертный контент.
- ✅ Tripwire-структура остаётся **как опция на будущее**, если стратегия изменится.

**Источник:** решение Инны 2026-05-09 в чате прогона архитектуры. Подробности: `memory/feedback_no_paid_advertising.md`.

## 🟢 2026-05-09 — Vultr London VPN изолирован от продуктовой инфры

**Решение:** VPS Vultr London — это **только** наш личный VPN. Он **не участвует** в продуктовой экосистеме Инны (HUB, поддомены, AI, CRM, Платформа). Никакие данные продуктовой инфры через него не маршрутизируются.

**Почему критично:**
- Vultr London — **заграничный сервер**. По ФЗ-152 персональные данные граждан РФ должны храниться и первично обрабатываться на серверах в РФ.
- Любая маршрутизация ПД через лондонский VPS (даже как туннель) = риск нарушения ФЗ-152.
- VPN — это инструмент личного доступа Инны к интернету. Не инфраструктурный компонент бизнеса.

**Что НЕ делать:**
- ❌ Не использовать London VPN как точку доступа к `crm.*`, `bz.*`, `ain8n.*`, `platforma.*` («VPN-only» через London — запрещено).
- ❌ Не разворачивать на нём n8n, базу данных, AI-бэкенд, CRM, любую часть продуктовой инфры.
- ❌ Не направлять туда логи и бэкапы серверов с ПД.
- ❌ Не давать его IP в whitelist российских серверов как «доверенный источник».
- ❌ Не упоминать его в публичных архитектурных схемах продукта.

**Что делать:**
- ✅ Использовать London VPN строго по назначению — личный VPN Инны (и при необходимости подрядчиков, у которых нет работы с ПД клиентов).
- ✅ Защита внутренних поддоменов (`bz.*`, `ain8n.*`, `crm.*` админка) — через **BasicAuth + IP-whitelist российских IP** (домашний IP Инны, IP VPS Cheerful Marik / Model Zarek). Не через лондонский туннель.
- ✅ Если у подрядчика динамический IP — выдавать BasicAuth, без IP-whitelist для него.
- ✅ В `reference_servers.md` (auto-memory) явно зафиксировано: London — изолирован.

**Источник:** правка Инны 2026-05-09 в Шаге 4 + первичная фиксация в формулировке идеи Шаг 1 («Vultr London — это VPN-сервер, не часть продуктовой экосистемы»).

## 🟢 2026-05-09 — База знаний: один источник, два интерфейса

**Решение:** `bz.ideidlyabiznesa1913.ru` остаётся в карте. Это **не отдельная база**, а второй интерфейс к тому же контенту, что лежит в `my-architecture/knowledge/` репозитория. Источник один, читалки две.

**Архитектура:**
```
ИСТОЧНИК (правится только тут):
  my-architecture/knowledge/    ← markdown-файлы в репозитории

ИНТЕРФЕЙСЫ (читают тот же источник):
  1. VS Code на ноуте Инны          ← Claude и агенты читают, Инна редактирует
  2. bz.ideidlyabiznesa1913.ru       ← веб-сайт для чтения с телефона / подрядчиками
```

**Как технически:** статический генератор сайта из markdown (MkDocs / Docusaurus). При коммите в `knowledge/` → CI/cron-скрипт пересобирает сайт → выкладывает на `bz.*`. Веб-морда — read-only зеркало репозитория, секунда в секунду.

**Имя продукта:** «База знаний» — одно для всех интерфейсов. Не «БЗ», не «KB», не «вики».

**Почему критично:**
- Конфликт двух баз физически невозможен — источник один.
- Не нужна отдельная админка / CMS — правки только в VS Code через markdown.
- Подрядчики и команда могут открыть с телефона / без репозитория, не нужен доступ к git.
- Поиск, навигация, оглавление — генератор делает автоматически.

**Что НЕ делать:**
- ❌ Не редактировать веб-морду `bz.*` напрямую — там нет редактирования, только просмотр.
- ❌ Не заводить вторую базу знаний (Notion / Confluence / wiki.js) параллельно с `knowledge/`.
- ❌ Не дублировать контент между `knowledge/` и `bz.*` вручную — только через генератор.
- ❌ Не открывать `bz.*` для редактирования никому — это read-only.

**Что делать:**
- ✅ Все правки → только в `knowledge/` через VS Code.
- ✅ Генератор и публикация — через cron / GitHub Actions / коммит-хук.
- ✅ Доступ к `bz.*` — публичный (если нужно подрядчикам) или под BasicAuth (если только для команды). Решается отдельно при запуске.
- ✅ Если возникает потребность в редактировании с телефона — добавляем PR-flow через GitHub-веб-интерфейс, не редактирование на самом `bz.*`.

**Источник:** решение Инны 2026-05-09 в чате прогона доменной экосистемы.

## 🟢 2026-05-09 — AI-помощник на корне HUB = главная фишка

**Решение:** AI-помощник Инны живёт на главной странице HUB (`ideidlyabiznesa1913.ru/`) и доступен бесплатно случайному посетителю с улицы.

**Почему критично:**
Research 2026-05-09 показал — из 7 топовых РФ-экспертов (Высоцкий, Гребенюк, Батырев, Гандапас, Зимбицкий, Христосенко, Лайк Центр) **никто** не выставляет AI-чат «спросить эксперта» на главную. Все прячут AI внутрь платных тарифов. Это **белое пятно рынка** — наша уникальная воронка для ниши консалтинга.

**Что НЕ делать (запреты при разработке):**
- ❌ Не убирать AI-помощника на отдельный поддомен или страницу — он должен быть на главной.
- ❌ Не прятать AI за форму регистрации, email-gate, captcha до первого вопроса. Первое сообщение — без барьеров.
- ❌ Не заменять AI-помощника обычной формой обратной связи или чатом с человеком («Напишите нам»).
- ❌ Не делать AI-помощника платным или с лимитом вопросов на старте — это убивает воронку «попробуй → купи».
- ❌ Не убирать AI с лендингов по болям. Он должен быть везде, где есть точка контакта с лидом.

**Что делать:**
- ✅ AI-помощник Инны на главной + на каждом из 5 лендингов по болям + на странице каталога.
- ✅ Один движок (`ai.ideidlyabiznesa1913.ru` бэкенд), разные контексты (HUB-версия / Платформа-версия / демо-ниши).
- ✅ В позиционировании, в текстах для соцсетей, в КП — явно подсвечивать «AI-помощник 24/7, спроси что угодно про автосистемы для твоего бизнеса».
- ✅ AI представляется первым сообщением: «Привет, я AI-помощник Инны, виртуальный ассистент» (юридическая чистота + маркировка AI).

**Источник:** Research 2026-05-09, [agent-runtime/shared/2026-05-09-analyst-domain-architecture-research.md](../agent-runtime/shared/2026-05-09-analyst-domain-architecture-research.md), раздел 3.В и Выводы п.3.

## 🟢 2026-05-09 — Один поддомен = одна функция (дисциплина)

**Решение:** каждый поддомен экосистемы выполняет ОДНУ чёткую функцию. Не допускаем параллельных доменов с пересекающимся контентом.

**Почему критично:**
Research показал грабли Гандапаса — 4 параллельных домена (`radislavgandapas.com` + `gandopas.ru` + `radislav-gandapas.com` + `club.*`) с пересекающимся контентом. Пользователь не понимает, куда идти, конверсия падает.

**Что НЕ делать:**
- ❌ Не создавать второй поддомен под ту же функцию (например, два разных лендинга «упаковка бренда» на разных поддоменах).
- ❌ Не дублировать каталог продуктов на нескольких поддоменах.
- ❌ Не делать «новую версию» поддомена параллельно со старой — переезжаем целиком, старое отключаем.

**Что делать:**
- ✅ Жёсткая карта поддоменов: `inna.*` (визитка), `mini.*` (демо), `demo.*` (клиенты), `platforma.*` (платформа), `ai.*` (бэкенд), `crm.*` (моя CRM), `bz.*` (внутренняя БЗ), `ain8n.*` (n8n).
- ✅ При появлении нового продукта — определять, в какой существующий поддомен он встаёт. Новый поддомен заводим только если функция действительно отдельная.

**Источник:** Research 2026-05-09, раздел 3.А и Выводы п.2 (грабли Гандапаса).

## 🟢 2026-05-09 — Свой VPS, не GetCourse / Bizon365

**Решение:** платформа, AI, CRM, n8n живут на собственном VPS (Cheerful Marik сейчас → Model Zarek после перехвата). НЕ используем готовые SaaS-платформы для инфобизнеса.

**Почему критично:**
- Research подтвердил: топ-игроки рынка (Высоцкий `my.bbooster.online`, Лайк Центр `Toolbox`, Христосенко «Универиум») идут своим путём.
- GetCourse/Bizon365 = противоречие позиционированию «Архитектор автосистем». Использовать чужой SaaS = выдать себя за рядовую инфошколу.
- ОКВЭД 70.22 «консультирование с использованием AI-инструментов» — нужен полный контроль над AI и интеграциями, чужие SaaS этого не дают.

**Что НЕ делать:**
- ❌ Не уходить на GetCourse / Bizon365 / Антитренинги ради «быстрого старта».
- ❌ Не размещать платформу С.С.С.Р. на чужом домене (например, `inna.getcourse.ru`).
- ❌ Не делать ЛК подписчиков на сторонней SaaS.

**Что делать:**
- ✅ Своя разработка через vibe coding на нашем VPS.
- ✅ Tilda допустима **только** для одиночных лендингов в режиме MVP — но финальная версия HUB и поддоменов = на нашем VPS / Beget shared.
- ✅ При выборе технологий (Системы, AI Builder) — приоритет open-source решений (Docker, nginx, n8n, PostgreSQL).

**Источник:** Research 2026-05-09, раздел 4 «Технологии», варианты 1-3.

## 🟢 2026-05-09 — Поэтапный запуск экосистемы (не одновременный)

**Решение:** не строим все 8 поддоменов одновременно. Запуск идёт по триггерам, в порядке от блокирующих к опциональным.

**Почему критично:**
Поддержка 8 сервисов одновременно лопнет на 3-м месяце (на Cheerful Marik уже сейчас n8n+Baserow остановлены 5 дней назад, в `/root` хаос бэкапов). Нужно сначала навести порядок и поднять MVP, потом расширять.

**Порядок запуска:**

| Этап | Что | Триггер запуска |
|---|---|---|
| 1 | HUB корень + 1 лендинг боли | Сейчас — главный блокер бизнеса |
| 2 | `inna.*` визитка | Под VK-упаковку и QR (задача из VK-плана) |
| 3 | `crm.*` минимум | Перед запуском AI-помощника, чтобы лиды не терялись |
| 4 | `ai.*` бэкенд + AI на HUB | После Этапов 1-3 |
| 5 | `mini.*` с 1 демо | После запуска AI-помощника, как витрина продукта №4 |
| 6 | `demo.*` | По факту появления клиента в работе |
| 7 | `platforma.*` | После 3-5 первых платных подписчиков (не строим пустой кабинет) |
| 8 | `bz.*`, `ain8n.*` | Внутренние, можно отложить — поднимаются под BasicAuth |

**Источник:** Business-Architect анализ 2026-05-09, раздел «Реализуемо?» в чате.

## 🟢 2026-05-09 — `mini.*` под ОКВЭД 70.22, формула на странице

**Решение:** на каждой странице `mini.*/<имя>/` обязательная плашка «Это демонстрационный образец решения для аудита/консультации, не самостоятельный продукт».

**Почему критично:**
Если AI-помощник в `mini.*` работает как полноценный публичный продукт (отвечает 24/7 анонимным юзерам, выглядит как чат-сервис) — налоговая может квалифицировать как отдельную услугу не по 70.22.

**Что делать:**
- ✅ На каждой странице `mini.*/<имя>/` — плашка с формулой: «Демонстрационная среда. Образец решения, которое мы внедряем под клиента в формате сопровождения. Пройдите демо → получите рекомендацию AI-помощника на консультации.»
- ✅ Лимит на демо: 5 вопросов / Email-gate перед длинной сессией / время сессии 10 минут — что-то одно, чтобы было видно «это демо, а не сервис».
- ✅ В оферте сайта: «`mini.*` — демонстрационные среды, доступ безвозмездный, в рамках консультационных услуг по ОКВЭД 70.22».

**Источник:** Yuridika+PD-Lawyer черновой анализ (зафиксирован в чате до перестройки порядка), [memory/feedback_audit_order_business_first.md](../memory/feedback_audit_order_business_first.md).
