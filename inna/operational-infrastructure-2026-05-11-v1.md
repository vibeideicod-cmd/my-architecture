## 10. Почему такое распределение — три причины

### Причина 1 — техника

Cheerful Marik физически не вытянет ещё 6 поддоменов поверх работающего n8n+Baserow:

| Параметр | Cheerful Marik | Model Zarek |
|---|---|---|
| Диск | 64% занят (11 ГБ свободно из 29) | 26% занят (21 ГБ свободно из 29) |
| RAM | 1.5 ГБ из 1.9 ГБ занято | 0.4 ГБ из 1.9 ГБ занято |
| Запущенные сервисы | n8n + Baserow + traefik + nginx | nginx + certbot + 6 server-блоков |

PostgreSQL для CRM + Платформа + AI-движок на Cheerful Marik = OOM-killer убьёт боевой n8n клиента Ирины.

### Причина 2 — изоляция рисков

Cheerful Marik = **боевой клиентский сервер**. Если при разработке наших поддоменов что-то сломаем — упадёт бот клиента Ирины.

На Model Zarek можно ломать сколько угодно — там только наше.

### Причина 3 — нельзя повышать тариф Beget

По решению `feedback_no_beget_tariff_upgrade.md` — тариф Beget shared не повышаем. Поэтому статичные поддомены идут не на shared, а на VPS Model Zarek.

На Beget shared лимит 5 — это количество **физических папок** под отдельные домены/поддомены. Папка = сайт = слот.

```

─────────────────────────────────             ────────────────────────────────
СЛОТ 1:  ideidlyabiznesa1913.ru ◄─── @  ──► IP Beget shared        (1 слот)
СЛОТ 2:  stydiyatsi.ru          ◄─── @  ──► IP Beget shared        (1 слот)

(не на Beget — DNS-записи на VPS, файлы НЕ на Beget)
                                       inna       ──► 31.128.41.17  ← 0 слотов
                                       mini       ──► 31.128.41.17  ← 0 слотов
                                       ai         ──► 31.128.41.17  ← 0 слотов
                                       crm        ──► 31.128.41.17  ← 0 слотов
                                       bz         ──► 31.128.41.17  ← 0 слотов
                                       platforma  ──► 31.128.41.17  ← 0 слотов
```

**Итог:** 6 A-записей на Model Zarek не съели ни одного слота на Beget shared.

### Способ A правильный, способ B — НЕТ

Beget даёт два способа создать поддомен. Они РАЗНЫЕ.

**✅ Способ A — правильный (нашёл):**
```
cp.beget.com
   → Домены и поддомены
   → ideidlyabiznesa1913.ru
   → Управление DNS-записями
   → Добавить запись
   → Тип: A, Имя: <поддомен>, Значение: <IP>
```
- Создаётся только строка в DNS-таблице
- Файлов и папок не создаётся
- Слот не съедается
- Используем для всех 6 поддоменов на Model Zarek

**❌ Способ B — неправильный (НЕ ДЕЛАТЬ):**
```
cp.beget.com → Сайты → Создать сайт → Поддомен
```
- Создаётся папка `~/<sub>/` на Beget shared
- DNS-запись смотрит на IP Beget shared (не на Model Zarek)
- Съедает слот из 5
- Не работает для нас

**Beget может предлагать Способ B как «удобный» — кликать его не нужно.**

## 12. DNS — куда ведём управление

**Решение на Этапе 1:** DNS-зона `ideidlyabiznesa1913.ru` переведена с Beget DNS на **Cloudflare DNS** (через смену NS-записей в панели Beget).

| Зона | Где управляется |
|---|---|
| `ideidlyabiznesa1913.ru` (все A/CNAME-записи + SSL DNS-01 challenge) | Cloudflare (бесплатный аккаунт на `vibeideicod@gmail.com`) |
| `stydiyatsi.ru` (Иринин домен) | Beget (не переносили) |
| Сертификаты Let's Encrypt | Cloudflare DNS API через `certbot-dns-cloudflare` плагин |

**Почему Cloudflare:**
- Один wildcard `*.ideidlyabiznesa1913.ru` покрывает все поддомены, текущие и будущие. Новый поддомен = одна A-запись, без возни с SSL.
- Индустриальный стандарт, бесплатно, uptime 99.99%+.
- ФЗ-152 не нарушается: через Cloudflare DNS не идут персональные данные. DNS = справочник «домен → IP», там нет содержимого, нет форм, нет переписки. Сами сайты лежат на российских серверах.
- Beget DNS API ненадёжен (нет официального плагина certbot) — отвергнут.

**Почта для уведомлений Let's Encrypt:** `vibeideicod@gmail.com`.

## 14. Пользователь `deploy` на Model Zarek

**Решение Этапа 1:** для деплоя статики на сервере создан пользователь `deploy` (login через ssh-ключ Инны), **без sudo**. Владелец папок `/var/www/<sub>/` — `www-data` (стандарт). `deploy` имеет права записи через группу `www-data`.

Root-доступ остаётся только для админских операций (Director через ssh-алиас `model-zarek`).

**Почему так:**
- Стандарт безопасности: ssh-ключ для деплоя статики не должен иметь админских прав.
- Если ключ утечёт — атакующий не может удалить сервер, перезаписать systemd-юниты, поломать nginx. Только подменить файлы статики (восстанавливается из репо за 30 секунд).
- Изоляция: deploy-скрипты статики `inna.*`, `mini.*`, `bz.*` работают под `deploy`, не имеют прав трогать `crm.*`, `ai.*`, `platforma.*` (там нужен root).

## 15. Структура файлов в репо — `deploy/zarek-nginx/`

Все nginx-server-блоки и базовые конфиги хранятся в репо. На сервере это **копии**, источник правды в репо, разворачиваются по deploy-скрипту.

```
deploy/
├── zarek-nginx/
│   ├── nginx.conf                  ← базовый конфиг
│   ├── inna.conf                   ← server-блок для inna.*       (Этап 2 ✅)
│   ├── mini.conf                   ← server-блок для mini.*       (Этап 5)
│   ├── ai.conf                     ← server-блок для ai.*         (Этап 4)
│   ├── crm.conf                    ← server-блок для crm.*        (Этап 3 🔜)
│   ├── bz.conf                     ← server-блок для bz.*         (Этап 8)
│   ├── platforma.conf              ← server-блок для platforma.*  (Этап 7)
│   └── demo.conf                   ← шаблон для demo.* (НЕ активируется до Этапа 6)
├── zarek-setup.sh                  ← скрипт первичной настройки сервера
└── zarek-deploy.sh                 ← скрипт раскатки конфигов с мака на сервер
```

**Почему так:**
- Единый стиль с deploy-скриптами в корне репо
- Все инфраструктурные файлы в одном месте — легко найти, легко аудировать
- Версионирование: изменение конфига = коммит в репо. История всех правок nginx
- Откат любых изменений на сервере — через `git checkout` + редеплой
