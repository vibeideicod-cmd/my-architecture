### 2.1 🔴 ОКВЭД 70.22 — нет образовательной лицензии

**Решение:** ИП Инна Андрейченко имеет ОКВЭД 70.22 (консультирование). НИ ОДИН продукт нельзя называть «курс», «обучение», «школа», «тренинг», «урок», «ученик», «преподаватель», «академия», «мини-курс».

**Почему критично:**
- Налоговая и Роспотребнадзор активно мониторят инфобизнес 2024-2026. «Курсы без лицензии» = типовое нарушение, штраф + закрытие.
- ОКВЭД 70.22 покрывает всё что нужно (консультации, аудит, сопровождение). Главное — **называть ровно так**, не сбиваться на образовательную лексику.
- Применимо к продуктам Ирины тоже — она реализует, Инна сопровождает, оба под 70.22.

**Что НЕ делать:**
- ❌ Не называть продукты «курс», «обучение», «школа», «тренинг», «уроки», «академия», «мини-курс»
- ❌ Не использовать «преподаватель», «ученик» в текстах сайта/оферты
- ❌ Не позиционировать Платформу С.С.С.Р. как «образовательную платформу»
- ❌ Не писать «учебный материал», «учебная программа», «диплом»

**Что делать:**
- ✅ Называть продукты: консультация, аудит, разбор, сопровождение, методические материалы, экспертная сессия, чек-лист, шаблон, PDF-разбор, видео-разбор, диагностика
- ✅ Tripwire-форматы: методичка, PDF-разбор, чек-лист, видео-разбор экспертной сессии, шаблон + сопровождение
- ✅ Платформа С.С.С.Р. = «среда абонентского консультационного обслуживания»
- ✅ Copywriter при любом тексте — проверка через словарь запретов
- ✅ Юрист при ревью оферты — отдельный чек на образовательные формулировки

### 2.2 🟢 Никакой платной рекламы (бюджет = 0₽)

**Решение:** Инна не использует платную рекламу ни в каком виде. Не VK Таргет, не Яндекс.Директ, не Telegram Ads, не закупка у блогеров и пабликов. Даже в собственных постах не делаем рекламные вставки.

**Почему критично:**
- Стратегия эксперта-консультанта под ОКВЭД 70.22 строится на **системе и доверии**, не на покупке трафика.
- У Инны уже есть тёплый круг (≈7000 VK + МГ + ученики + сарафан) — ему не нужны рекламные пуши.
- Реклама = деньги вперёд за непроверенную гипотезу. Контент + AI-помощник + автоматизация = строительство актива.
- Без рекламы tripwire-лестница не нужна на старте — она для холодного трафика.

**Что НЕ делать:**
- ❌ Не предлагать «запустим рекламу на лендинг», «таргет на лид-магнит»
- ❌ Не закладывать бюджет на рекламу в финансовых расчётах (строка = 0 ₽)
- ❌ Не делать «обходные пути через платный таргет»
- ❌ Не делать рекламные вставки в собственных постах
- ❌ Не строить полную tripwire-лестницу (990 → 3990 → 9990) на старте

**Что делать:**
- ✅ Органический контент VK/TG/своих каналов
- ✅ Сарафанное радио и рекомендации
- ✅ Автоматизация прогрева (email-цепочка, AI-помощник, n8n)
- ✅ Личные продажи через тёплый круг
- ✅ Лид-магниты как полезный материал за подписку
- ✅ SEO и поисковая органика
- ✅ Кейсы, отзывы, экспертный контент

### 2.3 🟢 Vultr London VPN изолирован от продуктовой инфры

**Решение:** VPS Vultr London — это **только** наш личный VPN. Он **не участвует** в продуктовой экосистеме Инны (HUB, поддомены, AI, CRM, Платформа). Никакие данные продуктовой инфры через него не маршрутизируются.

**Почему критично:**
- Vultr London — **заграничный сервер**. По ФЗ-152 персональные данные граждан РФ должны храниться и первично обрабатываться на серверах в РФ.
- Любая маршрутизация ПД через лондонский VPS (даже как туннель) = риск нарушения ФЗ-152.
- VPN — это инструмент личного доступа Инны. Не инфраструктурный компонент бизнеса.

**Что НЕ делать:**
- ❌ Не использовать London VPN как точку доступа к `crm.*`, `bz.*`, `ain8n.*`, `platforma.*`
- ❌ Не разворачивать на нём n8n, базу данных, AI-бэкенд, CRM
- ❌ Не направлять туда логи и бэкапы серверов с ПД
- ❌ Не давать его IP в whitelist как «доверенный источник»
- ❌ Не упоминать его в публичных архитектурных схемах продукта

**Что делать:**
- ✅ Использовать London VPN строго по назначению — личный VPN Инны
- ✅ Защита внутренних поддоменов — через **BasicAuth + IP-whitelist российских IP** (дом. IP, VPS Cheerful Marik/Model Zarek)
- ✅ Если подрядчик динамический IP — выдавать BasicAuth без IP-whitelist
- ✅ В документации явно: London — изолирован

### 2.4 🟢 База знаний: один источник, два интерфейса

**Решение:** `my-architecture/knowledge/` = **источник** (markdown-файлы в репозитории, редактируются в VS Code). `bz.ideidlyabiznesa1913.ru` = **веб-интерфейс** (read-only зеркало, статический генератор).

**Почему критично:**
- Конфликт двух баз физически невозможен — источник один.
- Не нужна отдельная админка/CMS — правки только в VS Code через markdown.
- Подрядчики и команда могут открыть с телефона без репозитория, не нужен доступ к git.
- Поиск, навигация, оглавление — генератор делает автоматически.

**Как технически:** статический генератор сайта из markdown (MkDocs/Docusaurus). При коммите в `knowledge/` → CI/cron-скрипт пересобирает сайт → выкладывает на `bz.*`.

**Что НЕ делать:**
- ❌ Не редактировать веб-морду `bz.*` напрямую — там нет редактирования
- ❌ Не заводить вторую базу знаний (Notion/Confluence) параллельно
- ❌ Не дублировать контент между `knowledge/` и `bz.*` вручную
- ❌ Не открывать `bz.*` для редактирования никому

**Что делать:**
- ✅ Все правки → только в `knowledge/` через VS Code
- ✅ Генератор и публикация — через cron/GitHub Actions/коммит-хук
- ✅ Доступ к `bz.*` — публичный (если нужно подрядчикам) или BasicAuth (только для команды)
- ✅ Если нужно редактировать с телефона — PR-flow через GitHub, не редактирование на `bz.*`

### 2.5 🟢 AI-помощник на корне HUB = главная фишка

**Решение:** AI-помощник Инны живёт на главной странице HUB (`ideidlyabiznesa1913.ru/`) и доступен бесплатно случайному посетителю с улицы.

- ❌ Не делать AI платным или с лимитом вопросов на старте
- ❌ Не убирать AI с лендингов по болям

**Что делать:**
- ✅ AI-помощник на главной HUB + на каждом из 5 лендингов по болям + на странице каталога

### 2.6 🟢 Один поддомен = одна функция (дисциплина)

**Решение:** каждый поддомен экосистемы выполняет ОДНУ чёткую функцию. Не допускаем параллельных доменов с пересекающимся контентом.

**Почему критично:**
- Research показал грабли Гандапаса — 4 параллельных домена (`radislavgandapas.com` + `gandopas.ru` + `radislav-gandapas.com` + `club.*`) с пересекающимся контентом. Пользователь не понимает куда идти, конверсия рухнула.
- Жёсткая карта предотвращает путаницу и потерю конверсии.

**Что НЕ делать:**
- ❌ Не создавать второй поддомен под ту же функцию
- ❌ Не дублировать каталог продуктов на нескольких поддоменах
- ❌ Не делать «новую версию» поддомена параллельно со старой

**Что делать:**
- ✅ Жёсткая карта поддоменов:
  - `inna.*` — визитка Инны
  - `mini.*` — демонстрационные среды
  - `demo.*` — портал для клиентов
  - `platforma.*` — Платформа С.С.С.Р. (кабинет + обслуживание)
  - `ai.*` — бэкенд AI-помощника
  - `crm.*` — моя база клиентов (админка)
  - `bz.*` — База знаний (read-only веб-интерфейс)
  - `ain8n.*` — n8n админка (автоматизация, BasicAuth)
- ✅ При появлении нового продукта — определять в какой существующий поддомен он встаёт
- ✅ Новый поддомен заводим только если функция действительно отдельная

### 2.7 🟢 Свой VPS, не GetCourse/Bizon365/SaaS-платформы

**Решение:** Платформа, AI, CRM, n8n живут на собственном VPS (Cheerful Marik сейчас, Model Zarek после миграции). НЕ используем готовые SaaS-платформы для инфобизнеса (GetCourse, Bizon365, BotHelp, SaleBot и т.п.).

**Почему критично:**
- Research подтвердил: топ-игроки рынка (Высоцкий `my.bbooster.online`, Лайк Центр собственная платформа, Христосенко «Универиум») идут своим путём.
- GetCourse/Bizon365 = противоречие позиционированию «Архитектор автосистем». Использовать чужой SaaS = выдать себя за рядовую инфошколу.
- ОКВЭД 70.22 требует полный контроль над AI и интеграциями, чужие SaaS этого не дают.
- Если переезжать потом — потеря данных и клиентов. Лучше с нуля на своём.

**Что НЕ делать:**
- ❌ Не уходить на GetCourse/Bizon365/BotHelp ради «быстрого старта»
- ❌ Не размещать Платформу на чужом домене (например, `inna.getcourse.ru`)
- ❌ Не делать ЛК подписчиков на сторонней SaaS
- ❌ Не интегрировать Мастерприкладов с SaaS-платформой через API без плана миграции

**Что делать:**
- ✅ Своя разработка через vibe coding на нашем VPS
- ✅ Tilda допустима **только** для одиночных лендингов в режиме MVP (1-2 дня)
- ✅ Финальная версия HUB, поддоменов, платформы = на нашем VPS/Beget shared

- ✅ При росте нагрузки — масштабируем на своём VPS, не переезжаем на SaaS

### 2.8 🟢 Поэтапный запуск экосистемы (не одновременный)

**Решение:** не строим все 8 поддоменов одновременно. Запуск идёт по триггерам, в порядке от блокирующих к опциональным.

**Почему критично:**
- Поддержка 8 сервисов одновременно лопнет на 3-м месяце на организационном уровне.
- На Cheerful Marik уже сейчас есть проблемы (n8n+Baserow были остановлены 5 дней, в `/root` хаос бэкапов). Нужно сначала навести порядок на одном сервисе, потом расширять.
- Каждый запущенный поддомен = мониторинг, апдейты, бэкапы, поддержка. Параллельный запуск = потеря контроля.

**Порядок запуска (блокирующие → опциональные):**

| Этап | Что | Триггер запуска | Примечание |
|---|---|---|---|
| **1** | HUB корень (ideidlyabiznesa1913.ru) + 1 лендинг боли | Сейчас — главный блокер бизнеса | Главная витрина + первая точка контакта |
| **2** | `inna.*` визитка (статика) | Под VK-упаковку и QR | Брендирование личного бренда |
| **3** | `crm.*` минимум (Моя база клиентов) | Перед запуском AI-помощника | Чтобы лиды не терялись при контакте |
| **4** | `ai.*` бэкенд + AI на HUB | После Этапов 1-3 | Главная фишка воронки |
| **5** | `mini.*` с 1 демо | После запуска AI-помощника | Витрина продукта, демонстрация |
| **6** | `demo.*` | По факту появления клиента в работе | Портал для конкретного платного клиента |
| **7** | `platforma.*` (Платформа С.С.С.Р.) | После 3-5 первых платных подписчиков | Не строим пустой кабинет, пока нет клиентов |
| **8** | `bz.*`, `ain8n.*` | Внутренние, можно отложить | Поднимаются под BasicAuth после стабилизации |

**Что НЕ делать:**
- ❌ Не запускать все поддомены одновременно
- ❌ Не делать параллельную разработку Платформы и HUB без MVP на HUB
- ❌ Не строить сложный `crm.*` прежде чем есть первый клиент
- ❌ Не ждать идеального `inna.*` прежде чем лайкать HUB

**Что делать:**
- ✅ Минимальный рабочий маршрут = Этап 1-3 (HUB + визитка + CRM) должны работать
- ✅ После стабилизации Этапа 1-3 — запуск AI (Этап 4)
- ✅ Параллельная разработка: пока живёт Этап 1, готовим Этап 2-3 в брeтах
- ✅ Каждый новый этап — явный триггер (время / кол-во клиентов / готовность обслуживания)

## 10. Почему такое распределение — три причины

### Причина 1 — техника

| Параметр | Cheerful Marik | Model Zarek |

PostgreSQL для CRM + Платформа + AI-движок на Cheerful Marik = OOM-killer убьёт боевой n8n клиента Ирины.

### Причина 2 — изоляция рисков

Cheerful Marik = **боевой клиентский сервер**. Если при разработке наших поддоменов что-то сломаем — упадёт бот клиента Ирины.

На Model Zarek можно ломать сколько угодно — там только наше.

### Причина 3 — нельзя повышать тариф Beget

По решению `feedback_no_beget_tariff_upgrade.md` — тариф Beget shared не повышаем. Поэтому статичные поддомены идут не на shared, а на VPS Model Zarek.

## 14. Пользователь `deploy` на Model Zarek

**Решение Этапа 1:** для деплоя статики на сервере создан пользователь `deploy` (login через ssh-ключ Инны), **без sudo**. Владелец папок `/var/www/<sub>/` — `www-data` (стандарт). `deploy` имеет права записи через группу `www-data`.

Root-доступ остаётся только для админских операций (Director через ssh-алиас `model-zarek`).

**Почему так:**
- Стандарт безопасности: ssh-ключ для деплоя статики не должен иметь админских прав.
- Если ключ утечёт — атакующий не может удалить сервер, перезаписать systemd-юниты, поломать nginx. Только подменить файлы статики (восстанавливается из репо за 30 секунд).
- Изоляция: deploy-скрипты статики `inna.*`, `mini.*`, `bz.*` работают под `deploy`, не имеют прав трогать `crm.*`, `ai.*`, `platforma.*` (там нужен root).

## 15. Структура файлов в репо — `deploy/zarek-nginx/`

Все nginx-server-блоки и базовые конфиги хранятся в репо. На сервере это **копии**, источник правды в репо, разворачиваются по deploy-скрипту.

```
deploy/
├── zarek-nginx/
│   ├── nginx.conf                  ← базовый конфиг
│   ├── inna.conf                   ← server-блок для inna.*       (Этап 2 ✅)
│   ├── mini.conf                   ← server-блок для mini.*       (Этап 5)
│   ├── ai.conf                     ← server-блок для ai.*         (Этап 4)
│   ├── crm.conf                    ← server-блок для crm.*        (Этап 3 🔜)
│   ├── bz.conf                     ← server-блок для bz.*         (Этап 8)
│   ├── platforma.conf              ← server-блок для platforma.*  (Этап 7)
│   └── demo.conf                   ← шаблон для demo.* (НЕ активируется до Этапа 6)
├── zarek-setup.sh                  ← скрипт первичной настройки сервера
└── zarek-deploy.sh                 ← скрипт раскатки конфигов с мака на сервер
```

**Почему так:**
- Единый стиль с deploy-скриптами в корне репо
