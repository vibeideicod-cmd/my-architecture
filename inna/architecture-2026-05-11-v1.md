# ЧАСТЬ 2. ПРИНЦИПЫ MUST-KEEP

> Это не «нравится — соблюдаем, не нравится — нет». Это **жёсткие правила**, которые применяются командой агентов при принятии любых архитектурных решений. При конфликте принципов побеждает тот, что выше в списке внутри каждого подраздела.

## 2.1. Юридические принципы

### 2.1.1. ОКВЭД 70.22 — железное правило
Все публичные тексты, оферта, чеки, лендинги, AI-помощник, лид-магниты — только через формулу **«Аудит → Рекомендации → Сопровождение внедрения»**. Запрещены слова: создаю, разрабатываю, делаю сайты/боты, программирую, вайбкодер.

### 2.1.2. ФЗ-152 — персональные данные на российских серверах
Все ПД граждан РФ хранятся и первично обрабатываются на серверах в РФ. На каждом сайте — Политика конфиденциальности и Согласие на обработку ПД.

### 2.1.3. Нет лицензии на обучение
У ИП Инны нет образовательной лицензии. Запрещены слова: курс, обучение, школа, тренинг, урок, преподаватель, ученик, академия, мини-курс, диплом. Разрешены: консультация, аудит, разбор, сопровождение, методические материалы, экспертная сессия, чек-лист, шаблон, видео-разбор.

### 2.1.4. mini.* — обязательная плашка
На каждой странице `mini.*/<имя>/` — плашка «Демонстрационная среда. Образец решения, не самостоятельный продукт.» + `noindex`.

### 2.1.5. Маркировка AI
AI-помощник первым сообщением представляется: «Привет! Я — AI-помощник Инны.» Не выдаёт себя за Инну. AI = инструмент квалификации запросов, не самостоятельный товар.

## 2.2. Темп и приоритизация

### 2.2.1. Расширяем, не урезаем
Все идеи сохраняются в архитектуре, переносятся в очередь, не отрезаются (tripwire, платный таргет, MAX/YouTube/Дзен, Подбренд Нейро Бабки, MINI-6 SaaS, чат-сообщество и т.д.). Разворачиваются по триггерам.

### 2.2.2. Темп — дни и часы, не недели
Через vibe coding и параллельную работу команды агентов. Узкое место — данные клиента, не реализация.

### 2.2.3. Поэтапный запуск, не одновременный
6 шагов первой волны → реактивация каналов → расширение. Не строим всё одновременно. Дисциплина по этап-карте.

### 2.2.4. Никакой платной рекламы
Бюджет на трафик = 0 ₽. Нет VK Таргет / Яндекс.Директ / TG Ads / закупка у блогеров. Даже посты без CTA «купи/акция/скидка». Только органика + сарафан + автоматизация прогрева. Tripwire-лестница на старте НЕ строится (без рекламы холодного потока нет).

## 2.3. Архитектурные принципы

### 2.3.1. AI-помощник Инны на корне HUB — главная фишка
Виджетом на всех страницах + hero-блок на главной. Юр.чисто. Маршрутизация по 5 болям × 8 нишам × этапам A/B/C. По данным Research v2 (09.05): из 8 топ РФ-экспертов **никто** не выставляет AI-чат на главную — белое пятно рынка на 6-9 месяцев.

### 2.3.2. Один поддомен = одна функция
Не дублируем (грабли Гандапаса: 4 параллельных домена с пересекающимся контентом → конверсия падает).

### 2.3.3. Свой VPS, не GetCourse / Bizon365
Соответствие позиционированию «Архитектор автосистем» и ОКВЭД 70.22. Чужой SaaS = выдать себя за рядовую инфошколу. Tilda допустима только для одиночных лендингов в режиме MVP.

### 2.3.4. bz.* — один источник, два интерфейса
База знаний `bz.ideidlyabiznesa1913.ru` = read-only веб-рендер `knowledge/` через статический генератор. Правки только в VS Code через markdown. Дублирование контента запрещено.

## 2.4. Процессные принципы

### 2.4.1. Принцип живой витрины
Сначала строю у себя — потом предлагаю клиентам. Всё что работает в моём бизнесе (HUB, Платформа С.С.С.Р., AI-помощник) — живой пример того, что я делаю для клиентов. Не обещаю — показываю.

Параллельно тиражируется Ирине под её нишу (бренд-дизайн). Когда Инна готова — выходим на платных клиентов из тёплого круга, не ждём Ирину.

### 2.4.2. Технологии показываем цветом, не словами
Главное правило бренда. В публичных текстах — только «аудит, сопровождение, экспертиза». В реальности — делаем через vibe coding. Технологичность даём визуалом (палитра, шрифты, акценты).

### 2.4.3. Начинаем с этапа клиента (C/B/A)
Каждое предложение AI-помощника зависит от этапа клиента. Если у клиента есть клиенты (сарафан) — усилить (не терять). Если есть соцсети — усилить. Нет соцсетей — начать с этого. Нет автоматизации — мини-продукт под 1 задачу.

### 2.4.4. Фокусные документы обновляем синхронно
При появлении нового архитектурного решения — обновляются ВСЕ релевантные документы за одну сессию, не «потом синхронизируем». Иначе через 3 дня источники правды расходятся.

### 2.4.5. Сначала готовим куда придёт клиент — потом идём к клиенту
**Обратная очерёдность**: CRM «Моя база клиентов» → HUB+лид-точка → AI → реактивация VK/TG. Не наоборот. Если реанимировать VK без места приёма — заведём людей в пустоту, сожжём тёплый контакт. См. подробности в Части 4.

### 1.3. Целевая аудитория

> *«Для тех, кто живёт в бизнесе 24/7 и тянет его на себе»*

**Ниша — вторична. Первичен фильтр по состоянию клиента.** Один человек тянет всё на себе, соцсети заброшены, автоматизации нет, времени разбираться — нет.

**Приоритетные ниши (8):**
1. Инфобизнес и онлайн-образование
2. Финансовые консультанты и брокеры
3. Бьюти (салоны, барбершопы, косметология, СПА)
4. Коучи, психологи, наставники
5. Фитнес-клубы и тренеры
6. Торговля — офлайн и онлайн
7. Риэлторы и агентства недвижимости
8. Санатории и отели

**Фильтр по клиенту (важнее ниши):**
- Платёжеспособный — не ищет «самое дешёвое»
- Готов вкладываться в развитие — не ждёт «волшебную таблетку»
- Возраст обычно 30–60 лет
- Ценит включённость и долгосрочные отношения

**Детальные портреты** (9 портретов A1-C3 по готовности к AI; разрезы по 8 нишам): `inna/audience.md`.

### 1.4. Методология С.С.С.Р.

Не просто аббревиатура — это рабочий цикл с каждым клиентом. Каждая буква отвечает на боль.

| Буква | Значение | Боль клиента | Что решает |

### 3.2. Принципы работы команды

**Директор — маршрутизатор верхнего уровня.** Все задачи входят через Директора. Он управляет только **11 руководителями департаментов** — не каждым специалистом и не каждым скиллом напрямую.

**Принцип иерархии:**

```
Директор
   ↓ управляет только 11 руководителями департаментов
Руководитель департамента
   ↓ управляет своими специалистами и скиллами
Специалист (агент) со своими скиллами
   ↓ выполняет конкретную задачу
Результат
```

**Скилл = инструмент агента.** Скилл живёт отдельным файлом, но **принадлежит конкретному агенту**. Директор не держит каталог скиллов в голове — он знает только руководителей департаментов.

**Два режима работы:**

| Режим | Когда | Как |
|---|---|---|
| **Линейный** | Стандартная задача | Директор → руководитель → специалист + скилл → результат |
| **Параллельный** | Сложная задача, несколько доменов | Директор → несколько руководителей → собирает результат |

## 5.5. Открытая точка: что заменит «бесплатную диагностику 30-40 мин»

> ⚠️ **Решение требуется до запуска AI-помощника (Шаг 3 первой волны).**

В Идее v3 (09.05) был встроен мост «лид → бесплатная диагностика 30-40 мин с Инной лично» по образцу Ледовских (`aibasis.ru/consultai`). Инна **отвергла этот ход 09.05 вечером** — у неё нет ресурса на массовые личные встречи.

Без этого моста разрыв: AI-помощник довёл клиента до сегмента B → лендинг боли → дальше что? Большой прыжок от 0 ₽ (диагностика AI) к продукту 30-80к (mid).

**Опции замены (требуют решения Инны):**

| Опция | Описание | Плюсы | Минусы |
|---|---|---|---|
| A — Запись в очередь | Клиент оставляет контакт → Инна связывается когда есть время | Не съедает время Инны | Конверсия упадёт (долго ждать) |
| B — AI-отчёт по диалогу | После 4-6 вопросов AI даёт развёрнутый отчёт «вот ваш диагноз и что делать» | Полная автоматизация | Меньше доверия чем личная встреча |
| C — Платный экспресс-аудит | 3-7к за 30-минутный разбор с Инной | Фильтр серьёзных + доход | Снижение конверсии на этом шаге |
| D — Регулярные групповые разборы | Раз в неделю Zoom-разбор на ~10 человек одновременно | Окупает время Инны | Требует регулярности |
| E — Бесплатный текстовый разбор по форме | Клиент заполняет анкету → Инна отвечает текстом 1 раз | Контролируемая нагрузка | Длинно текстом, не диалог |

Выбор моста — за Инной. До решения по этому пункту — на лендингах болей используется временный CTA «Записаться на разбор» с формой ожидания.

# ЧАСТЬ 7. КАК ИСПОЛЬЗОВАТЬ ЭТОТ ДОКУМЕНТ

Этот документ — **системный контекст** для AI-агентов. Подгружается в начале каждой сессии с командой. Все архитектурные решения сверяются с семью слоями + принципами must-keep.
