## Цель Client Zero

Доказать на Инне, что продукт работает:

> Берём бизнес с уже существующими активами → отделяем актуальное от тестового → собираем архитектуру и компоненты → запускаем первый рабочий маршрут → получаем модель, которую можно повторять на Ирине и клиентах.

## Главный вопрос перед реализацией

Не “что построить первым?”, а:

> Какие компоненты обязательны, чтобы клиентский маршрут стал рабочим?

Для Инны сейчас минимальный рабочий маршрут:

```text
HUB
→ Карта точек роста
→ контакт
→ статус лида
→ ручной follow-up
→ предложение платного AI-аудита
```

## P0 — без этого продукт не считается рабочим

| P0 | Почему |
|---|---|
| Назначить product truth | Чтобы команда знала, что главный продукт = не CRM, не HUB, а бизнес-архитектура под ключ |
| Компонентная карта | Чтобы не строить куски вслепую |
| Roadmap 7/30/90 | Чтобы после архитектуры было понятно, что внедрять |
| Первый маршрут | Чтобы продукт доводил до действия, а не только до документа |
| CRM/status или честный fallback | Чтобы лид не терялся |
| Юр.минимум ПД | Чтобы контакт можно было собирать публично |

## P1 — усиливает первый запуск

| P1 | Почему |
|---|---|
| Переписать HUB под актуальное позиционирование | Сейчас HUB частично старый |
| Подключить CRM API | Уберёт ручной Telegram follow-up |
| Создать результат “Карта точек роста” в PDF/тексте | Повысит ценность входа |
| Сформировать оффер платного AI-аудита | Нужен мост в оплату |

## P2 — после первого маршрута

| P2 | Почему |
|---|---|
| AI-база | Чтобы AI отвечал не методологически, а клиентским языком |
| AI-помощник на HUB | Главная фишка, но без базы и CRM преждевременно |
| VK/TG реактивация | Трафик ведём только когда вход готов |
| Кейсы и блог | Усиление доверия |

## P3 — позже

| P3 | Почему |
|---|---|
| Платформа С.С.С.Р. | Нужна после первых платных клиентов/подписчиков |
| Личный кабинет | Не нужен для проверки первого маршрута |
| Автоматические PDF и письма | После ручной валидации |
| Полная CRM | После первых лидов и понимания реального процесса |

## Порядок работы теперь

### Шаг 1. Зафиксировать продукт

Готово этим контуром:

- `main-product/idea.md`;
- `main-product/components-map.md`;
- `main-product/client-zero-plan.md`.

### Шаг 2. Сформировать Roadmap 7/30/90

Следующий документ:

`main-product/roadmap-7-30-90.md`

Он должен ответить:

- что запускаем за 7 дней;
- что доводим за 30 дней;
- что превращаем в продаваемый продукт за 90 дней.

### Шаг 3. Проверить первый маршрут по P0

Проверяем:

- HUB ведёт на Карту точек роста;
- Карта собирает контакт;
- есть статус/follow-up;
- есть юридический текст;
- есть следующий оффер.

### Шаг 4. Только потом строить компоненты

Если P0 говорит “ломается CRM” — строим CRM.

Если P0 говорит “ломается юр.согласие” — сначала юр.минимум.

Если P0 говорит “непонятно что продаём после карты” — сначала оффер платного AI-аудита.

## Текущий вывод

CRM действительно нужна, но решение “начать с CRM” было преждевременным как продуктовая логика.

Правильнее:

1. Зафиксировать главный продукт.
2. Разложить его компоненты.
3. Составить roadmap.
4. Проверить P0 первого маршрута.
5. Потом строить CRM/AI/HUB в правильном порядке.
