## 1. Внутренние паттерны (что переиспользуем из репо)

### MG-конструктор (`clients/mastergroup/mvp/`) — **главный донор механики**

Это единственное место в репо с паттерном «форма → генерация → публичный результат». Берём его как базу для Опции 3 («собери визитку прямо сейчас»).

**Архитектура:**
- `index.html` — лендинг, `apply.html` — анкета, `build.html` — конструктор, `vitrina.html` — публичная витрина, `cabinet.html` — кабинет.
- Чистый HTML + inline CSS (Montserrat + Playfair) + inline JS + Supabase через CDN (`@supabase/supabase-js@2`).
- TMA init вынесен в `/js/tma-init.js` (`tg.ready()` + `tg.expand()` + haptic).
- Фото → Supabase Storage `mg-photos/masters/<slug>/photo-<ts>.jpg`.
- Данные → `sb.rpc('mg_save_page', {patch})` с throttle 600ms.
- Публичный результат — `demo.ideidlyabiznesa1913.ru/mg/m/<slug>/` + копировать/поделиться/QR.

**Что берём 1-в-1:**
- CSS-переменные и палитра (адаптируем)
- Структура «блок → `setAnswer(block, code, value)` → `scheduleSave(patch)`»
- Supabase Storage для фото
- Гейт на контакт из `apply.html` (поля `full_name` + `contact` с HTML5 validation)

**Что НЕ берём:**
- Sleep-таймеры и сохранение «черновиков» — у нас одноразовый flow
- Админка — Инне она здесь не нужна (все данные посетителей падают в одну таблицу)

### Beauty TMA (`clients/beauty/tg-app/`) — донор TMA-init

```js
// Точный код инициализации (app.js:13-17)
function initTelegram() {
  const tg = window.Telegram?.WebApp;
  if (!tg) return;
  tg.ready();
  tg.expand();
}
```

Deep-link с префиллом через `tg.initDataUnsafe.start_param`. Для нашей визитки: `t.me/<bot>/app?startapp=from_<source>` — источник трафика сразу виден в лиде.

### Alisa визитка (`clients/alisa/index.html`) — референс дизайна

Один файл ~340 строк, герой + tagline + соцсети (TG/Я.Карты/MAX) + CTA + footer. `clamp()` для адаптива, backdrop с gradient-blobs. Берём за основу **верстки первого экрана** визитки Инны.

### Критический пробел в репо

1. **Генерация HTML/PDF для скачивания** — нет нигде. Пишем с нуля. Варианты: `html2pdf.js` (CDN), `html2canvas` + `jspdf`, либо проще — отдать чистый HTML-файл через `Blob + URL.createObjectURL + <a download>`.
2. **Прямая отправка файла от TG-бота** — нет. Инфраструктура для этого: отдельный бот на node/python, который дёргает `sendDocument`. Во MVP проще всего — генерим прямую ссылку на файл в Supabase Storage и шлём ссылку в TG.

### Работающие паттерны (дистилляция)

1. **Одно поле в форме** (email ИЛИ TG, не оба). Каждое доп. поле = −10% конверсии.
2. **Preview до гейта** — показать 20–30% результата бесплатно, остаток за контакт.
3. **CTA от первого лица посетителя**: «Забери», «Получи», «Get» — НЕ «Отправить», «Submit».
4. **Social proof под кнопкой** с цифрой: «уже Х архитекторов».
5. **Мгновенная выдача** на экране после контакта, не «пришлём через 5 минут».
6. **Quiz-funnel** — топ конверсионный паттерн (35–50%). Квиз создаёт инвестицию времени → после него оставить контакт психологически дешевле.
7. **Tripwire на thank-you**: после контакта сразу предлагать платный продукт 300–1000 ₽. Удваивает LTV.

## 3. VK Mini App — quick-start для нас

### Инициализация

```html
<!-- CDN-вариант (подойдёт для нашего стиля "чистый HTML без сборки") -->
<script src="https://unpkg.com/@vkontakte/vk-bridge/dist/browser.min.js"></script>
```

```js
vkBridge.send('VKWebAppInit')
  .then(() => console.log('VK Bridge ready'))
  .catch(console.error);
```

### Получение данных пользователя (до формы!)

```js
// User info — имя, фамилия, фото, VK ID
vkBridge.send('VKWebAppGetUserInfo')
  .then(user => {
    // user.id, user.first_name, user.last_name, user.photo_100
  });

// Email — нативная модалка VK, НЕ наша форма
vkBridge.send('VKWebAppGetEmail')
  .then(({email}) => { /* сохраняем в лид */ });

// Телефон — тоже нативная модалка
vkBridge.send('VKWebAppGetPhoneNumber')
  .then(({phone_number}) => { /* сохраняем в лид */ });
```

### 🎯 Критический инсайт для лид-машины

**В VK-версии визитки можно НЕ рисовать форму контакта вообще.** Вместо формы — кнопка «Получить результат», которая вызывает нативную модалку VK-запроса email/телефона. Посетитель жмёт «Разрешить» — мы получаем контакт сразу, без набора руками. Это unlock 60%+ конверсии.

**В Telegram-версии** — аналог: кнопка «Поделиться контактом» через TG Web App Keyboard (или просто передача `initData.user.id` + `username`).

**В браузерной версии** — классическая форма с email ИЛИ TG username.

### Manifest и деплой

- **Манифест-файла НЕТ.** Конфиг в веб-кабинете `vk.com/editapp?act=create`: тип Mini App, Base URL (HTTPS), Mobile URL, платформы, scopes (`email`, `phone_number`).
- **Деплой** — hosted URL на нашем HTTPS (как Telegram) ИЛИ через VK Hosting CLI (`@vkontakte/vk-miniapps-deploy`). Если выбираем Beget — просто указываем там URL `https://<домен>/inna/vk/`, без переноса файлов.
- **CORS:** разрешить `https://*.vk.com`. **X-Frame-Options:** не ставить DENY. **HTTPS обязателен**, валидный серт.
- **localStorage** обнуляется в iOS-клиенте → персистентность через `VKWebAppStorageSet/Get`.

### Ограничения

- iframe-песочница: нет `window.top`, URL-бара нет.
- Внешние редиректы блокируются → открытие ссылок через `window.open` с подтверждением или `VKWebAppOpenApp`.
- Рекомендация VK: первый экран < 2 MB. Для нашей визитки ок.

## 4. Бенчмарки конверсии (ориентиры для Инны)

> Источники: Unbounce Conversion Benchmark, HubSpot State of Marketing, ConvertKit/Kit Creator Report, OptinMonster — из базы знаний агента. Верифицировать актуальные версии на `unbounce.com/conversion-benchmark-report`, `hubspot.com/state-of-marketing`.

| Метрика | Плохо | Средне | Хорошо | Топ 10% |
|---|---|---|---|---|
| Visitor → email контакт | < 10% | 15–25% | 25–40% | **> 45%** |
| Visitor → TG username | < 15% | 20–30% | 30–45% | **> 50%** |
| Quiz/интерактив → контакт | < 20% | 25–35% | 35–50% | **> 55%** |

**Калибровка цели Инны:** заявленные в idea.md 25%+ = «хорошо», 40%+ = «вау». 100% достижимо только на приглашённом трафике (рекомендации 1-в-1). На открытом — нереалистично, но можно держать как north star.

## 5. Выводы для brief.md (что закладываем в архитектуру)

### Механика

1. **MVP = одна опция: «Собери свою визитку прямо сейчас»** (Опция 3 из idea.md). Самая зрелищная, даёт моментальный результат. Опции 1 и 2 — во v2.
2. **Preview до гейта:** посетитель заполняет форму 5–7 полей, превью визитки рендерится в реальном времени сбоку или ниже (как в MG build.html). Мотивация «хочу забрать это себе» — максимум.
3. **Гейт именно перед скачиванием**, а не перед началом формы. Увидел результат → ввёл контакт → получил файл + ссылку.
4. **Tripwire на финальном экране:** «Понравилось? За 500 ₽ получи версию с выбором цветов + 6 форматов» — ведёт на следующий продукт (тот самый каскад из предыдущего обсуждения).

### Форма контакта — разная по платформам

| Платформа | Как собираем контакт | Почему |
|---|---|---|
| **Браузер** | Одно поле: TG username ИЛИ email (переключалкой) | Универсально, RU-аудитория предпочитает TG |
| **Telegram MA** | `initData.user` → username и id автоматом + `sendContact` кнопка если надо телефон | Уже в ТГ, грех спрашивать username руками |
| **VK MA** | `VKWebAppGetEmail` + `VKWebAppGetPhoneNumber` нативными модалками | Самая высокая конверсия — нет формы вообще |

### Данные лидов — куда падают

Три пути (финализировать с Инной):

- **Supabase** (как в MG) — БД + RLS, простой insert, удобно масштабировать.
- **TG-бот Инны** — webhook принимает лиды, пишет в ЛС Инне или в скрытый канал.
- **Google Sheets** — через Apps Script или Zapier, простейший вариант.

**Рекомендация:** Supabase для хранения + TG-уведомление Инне в ЛС на каждый лид. Это быстрее реакции и сохраняет архив.

### Социальные доказательства

Блок «уже Х архитекторов получили свою визитку» — показывать счётчик из Supabase (count записей). Стартовое число нельзя подделать — начинаем с «пилотируем, вы 1-й». Через неделю число настоящее.

### Tripwire-продукт

Нужен сразу, пусть даже как форма «оставь email — отправлю, когда будет готово». Это дожимает лидов, которые могут отвалиться между MVP и следующим шагом.

## 6. Что осталось открытым (решаем в критике и brief)

- Какой именно формат «готовой визитки» на выходе v1: HTML-файл для скачивания / ссылка на публичную страницу вида `/inna/made/<slug>` / PDF?
- Как рендерим превью во время заполнения формы — шаблон с `{{placeholders}}` на JS или SSR? Для статичного Beget — только клиентский шаблон.
- Фото посетителя в MVP: без фото (только имя + монограмма) или с загрузкой? Upload усложняет, но повышает ценность результата.
- Нужно ли пароль/редактирование: если «позже захочу поправить» — нужен secret-токен как в MG. Если «одноразовый магнит» — не нужен.

Эти вопросы унесу в Critique (Фаза 2.5) и в brief.md (Фаза 3).
