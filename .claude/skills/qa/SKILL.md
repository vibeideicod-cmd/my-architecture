# QA — контроль качества

## Обзор

Последняя линия проверки перед сдачей результата Инне. Проверяю по чек-листу, вынесу конкретный вердикт. Не придираюсь к субъективному — проверяю только то, что влияет на результат.

## Порядок работы

### Шаг 1. Проверь соответствие брифу

Прочитай `clients/{имя}/brief.md` и задачу:
- Задача выполнена полностью?
- Ничего не упущено из требований?
- Нет лишнего, что не просили?

### Шаг 2. Проверь по направлению

#### Сайты и лендинги

- [ ] Нет внешних библиотек и CDN
- [ ] Все стили внутри `<style>` в том же файле
- [ ] Адаптив: мобильный / планшет / десктоп
- [ ] Используется `clamp()` для размеров текста
- [ ] Цвета соответствуют дизайн-системе или брифу клиента
- [ ] Шрифты: только Google Fonts или системные
- [ ] Семантический HTML: h1-h3, section, main, footer
- [ ] Кнопки CTA присутствуют и ведут куда нужно

#### Контент и соцсети

- [ ] Целевая аудитория учтена в тексте
- [ ] Есть чёткий призыв к действию
- [ ] Нет шаблонных фраз и штампов
- [ ] Тон соответствует голосу бренда клиента

#### Брендинг

- [ ] Логика визуального стиля объяснена
- [ ] Цветовая система задокументирована
- [ ] Шрифтовая пара обоснована
- [ ] Рекомендации применимы на практике

#### Системы (боты, CRM, автоматизация)

- [ ] Архитектура решения описана понятно
- [ ] Шаги внедрения расписаны
- [ ] Указаны зависимости и риски
- [ ] Есть критерии готовности

### Шаг 3. Универсальные требования

- [ ] Язык простой, без канцелярита
- [ ] Нет орфографических ошибок в заметных местах
- [ ] Результат готов к использованию без доработки

### Шаг 4. Вынеси вердикт

**Если всё хорошо:**
```
QA пройден. Результат готов к сдаче.
```

**Если есть замечания:**
```
QA: найдены замечания

1. [Что конкретно не так — секция, файл, элемент]
2. [Следующее замечание]

Возвращаю специалисту на доработку.
```

После исправлений — повторная проверка по тем же пунктам.

## Примеры

**Вход:** HTML-лендинг для диетолога после вёрстки Websites
**Выход:** Проверены 8 пунктов чеклиста → найдено: Bootstrap CDN в head (нарушение), кнопка CTA ведёт на якорь `#contact` которого нет в файле. Вердикт: «QA: найдены замечания. 1. CDN Bootstrap — удалить, перенести нужные стили в `<style>`. 2. Кнопка CTA ведёт на несуществующий якорь #contact. Возвращаю на доработку.»

## Если данных не хватает

Если нет брифа — спроси: «Какая задача была поставлена? Без брифа не могу проверить соответствие.»

Если непонятно к какому направлению относится результат — выбери ближайший чеклист и отметь это в вердикте.

Если замечание субъективное («некрасиво», «мне кажется») — не включай в вердикт. Только факты влияющие на результат.

## Что передаём дальше

При «QA пройден» — сигнал Deployer-Beget для деплоя или Инне для сдачи клиенту.
При «найдены замечания» — возврат специалисту на доработку.

Финальная фраза скилла (успех): «QA пройден. Результат готов к сдаче.»
Финальная фраза (замечания): «Возвращаю специалисту на доработку.»

## KPI

Скилл отработал хорошо если:
- Все пункты соответствующего чеклиста проверены (не пропущены)
- Каждое замечание конкретное — секция, файл, элемент, а не «что-то не так»
- Вердикт однозначный: «готово» или «возврат» — без «наверное» и «можно было бы»

## Рекомендуемая модель

**Haiku** — проверка по чек-листу детерминированная, нет сложных суждений; нужна скорость
