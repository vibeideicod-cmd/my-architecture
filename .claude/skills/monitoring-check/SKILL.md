# Monitoring Check — Проверка инфраструктуры

## Обзор

Запускается DevOps/Infra по запросу Директора или автоматически перед деплоем. Даёт быстрый срез: всё живо, или есть проблемы. Результат: короткий отчёт в чат или `agent-runtime/logs/<дата>-monitoring.md`.

## Порядок работы

### Шаг 1. Получи список объектов проверки

Уточни что проверяем:
- Конкретный клиентский проект (укажи имя)
- Всю инфраструктуру команды целиком
- Конкретный компонент (только домены / только VPN / только боты)

### Шаг 2. Чек-лист — Домены и SSL

Для каждого домена под управлением:
```
[ ] Домен активен (не истёк)
[ ] SSL-сертификат действителен (до истечения > 14 дней)
[ ] Сайт открывается по HTTPS (редирект с HTTP работает)
[ ] DNS-записи не изменились
```

Где смотреть: панель Beget → Домены, или команды через SSH.

### Шаг 3. Чек-лист — Хостинг и серверы

```
[ ] Хостинг активен (оплачен, не заблокирован)
[ ] Сервер отвечает (ping / HTTP-запрос)
[ ] Дисковое пространство не забито (< 80%)
[ ] Нет критических ошибок в логах (error 500, OOM)
[ ] Бэкапы работают (если настроены)
```

### Шаг 4. Чек-лист — Боты и автоматизация

Для каждого активного бота/интеграции:
```
[ ] Бот отвечает на тестовое сообщение
[ ] Вебхуки не упали (если используются)
[ ] API-ключи не истекли
[ ] Лимиты API не исчерпаны
[ ] CRM-интеграции работают (если есть)
```

### Шаг 5. Чек-лист — Деплои

```
[ ] Последний деплой завершился успешно
[ ] Нет застрявших процессов
[ ] Версия в проде совпадает с ожидаемой
```

### Шаг 6. Составь отчёт

```markdown
## Monitoring Check — [дата и время]
**Объект:** [что проверяли]

### Статус
🟢 Всё работает / 🟡 Есть предупреждения / 🔴 Есть сбои

### Результаты по блокам
**Домены/SSL:** [статус]
**Хостинг/Серверы:** [статус]
**Боты/Интеграции:** [статус]
**Деплои:** [статус]

### Проблемы (если есть)
[Описание + приоритет: критично / требует внимания / не срочно]

### Рекомендации
[Что нужно сделать и в какой срок]
```

Если всё зелёное: «Мониторинг пройден. Инфраструктура работает штатно.»
Если есть проблемы: эскалируй Директору с описанием и предложением по решению.

## KPI

Скилл отработал хорошо если:
- Все блоки проверены — нет пропущенных пунктов
- Проблемы идентифицированы с точным описанием (не «что-то не работает», а «SSL истекает через 3 дня на domain.ru»)
- Критические проблемы эскалированы немедленно
- Отчёт сохранён в `agent-runtime/logs/`

## Рекомендуемая модель

**Haiku** — чек-листы и проверки по шаблону, не требует глубокой интерпретации
