# Типы продуктов в /discovery

> Справочник для скилла /discovery. Используется на Шаге 2 — определение типа,
> и на Шаге 8 — передача в работу.
>
> Под каждый тип — свой минимум вопросов и обязательные блоки.

---

## Список типов

| Тип | Обязательные блоки | Минимум вопросов | Передаём агенту |
|---|---|---|---|
| Сайт-визитка | А + Б + Д | 8-12 | `Websites` |
| Лендинг с конверсией | А + Б + В + Д | 12-18 | `Websites` |
| Telegram-бот | А + Б + В + Г + Д | 18-25 | `Systems` |
| Mini App | А + Б + В + Г + Д + блок SaaS | 18-25 | `Systems` + `Websites` |
| AI-консультант | А + Б + В + Г + Д + блок База знаний и сценарии | 18-25 | `AI Builder` |
| Квиз / калькулятор | А + Б + В + Г + Д | 15-20 | `Websites` + `Systems` |
| Гибрид (несколько в одном) | Все блоки + уточнение приоритета | 20-30 | определить ведущий тип |
| Не уверен | Базовые А + Б, тип определяем после | определяется по ходу | определить после блоков |

## Дополнительные блоки

### Блок SaaS (для Mini App с подпиской)

- Один сервер на всех мастеров или у каждого свой?
- Как разделяются данные между мастерами (по ID бота, по группе)?
- Платная подписка или бесплатное использование?
- Сколько мастеров планируется на старте?

### Блок База знаний и сценарии (для AI-консультанта)

- Где данные о продуктах/услугах для базы знаний? (документы, сайт, человек)
- Какие сценарии диалога обязательны? (запись, цены, FAQ, эскалация на человека)
- Должен ли AI собирать контакты лида? Куда сохранять?
- Подключение к мессенджеру (Telegram / WhatsApp) или только сайт?
- **Какие темы AI запрещено обсуждать (юр./этические ограничения)?** Особенно критично в нишах: медицина (диагностика, рекомендации лечения), финансы (гарантии доходности, прогнозы), юриспруденция (правовые консультации), бьюти-медицина (гарантии результата процедур). Если ниша из этого списка — обязательно фиксировать список «нельзя» в ТЗ.

## Гибридные продукты и экосистемы

Различай два разных сценария:

### Гибрид — несколько разных продуктов
Например: «нужен бот для записи + лендинг про услуги + AI-консультант». Это **разные продукты с разной логикой**, делаются в разное время.

1. Спроси: «Что делаем первым на старте? Что — потом, отдельным проектом?»
2. Если оба критичны — определи **ведущий** продукт и фиксируй второй как **«Этап 2»**.
3. Не пытайся уместить 2 продукта в одно ТЗ — выдай 2 связанных ТЗ (ведущий сейчас, второй позже через повторный `/discovery`).

### Экосистема — один продукт на нескольких платформах
Например: «приложение для клиентов: и в браузере, и Mini App в VK, и Mini App в Telegram». Это **один и тот же функционал, разные платформы развёртывания**.

1. Спроси: «Это один продукт на разных платформах или разные продукты?»
2. Если экосистема — переходи в режим **«зонтик-ТЗ + платформенные дополнения»**:
   - Один общий `idea.md` — общие блоки А-В описывают весь продукт сразу (ЦА, проблема, MVP-функционал общий для всех платформ).
   - Под каждую платформу — отдельная подсекция «Платформенное дополнение» с её спецификой (адаптация UI, ограничения платформы, особенности handoff).
   - Не дублируем общие блоки между платформами.
3. Handoff: каждое «Платформенное дополнение» → своему агенту (или паре агентов) параллельно.

## Если клиент не уверен в типе

Опция «не уверен — обсудим после блоков». Задай Базовые А + Б, по ответам ориентируйся:

| Что говорит клиент | Вероятный тип |
|---|---|
| «Приложение для людей», «в телефоне» | Mini App или мобильное приложение |
| «Онлайн-визитка», «расскажи о себе» | Сайт-визитка |
| «Продаёт услугу/товар» | Лендинг с конверсией |
| «Отвечает клиентам в чате» | AI-консультант или Telegram-бот |
| «Считает что-то / помогает выбрать» | Квиз или калькулятор |
| «Хочу всё сразу: сайт + бот + AI» | Гибрид — нужно расщеплять |

После — подтверди тип с клиентом и иди по нужному набору блоков.

## Связь типа с handoff

После Шага 7 (готовое ТЗ) ВСЕ ТЗ сначала идут в `Sales` — для формирования двух раздельных КП (зоны Инны и зоны Ирины), затем в `Contract-Lawyer` для договоров. И только после подписания договоров — в исполнителя.

| Тип | Куда сохранить ТЗ | После КП и договоров — кому | Где работает исполнитель |
|---|---|---|---|
| Сайт-визитка | `clients/[клиент]/idea.md` | `Websites` | вёрстка index.html |
| Лендинг | `clients/[клиент]/idea.md` | `Websites` | вёрстка + конверсионные блоки |
| Telegram-бот | `clients/[клиент]/idea.md` | `Systems` | бот на VPS / Beget Cloud |
| Mini App | `clients/[клиент]/idea.md` | `Systems` (бэкенд) + `Websites` (фронт) | SaaS-архитектура + Telegram WebApp |
| AI-консультант | `clients/[клиент]/idea.md` | `AI Builder` (берёт `idea.md` напрямую, без повторного интервью) | системный промпт + база знаний + интеграция |
| Квиз / калькулятор | `clients/[клиент]/idea.md` | `Websites` (UI) + `Systems` (логика) | фронт + расчёты/сохранение |
| **Гибрид** (несколько в одном) | `clients/[клиент]/idea.md` | определяется ведущим продуктом — он идёт первым; остальные — Этап 2 (отдельные ТЗ через повторный `/discovery`) | по типу ведущего продукта |
| **Экосистема** (зонтик-ТЗ) | `clients/[клиент]/idea.md` | каждое «Платформенное дополнение» → своему агенту параллельно | по платформам одновременно |
| **Не уверен** | определить тип после блоков А+Б, далее по таблице выше | — | — |
| Наш инструмент (не клиентский) | `idea.md` в корне репозитория | по тому же правилу (но без `Sales`/`Contract-Lawyer` — это для клиентов) | то же |

**Имя файла `idea.md`** синхронизировано с входом `agents/product-builder.md` (его Фаза 1 ожидает именно `idea.md`). Дальше `product-builder` сам пишет `brief.md` (после Critique) и `plan.md`.

Финальная фраза скилла после передачи: «ТЗ сохранено в `[путь к idea.md]`. Передаю `Sales` для формирования двух КП (зоны Инны и Ирины). Открыт ли вопрос ресёрча?»
