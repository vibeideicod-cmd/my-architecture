# Superpowers — инженерный слой дисциплины

## Обзор

Этот скилл адаптирует внешний подход `obra/superpowers` под архитектуру Инны. Он не заменяет `/brief`, `/discovery`, Product Builder, Playbook и QA. Он включает дисциплину: сначала цель и критерии готовности, потом план, потом реализация маленькими шагами, потом проверка фактами.

Канонический документ: `knowledge/methodology/superpowers-integration.md`.

## Когда использовать

Используй скилл при любой инженерной задаче:

- новый сайт, лендинг, квиз, калькулятор, Mini App, бот, веб-приложение, AI-консультант;
- изменение поведения существующего продукта;
- баг, ошибка, сбой деплоя, странное поведение сервера;
- API, Supabase, Telegram Bot API, авторизация, роли, платежи, интеграции;
- клиентский продукт перед публикацией;
- изменения в 3+ файлах кода или рефакторинг.

## Быстрый выбор режима

### Мини-режим

Для маленькой технической задачи перед реализацией заполни:

```markdown
## Мини-спека
Цель:
Файлы:
Критерий готовности:
Проверка:
Риск:
```

После этого выполняй задачу.

### Полный режим

Для клиентского продукта, бэкенда, бота, Mini App, AI-консультанта или крупной правки:

1. Проверь, есть ли `idea.md`, `brief.md`, `BACKEND-PLAN.md`, `PLAN.md`.
2. Если нет — не начинай код. Вернись к `/discovery` или Product Builder.
3. Разбей реализацию на маленькие шаги с чекбоксами.
4. Для логики и багов сначала создай failing test или воспроизводимый сценарий.
5. Реализуй минимально достаточный код.
6. Проверь результат командой, тестом, smoke test или визуальной проверкой.
7. Перед сдачей передай в QA.
8. После QA: git diff, проверка секретов, commit/push, обновление лога.

## Для багов

Не чини до понимания корня.

1. Прочитай ошибку полностью.
2. Воспроизведи стабильно.
3. Проверь последние изменения.
4. Сформулируй одну гипотезу: «думаю, причина X, потому что Y».
5. Сделай минимальную проверку гипотезы.
6. Только потом исправляй.
7. После фикса добавь тест или сценарий, который ловит проблему.

Если три гипотезы подряд не сработали — остановись и пересмотри архитектуру, а не добавляй четвёртый случайный фикс.

## Для новых функций

Не начинай с кода.

1. Цель пользователя.
2. Что входит в MVP.
3. Что НЕ входит.
4. Критерий готовности.
5. Как проверяем.
6. Какие файлы меняем.

Если функция меняет пользовательское поведение — Product Builder должен зафиксировать это в `brief.md` или `PLAN.md`.

## Для сайтов и лендингов

Superpowers не отменяет правила `websites`:

- структура согласована до HTML;
- архетип выбран до кода;
- один HTML-файл, стили внутри `<style>`;
- нет внешних CSS/JS-фреймворков;
- после вёрстки QA.

Для чистой статичной вёрстки TDD не обязателен. Вместо этого обязателен smoke test: открыть/экспортировать/проверить адаптив, ссылки, CTA, отсутствие битых якорей.

## Для Systems и AI Builder

Для API, ботов, Supabase, виджетов и AI-консультантов:

- тест или воспроизводимый сценарий до фикса;
- `.env.example` без секретов;
- `.gitignore` защищает `.env`;
- edge cases: пустой ввод, неверный токен, чужой пользователь, недоступный API, rate limit;
- перед деплоем QA + DevOps.

## Финальная проверка

Перед фразой «готово» должны быть факты:

- что изменено;
- чем проверено;
- где результат лежит;
- что осталось открытым, если осталось;
- commit/push сделан или явно отложен.

Финальная фраза скилла: «Superpowers-контур пройден: спека, реализация, проверка и handoff закрыты.»

## Когда отключить, если мешает

Если мини-спека занимает больше времени, чем сама правка, или Инна вынуждена три раза согласовывать мелочь — переходи в режим vibe (без спеки) и зафиксируй это в `project-log.md` или в дневнике сессии: «Superpowers был лишним здесь, потому что X». Раз в неделю Director собирает такие пометки и корректирует триггеры.

Это правило не отменяет дисциплину на серьёзном продукте (бот, бэкенд, AI-консультант, клиентская публикация). Оно защищает от ритуала ради ритуала на простых правках.

## Тесты vs воспроизводимый сценарий

Для логики (боты, API, AI, Mini App, бэкенд):

- **Желательный максимум** — failing test до фикса, потом фикс, потом тест проходит. Это правильная инженерная практика, и через это правило она формируется как привычка.
- **Минимум, который должен быть всегда** — воспроизводимый сценарий текстом: что вводим, что нажимаем, что должно получиться. Не заменяет тесты, но позволяет повторно проверить вручную.

Для статической вёрстки (HTML/CSS) тесты не пишутся — только smoke test (открыть, проверить адаптив, ссылки, CTA, отсутствие битых якорей).

Если на проекте тесты ещё не настроены — это технический долг. Зафиксируй в `project-log.md` клиента и постепенно закрывай.

## QA — один вызов, не два

Когда в скилле сказано «передай в QA» — это вызов того же `agents/qa.md`, не отдельный «Superpowers QA». В QA уже есть Шаг 3.5 с расширенным чек-листом для инженерных задач. Director вызывает QA один раз после реализации — этот вызов закрывает и обычный QA, и Superpowers-проверку.

## Фиксация применения

После каждой инженерной задачи Director добавляет в `clients/<name>/project-log.md` (или в `memory/sessions/YYYY-MM-DD.md` для архитектурных задач):

```markdown
#superpowers-applied
Режим: mini / full / off
Задача:
Помог:
Мешал:
Проверка:
```

`off` тоже фиксируется — это материал для калибровки триггеров.
