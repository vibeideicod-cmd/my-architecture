## Запчасть 1.7 — Автономность: GREEN/YELLOW/RED операции

**Источник:** [CLAUDE.md](../model-zarek-app/CLAUDE.md) раздел «Автономность и разрешения», 28 строк.

**Что это:**
Три уровня риска:

## Запчасть 1.8 — Запуск процессов: ps aux + один токен = один процесс

- После теста — проверь завершение: `ps aux | grep node`
- Если есть .service — `systemctl restart`, не `node` напрямую

Запуск фонового процесса = RED-операция.

**Источник:** [CLAUDE.md](../model-zarek-app/CLAUDE.md) и [CLAUDE-SYSTEM.md](../model-zarek-app/CLAUDE-SYSTEM.md), раздел «Устойчивость при сложных задачах», ~30 строк.

**Что это:**
Главное правило: **никогда не пиши файл целиком за один раз**. Работай инкрементально.

- Минимальный скелет (20-50 строк) → git commit
- Функции по одной → commit после каждой
- Файл > 100 строк → правь точечно, НЕ перезаписывай
- Каждый commit = рабочее состояние
- Перед рискованной операцией: `git add -A && git commit -m "[agent] checkpoint: before <операция>"`

**Ветки:**
- Папки с памятью/заметками — коммить в текущую ветку
- Код проектов — для новой фичи: `git checkout -b agent/название-задачи`, после завершения push
- В main напрямую — только мелкие правки (1-3 файла, не рефакторинг)

**`_progress.md`:**
- При задаче на 3+ минуты — создай `_progress.md`
- Обновляй после каждого коммита
- При завершении задачи — удали
- Незакоммиченные изменения от прошлой сессии — НЕ удаляй, разберись

**Восстановление после «Продолжи»:**
- Сначала `_progress.md` если есть
- Затем `git log --oneline -5`, `git status`, `ls`
- Если файл существует — продолжи правками, НЕ перезаписывай
- Непонятно — спроси

**Что у нас сейчас:**
Похожие, но разрозненные правила:
- В CLAUDE.md есть «коммитим и пушим регулярно без отдельного запроса» — решает «инкрементально + коммит как страховка»
- Чекпоинт-теги перед большими изменениями уже использовались (`before-zhorik-decomposition-2026-05-07`) — аналог `[agent] checkpoint`
- `agent-runtime/` — рабочий стол (в .gitignore), но это про обмен между агентами, не про прогресс
- TodoWrite — трекинг шагов в текущей сессии
- Дневник `memory/sessions/YYYY-MM-DD.md` — фиксирует прогресс в конце дня

**Чего нет:**
- **`_progress.md`** — нет файла-страховки на случай прерывания **текущей** задачи (между концом дня и записью в дневник могут быть часы потерянного контекста)
- **Правила «крупные правки кода клиента — в feature-ветке»** — у нас всё в main; для рефакторинга боевых проектов (Beauty TMA, Южнобережный, HUB) это риск сломать продакшн
- **Правило «чекпоинт-тег перед большими операциями»** работает де-факто, но не зафиксировано

**Куда вшить:**

1. **`agents/director.md`** — раздел «Recovery-протокол после прерывания»:
   - Задача > 5 шагов или > 15 минут → создавай `agent-runtime/_progress-<тема>.md` (gitignore-зона)
   - В нём: что сделано, на каком шаге, какие следующие
   - При продолжении в новой сессии — сначала читай `_progress`, потом `git log -5`, потом продолжай
   - При завершении задачи — удали

2. **`agents/git-manager.md`** (subagent) — раздел «Feature-ветки для крупных правок кода»:
   - Правки боевых проектов клиентов (Beauty TMA, Южнобережный, HUB и т.п.), > 3 файлов или рефакторинг → ветка `agent/<задача>`, не main
   - my-architecture/ (память, агенты, скиллы) — продолжаем работать в main
   - Жориковский префикс `[agent]` в коммитах **не внедряем** — у нас другой формат (`feat(scope): ...`)

3. **`agents/devops-infra.md`** — раздел «Чекпоинт-теги перед большими операциями»:
   - Перед разбором / переносом / большим рефакторингом — `git tag before-<операция>-YYYY-MM-DD`
   - Откат: `git reset --hard <tag>`

**Что усилит:**
- **`_progress.md`** — реальная защита от прерывания. У длинных задач (разбор Жорика, сборка брендбука, упаковка воронки) контекст теряется между сессиями

**Где появляется новое:**

## VS Code Remote-SSH для работы с серверами

**Что:** настроить расширение **Remote-SSH** в VS Code так, чтобы Инна могла видеть файлы на серверах (Cheerful Marik, Model Zarek, Beget shared, Vultr VPN) **прямо в IDE как локальные**, без `ssh + nano`/`vim`.

**Почему всплыло в разборе:** при анализе запчасти 1.14 (VS Code Tunnel у Жорика) стало видно, что мы и сама инфраструктура VS Code предоставляет аналогичную возможность для самой Инны — не для клиентов её ботов, а для неё лично, как для оператора серверов.

**Что усилит:**
- Правка конфигов на серверах (n8n, systemd, .env-наличие, nginx) — мгновенно через IDE, как локальные файлы
- Я смогу видеть структуру серверных проектов через Read/Edit без `ssh+cat+grep`
- Снижается шанс опечаток в командах (правка через Edit, не через `sed`)

**Действие в реализации:**
1. В VS Code установить расширение Remote-SSH (Microsoft)
2. Добавить хосты в `~/.ssh/config` (Cheerful Marik, Model Zarek, Beget, Vultr) — у Инны уже есть aliases `vpn-london` и др., расширить набор

# КОНТРОЛЬ КАЧЕСТВА РЕАЛИЗАЦИИ

Реализация по плану — это правка ~12-15 боевых файлов агентов и создание ~5-7 новых файлов методологии и справочников. Чтобы не сломать архитектуру по ходу — **четыре точки контроля**.

## 1. Self-check Director-а после каждой правки (постоянно)

После каждого `Edit` или `Write` боевого файла — обязательный `Read` того же файла:
- Правка попала в нужный раздел?
- Не сломан соседний блок (особенно когда правим существующий раздел)?
- Текст связный, нет дублей?
- Если в файле появилась новая ссылка на другой файл — этот файл существует?

Это **базовая гигиена**, не отдельный шаг. Применяется к каждой правке без исключений. По методологии запчасти **2.3 (SOUL-coder)** — после реализации Группы 2 это станет писаным правилом в `agents/qa.md` и `agents/director.md`.

## 2. Чекпоинт-тег перед стартом реализации каждой Группы

Аналог тега `before-zhorik-decomposition-2026-05-07`, который мы поставили перед началом самого разбора.

Перед стартом реализации каждой группы:
```
git tag before-impl-group-<N>-YYYY-MM-DD
git push --tags
```

Если в процессе реализации сломали что-то критическое — `git reset --hard <tag>` возвращает к состоянию до реализации, и можно зайти заново с другой стороны. Чекпоинт-теги остаются в репо как контрольные точки.

## 3. Прогон `agents/skill-auditor.md` после реализации каждой Группы

Скилл-аудитор — независимый агент, он проходит по изменённым файлам и проверяет:
- Нет противоречий между правилами в разных файлах (например: запчасть 1.6 «жди подтверждения» отброшена, нет ли остаточных следов в `director.md`?)
- Все ссылки между файлами рабочие (нет битых путей)
- Соблюдена структура frontmatter в новых файлах (`model:`, `name:`, `description:`)
- Нет дублирующихся правил (одно правило в двух местах = технический долг)
- Реализация совпадает с планом (нет «забытых» запчастей или внедрённых не туда)

Запускается **после реализации каждой группы**, не в конце всего. Если аудитор находит проблемы — фиксим, потом следующая группа.

## 4. Финальный QA-прогон через `agents/qa.md` после полной реализации

QA — приёмщик результата по чек-листу. Проверяет общую картину:
- Все ли запчасти из плана внедрены?
- Что отложено в холодильник — реально ли там, с правильным триггером разморозки?
- Что пропущено — пропущено осознанно (с причиной в плане) или забыто?
- Бонусные инфраструктурные апгрейды (VS Code Remote-SSH и др.) реализованы или вынесены в отдельный задачник?
- Дневник `memory/sessions/` отражает все этапы реализации?
- auto-memory обновлена (если появились новые правила работы)?

Вердикт QA: «✅ готово к закрытию задачи» / «🔄 возврат на доработку с конкретным списком».

## 5. Опциональный `/ultrareview` от Инны перед закрытием задачи

Облачный многоагентский ревью текущей ветки. **Запускается только Инной** — я не могу. Имеет смысл после прохождения четырёх предыдущих точек, как финальная независимая проверка от внешних агентов на качество архитектурных изменений.

Не обязательно: если QA вынес «✅ готово», и self-check + skill-auditor не нашли проблем — Инна может закрыть задачу без ultrareview. Если есть сомнения по крупным архитектурным изменениям — лучше прогнать.
