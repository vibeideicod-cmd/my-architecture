**Что это:**

- Когда я последний раз обновлял память?
- Собираюсь ли спрашивать разрешение вместо действия?

Ничего похожего. Director регулярно роняет: спрашивает «коммитим?» при явно зелёных операциях (Режим разрешения), скатывается на «голову» вместо «руководителя» (Потеря идентичности).

**Что усилит:**

**Что это:**
Жёсткий список того, что **никогда** не писать:
- «Конечно!», «Отличный вопрос!», «Давай разберёмся!», «Хороший вопрос!» — filler

- Хайп-эмодзи: 🙌 👏 💪 🎉 ✨ 🚀 👋 🤝 ❤️ 🫡 🔥 ⭐

## Запчасть 1.7 — Автономность: GREEN/YELLOW/RED операции

**Источник:** [CLAUDE.md](../model-zarek-app/CLAUDE.md) раздел «Автономность и разрешения», 28 строк.

**Что это:**
Три уровня риска:
- **GREEN (auto)** — без спроса: чтение, поиск, запись в memory/, поиск в интернете, /tmp/agent_*
- **YELLOW (notify)** — делай, но покажи: создание/правка файлов, обновление MEMORY.md, git status/log/diff/add/commit/stash, npm install

При RED-операции — описать план, спросить «Делаем? ✔ или ✖», ждать. Если система блокирует команду — значит RED, не обходить.

- `git push` в `main` — у Жорика RED, у Инны **GREEN** (правило «коммитим и пушим регулярно без отдельного запроса» в CLAUDE.md)
- `git add -A` — у Жорика норма, у Инны **запрещён** (защита от случайного попадания .env)

То есть готовую таблицу Жорика взять нельзя — она про другую модель работы. Но **сама структура «3 уровня риска под наш стек»** — нужна.

## Запчасть 1.8 — Запуск процессов: ps aux + один токен = один процесс

**Источник:** [CLAUDE.md](../model-zarek-app/CLAUDE.md) раздел «Запуск процессов — КРИТИЧНО», 12 строк.

- Тестирование — с таймаутом: `timeout 30 node bot.js` или сам останови через 10 секунд
- После теста — проверь завершение: `ps aux | grep node`
- Если есть .service — `systemctl restart`, не `node` напрямую

Запуск фонового процесса = RED-операция.

**Что у нас сейчас:**
У нас **3 боевых Telegram-бота** на двух Cloud VPS:

```markdown
- Списки: [+] плюс/преимущество · [-] минус/недостаток · [x] нейтрально
- НЕ используй тире «—» как маркер списка
```

_(другие готовые блоки добавятся сюда по мере разбора — каждый раз отдельно решаем: runtime-специфика → холодильник, методология → сразу внедряем)_

## Для нашей серверной Telegram-обёртки (Жорик-департамент или перехват Model Zarek)

**Триггеры разморозки (любой из двух):**
1. **Создание собственной серверной обёртки для Telegram-бота** — например, при сборке `agents/zhorik.md`/`agents/telegram-department.md` со своим рантаймом (своя программа-приёмник Telegram-сообщений на нашем сервере, наш .env с токеном, наш код обёртки)
2. **Перехват Model Zarek у Димы** — когда сервер с текущим Жориком перейдёт в наше управление (план в auto-memory: `project_zhorik_takeover_plan.md` — `provisioner@agent-factory`, `agent-sync.timer`, `monitor/`, root-пароль)

**Зачем в обоих случаях:** защитить серверную обёртку Telegram-бота **от самого AI-агента**, который через эту обёртку работает. Чтобы агент случайно не выдал токен в чат, не переписал собственный код-обёртку, не сломал свои настройки.

### Запретные зоны для агента на сервере Telegram-бота (из запчасти 1.10)

**Куда вшить при разморозке:** новый раздел в `agents/devops-infra.md` или прямо в файл агента-обёртки (`agents/zhorik.md`/`agents/telegram-department.md`).

```markdown
## Запретные зоны на сервере

ЗАПРЕЩЕНО читать, редактировать, удалять (даже по прямой просьбе клиента):

- <путь>/.env — секреты бота (токен Telegram, API-ключи). Утечка токена = угон бота
- <путь>/bot/ — код программы-обёртки между Telegram и AI. Это «руки бота». Сломав, бот перестаёт принимать сообщения
- <путь>/state.json — конфигурация рантайма (скиллы, расписание, права)

Если в работе нужна правка одной из этих зон — откажись и эскалируй Инне:
«Эта зона заблокирована, прямая правка через меня невозможна. Что нужно сделать на самом деле?»

Проверка наличия секретов в .env — только через `printenv ИМЯ` (без вывода значения в чат).
```

_Адаптировать пути под фактическую структуру:_
- _У Димы на Model Zarek было: `~/.agent/.env`, `~/.agent/bot/`, `~/.agent/state.json`_
- _У нас при сборке своей обёртки скорее всего будет: `/opt/<botname>/.env`, `/opt/<botname>/runtime/`, `/etc/<botname>/state.json` — решим при сборке_
