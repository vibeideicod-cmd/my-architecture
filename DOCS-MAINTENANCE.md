# DOCS-MAINTENANCE — регламент обновления документации

Дата обновления: 2026-05-15

Этот файл отвечает на вопрос: **как не дать архитектуре снова превратиться в набор разрозненных документов**.

## Принцип

Документация обновляется не "когда-нибудь", а в момент, когда меняется реальная работа:

- появился новый клиент;
- изменился текущий фокус;
- появилось больше 5 файлов в папке;
- один документ стал устаревшим, но его всё ещё можно случайно открыть;
- добавился новый тип задач или новый агент;
- из обучения пришла методика, которую решили применить.

## Еженедельный ритуал

Владелец: Director + Planner.  
Поддержка: Librarian, Project-Manager, QA.

Раз в неделю пройти 6 вопросов:

1. `PLAN.md` отражает реальный фокус недели?
2. В активных клиентских папках есть локальные `FILES.md`, если файлов стало много?
3. В `FILES.md` нет активных документов, которые на самом деле устарели?
4. `DISPATCH-README.md` и `agents/routing-table.md` не противоречат друг другу?
5. В `RESPONSIBILITY-MATRIX.md` у живых справочников есть владельцы?
6. По каждому активному клиенту понятен следующий шаг и кто его делает?

## После каждой значимой задачи

Если менялись клиентские материалы:
- обновить `clients/<name>/project-log.md`;
- обновить `clients/<name>/FILES.md`, если появился новый файл или финальная версия;
- обновить `clients/<name>/PLAN.md`, если изменилась фаза.

Если менялась архитектура:
- обновить `PLAN.md`;
- обновить корневой `FILES.md`;
- при необходимости обновить `DISPATCH-README.md`, `routing-table.md`, `team-overview.md`;
- записать решение в `memory/sessions/YYYY-MM-DD.md`.

## Smoke-test архитектуры

Перед переходом от архитектурной фазы обратно к клиентской работе прогнать 3 сценария:

| Сценарий | Что проверяем |
|---|---|
| Задача по существующему клиенту | Директор находит `clients/<name>/PLAN.md`, `FILES.md`, `brief.md` и правильный департамент |
| Новый клиент | Директор идёт через `clients/_template/`, создаёт минимальный слой и фиксирует клиента в картах |
| Материал из обучения | Librarian сравнивает методику с нашей архитектурой и предлагает только полезное усиление |

Результат smoke-test фиксировать в `ARCHITECTURE-QA-YYYY-MM-DD.md` или в существующем QA-файле, если это продолжение той же проверки.

## Что считать долгом

Документационный долг появляется, если:

- есть 2+ источника правды по одному вопросу;
- активный файл не упомянут в ближайшем `FILES.md`;
- новый процесс есть в разговоре, но не зафиксирован в документе;
- клиентская папка выросла, но не имеет `FILES.md`;
- агенту нужно догадываться, куда идти.

## Быстрое правило

Если документ нужен агенту для действия, у него должны быть:

- понятный статус;
- владелец;
- дата обновления;
- ссылка из ближайшей карты файлов.
