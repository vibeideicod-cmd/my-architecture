# Global Architecture Audit — my-architecture

Дата: 2026-05-15  
Статус: первичный каскадный аудит всего проекта

## 1. Короткий вывод

Проект уже не является просто папкой с клиентскими материалами. Это рабочая бизнес-операционная система Инны:

- корневая архитектура и правила работы;
- агентная команда;
- личный контур Инны;
- партнёрский контур Ирины;
- клиентские проекты;
- knowledge-база;
- входящие материалы и обучение;
- production-страницы, деплой и инфраструктура.

Главная проблема сейчас не в отсутствии документов, а в смешении контуров и статусов. У части зон уже есть понятные `PLAN.md`/`CLAUDE.md`, но локальные `FILES.md` есть только у `clients/_template/` и `clients/sveta/`. Поэтому агентам всё ещё приходится угадывать, где финал, где черновик, где архив, где входящие.

## 4. Ключевые риски

1. **Смешение контуров.** `clients/` содержит не только клиентов, но и внутренние продукты, Ирину, МГ. Это допустимо временно, но требует строгой классификации.
2. **Дубли Инны и Ирины.** Есть `inna/` и `clients/inna/`, есть `irina/` и `irina/`. Нужно решить, где стратегический контур, а где production/deliverables.
3. **Пустая или битая зона `clients/neuro-babki/`.** `README.md` ссылается на `clients/neuro-babki/about.md`, но в папке виден только `.DS_Store`.
4. **Тяжёлые входящие.** `inbox-inna/` занимает около 199M, `knowledge/` около 107M, `clients/` около 137M.
5. **Локальные карты есть не везде.** `FILES.md` есть только у корня, шаблона и Светы. Для `inna/`, `irina/`, `clients/beauty/`, `irina/`, `clients/mastergroup/`, `clients/yuzhnoberezhniy/` их не хватает.
6. **Служебные файлы.** `.DS_Store` встречается в 14 местах. Это мусор, можно удалить отдельной безопасной операцией.
7. **Входящие уже частично обработаны, но не закрыты.** Например, в `inbox20/` есть готовый текст Димы и аудио `корректировка от ирины.ogg`.

## 5. Анализ по папкам верхнего уровня

### `.claude/`

Назначение: локальные агенты, hooks, skills и настройки Claude/Codex.

Что хорошо:
- есть большое количество skills под департаменты;
- есть hook для коммитов;
- `settings.local.json` игнорируется.

Риски:
- дублирование между `.claude/skills/*` и `agents/*.md`;
- возможная рассинхронизация: агент обновлён, skill нет.

Рекомендации:
- сделать ревизию skill ↔ agent: какие skills актуальны, какие устарели;
- создать/обновить карту skills в `knowledge/tools/skills-catalog.md`;
- не хранить секреты в `.claude/settings.json`.

### `_архив-возможно-удалить/`

Назначение не подтверждено. Сейчас виден только `.DS_Store`.

Рекомендации:
- если там действительно нет полезных файлов, удалить папку;
- перед удалением проверить `git ls-files _архив-возможно-удалить`.

### `agent-runtime/`

Назначение: временные рабочие файлы агентов.

Что видно:
- есть `agent-runtime/_progress-hub-mvp.md`;
- есть README в `logs/`, `outputs/`, `shared/`.

Риски:
- runtime должен быть временным, но `_progress-hub-mvp.md` похож на старый незакрытый прогресс.

Рекомендации:
- проверить актуальность `_progress-hub-mvp.md`;
- если задача закрыта — удалить runtime-прогресс;
- оставить только README и временные файлы, которые не коммитятся.

### `agents/`

Назначение: команда агентов и маршрутизация.

Что хорошо:
- есть Director, routing-table, pipeline;
- покрыты ключевые департаменты: маркетинг, продукт, аналитика, продажи, финансы, юридика, инфраструктура, QA;
- недавно добавлена связка с `DISPATCH-README.md`.

Риски:
- `team-overview.md`, `model-map.md` и реальные `agents/*.md` могут разойтись;
- Client Success / Partner Manager пока не создан, а партнёрская модель уже появилась.

Рекомендации:
- провести отдельный audit `agents/` и `.claude/skills/`;
- решить, создавать ли `client-success.md` / `partner-manager.md`;
- проверить, что каждый агент знает, где брать корневой `PLAN.md`, локальный `PLAN.md`, `FILES.md`.

### `clients/`

Назначение: сейчас это не только клиенты, а все проектные ветки команды.

Что хорошо:
- есть `clients/README.md` с классификацией типов проектов;
- есть `_template/`;

Рекомендации:

- сначала добавить локальные `FILES.md` в активные папки;
