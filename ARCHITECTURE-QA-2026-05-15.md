# Architecture QA — навигационный слой проекта

Дата: 2026-05-15

## Цель проверки

Проверить, что новый навигационный слой не просто создан, а реально встроен в работу команды:

- Директор понимает, куда идти по типу задачи;
- агенты получают маршрутизацию через `DISPATCH-README.md` и `routing-table.md`;
- текущий фокус проекта берётся из `PLAN.md`, а не из устаревшего `PLAN.md`;
- клиентские проекты получают локальные `PLAN.md`, `FILES.md`, `brief.md`, `project-log.md`;
- есть владельцы живых документов.

## Что было найдено

### 1. Директор не был подключён к новому пульту

В `agents/director.md` Директор всё ещё маршрутизировал задачи только через `routing-table.md` и брал текущие приоритеты из корневого `PLAN.md`.

Проблема: корневой `PLAN.md` уже помечен как исторический, поэтому команда могла продолжать смотреть в устаревший источник.

Статус: **исправлено**.

Теперь Директор:
- сверяется с `DISPATCH-README.md`;
- для всего проекта читает `PLAN.md` и ближайший `FILES.md`;
- для клиента читает `clients/<name>/PLAN.md`, `FILES.md`, `brief.md`;
- использует `routing-table.md` для выбора департамента/агента.

### 2. Корневой `CLAUDE.md` не знал про новый навигационный слой

Проблема: правила проекта были сильными, но в них не было явной иерархии `PLAN / FILES / DISPATCH / RESPONSIBILITY`.

Статус: **исправлено**.

Добавлен раздел "Навигационный слой проекта".

### 3. Клиентский шаблон ссылался на старый корневой `PLAN.md` как стратегический

Проблема: новые клиенты могли наследовать устаревшую ссылку на старое содержание корневого `PLAN.md`.

Статус: **исправлено**.

`clients/_template/PLAN.md` теперь ссылается на корневой `PLAN.md` как актуальный пульт проекта. Старый план агентской реструктуризации перенесён в `PLAN-legacy-2026-05-08.md`.

### 4. В шаблоне клиента не было правила локального `FILES.md`

Проблема: при росте количества документов в клиентской папке снова возникал бы хаос версий.

Статус: **исправлено**.

Что сделано:
- добавлен шаблон `clients/_template/FILES.md`;
- в `_HOW-TO-USE.md` добавлен шаг заполнения `FILES.md`;
- в `project-log.md` добавлено правило обновлять карту при росте количества документов.

### 5. Инструкция создания нового клиента ссылалась на корневой `PLAN.md`

Проблема: `clients/_template/_HOW-TO-USE.md` отправлял нового клиента в корневой `PLAN.md`, хотя он уже исторический.

Статус: **исправлено**.

Теперь инструкция отправляет:
- в `PLAN.md` — если клиент становится текущим фокусом;
- в корневой `README.md` — если клиент должен отображаться в списке активных;
- в `clients/README.md` — для классификации проекта.

## Текущая рабочая цепочка

```text
Инна ставит задачу
      ↓
Director читает CLAUDE.md
      ↓
Director определяет уровень задачи
      ↓
DISPATCH-README.md говорит, куда идти
      ↓
routing-table.md говорит, какой департамент/агент нужен
      ↓
локальный PLAN.md / brief.md / FILES.md дают контекст
      ↓
агент выполняет работу
      ↓
QA проверяет результат
      ↓
project-log.md / PLAN.md / FILES.md обновляются при необходимости
      ↓
git commit + push
```

## Проверка на клиенте Светлана

Папка `clients/sveta/` теперь проходит минимальный стандарт активного клиента:

- есть локальный контекст `CLAUDE.md`;
- есть текущий план `PLAN.md`;
- есть чистый бриф `brief.md`;
- есть журнал работы `project-log.md`;
- есть карта файлов `FILES.md`;
- финальный доклад явно помечен как активный финал;
- старые версии помечены как черновики/устаревшие.

Статус: **пройдено**.

## Что всё ещё не идеально

### 1. Не все активные клиенты приведены к стандарту

Пока стандарт применён только к Светлане.

Следующие кандидаты:
- `clients/beauty/`;
- `clients/yuzhnoberezhniy/`;
- `clients/mastergroup/`;
- `inna/`;
- `irina/`.

Решение: делать постепенно, когда возвращаемся к соответствующему направлению.

### 2. Корневой `PLAN.md` всё ещё лежит в корне

Он помечен историческим внутри самого файла и в новых картах, но физически остаётся рядом с актуальными документами.

Решение: пока не переименовывать и не переносить, чтобы не ломать ссылки. Позже можно решить отдельно.

### 3. Нет автоматического ритуала обновления документации

Димина идея "раз в неделю обновлять техническую документацию по фактическому состоянию" пока зафиксирована, но не автоматизирована.

Решение: добавить в будущий weekly-review/ритуал Planner-а.

### 4. Нет Client Success / Partner агента

Для партнёрской модели Светланы нужен контур:
- партнёрское предложение;
- проценты;
- ведение рефералов;
- фиксация договорённостей.

Сейчас это распределено между Sales + Financial + Yuridika. Отдельный агент Partner пока не создан.

## Вердикт

Навигационный слой теперь не просто добавлен, а встроен в рабочую цепочку Директора.

Архитектура стала устойчивее:
- есть текущий пульт;
- есть карта файлов;
- есть диспетчер задач;
- есть владельцы документов;
- есть локальный стандарт активного клиента.

Следующий крупный архитектурный шаг — не создавать новые файлы, а проверить применение стандарта на 1-2 других активных направлениях, когда они снова попадут в работу.
