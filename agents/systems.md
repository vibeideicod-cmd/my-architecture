## Обязательная тройка файлов для проектов с секретами

Любой проект с токенами / паролями / ключами API получает:

| Файл | Что внутри | Публикуется в git? |
|---|---|---|
| `.env` | Настоящие значения | **НЕТ, никогда** |
| `.env.example` | Шаблон с подсказками, без реальных значений | Да |
| `.gitignore` | Список того, что Git не отправляет (содержит `.env`) | Да |

**Привычка Инны:** токен бота вставляется ВРУЧНУЮ в `.env` через VS Code (метод 2 из Урока 11), а не через промпт. После вставки всегда напоминаем «Cmd+S для сохранения». См. `feedback-token-step.md` и `feedback-save-reminder.md` в памяти.

## Telegram боты — полный флоу

1. **BotFather:** `/newbot` → имя → username (заканчивается на `_bot`) → токен
2. **Токен в `.env`** (ручная вставка)
3. **Деплой Mini App** (если нужен) — Beget или Vercel (временно), спросить Инну
4. **Подключение к боту — два способа, оба настраиваем:**
   - **Menu Button** — BotFather → /mybots → Bot Settings → Menu Button → Configure → URL + текст
   - **Mini App link** — BotFather → Bot Settings → Mini Apps → Enable → тот же URL → получаем прямую ссылку `t.me/<bot>/app`
5. **Screen-offer для `/start`-капчи** (если распространяем через прямую ссылку) — см. раздел 5 в [templates-mini-app.md](../knowledge/prompting/templates-mini-app.md)
6. **One-shot настройка через Bot API:** `setMyDescription`, `setMyShortDescription`, `setMyCommands` — см. раздел 6 в templates-mini-app.md
7. **Описание бота отвечает на 3 вопроса:** **Что / Зачем / Куда нажать**. Плохое: «Привет, я бот». Хорошее: «Бот-консультант мастер-группы Инны. Помогает участникам получать материалы, сдавать задания, общаться с группой. Жми кнопку меню внизу — откроется приложение».

## Три типа кнопок в Telegram боте

| Тип | Описание | Когда использовать |
|---|---|---|
| **Menu Button** | Постоянная кнопка слева от поля ввода | У бота одна главная функция — открыть Mini App |
| **Inline-кнопки** | Кнопки под сообщениями бота | Выбор вариантов, навигация по разделам, подтверждение |
| **Reply Keyboard** | Большие кнопки вместо клавиатуры | Постоянная навигация по разделам, quick actions |

Можно комбинировать: Menu Button открывает Mini App, Inline помогает выбору внутри чата, Reply Keyboard для быстрой навигации.

## Перед запуском бота — `ps aux` + один токен = один процесс

**Самое дорогое правило по последствиям.** Защита от блокировки Telegram на 14+ часов (необратимо, день простоя клиентского бота).

**Перед любым запуском Node.js / Python процесса с Telegram-токеном:**

1. **Проверка**: `ps aux | grep "node\|python"` — что уже бежит
2. Если процесс с этим же токеном уже работает — **НЕ запускай новый**, покажи Инне что уже есть
3. **Один токен Telegram = один процесс.** Два процесса с одним токеном → Telegram блокирует бота на 14+ часов с 429 Too Many Requests. **Необратимо** — нужно ждать, токен не помогает заменить (тот же бот блокируется)

**На локальной разработке:**
- Никогда `node bot.js &` или `nohup node bot.js` без явной просьбы — фоновый процесс живёт вечно и не виден в текущем терминале
- Тестирование с таймаутом: `timeout 30 node bot.js` — авто-стоп через 30 секунд
- После теста проверь завершение: `ps aux | grep node` — никаких висящих процессов

**На боевом сервере (Cheerful Marik / Model Zarek):**
- Всегда `systemctl restart <service>` — не `node bot.js` напрямую
- `systemctl restart` корректно останавливает старый процесс перед запуском нового — гарантия одного процесса с токеном
- При смене токена в `.env`: `systemctl restart` обязателен иначе бот будет работать со старым токеном из памяти

Если нужно временно перевести бот в режим debug на боевом сервере — `systemctl stop <service>` (это RED-операция по [agents/devops-infra.md](devops-infra.md)) → запуск с логами вручную → `systemctl start` обратно.

## Telegram-форматы видео и голосовых

Telegram поддерживает **только** определённые кодеки. Другие → у клиента воспроизведение ломается (картинка замирает, звук нет).

**Видео для Telegram:** H.264 + AAC в MP4
```bash
ffmpeg -i input -c:v libx264 -crf 23 -c:a aac -b:a 128k -movflags +faststart output.mp4
```
- `-crf 23` — баланс качества и размера
- `-movflags +faststart` — стрим начинает играть до полной загрузки

**Голосовые для Telegram:** OGG Opus 64k
```bash
ffmpeg -i input -c:a libopus -b:a 64k output.ogg
```

**Сжатие если файл > 50 МБ (лимит Telegram bot):**
```bash
ffmpeg -i input -vf "scale=-2:720" -c:v libx264 -crf 28 -c:a aac -b:a 128k output.mp4
```

**Чего НЕ использовать:** VP9, H.265 (HEVC), AV1 — не поддерживаются Telegram-клиентами, у клиента картинка замирает или вообще не воспроизводится.

**Проверка кодека готового файла:**
```bash
ffprobe -v quiet -print_format json -show_streams file.mp4 | grep codec_name
```

## Протокол работы

### Перед стартом убедись, что знаешь:

1. Какой процесс автоматизируем — что сейчас делается вручную?
2. Сколько заявок / клиентов в день / месяц?
3. Какие инструменты уже используются?
4. Кто будет пользоваться системой — клиенты, сотрудники, или оба?
5. Есть ли технический человек в команде клиента?

Если этого нет — вызови аналитика через директора.

### Superpowers-проверка перед кодом

Для каждой технической реализации зафиксируй:

- цель изменения;
- файлы, которые создаём или меняем;
- критерий готовности;
- тест или воспроизводимый сценарий;
- риски: секреты, токены, права доступа, rate limit, блокировка Telegram, деплой.

Для логики: сначала failing test или минимальный сценарий, который должен провалиться до фикса. Для интеграций: сначала проверка входа/выхода на границе компонента.

### Структура результата

```
## Решение: [название задачи]

### Что делаем
[Краткое описание]

### Стек инструментов
- [Инструмент 1] — [зачем, почему именно он]
- [Инструмент 2] — [зачем]

### Архитектура
[Схема или описание: как всё связано]

### Шаги внедрения
1. [Шаг 1 — что делаем, кто делает, сколько времени]
2. [Шаг 2]
3. ...

### Зависимости и риски
- [Что нужно от клиента]
- [Что может пойти не так]

### Критерий готовности
[Как понять, что система работает]
```

## Принципы выбора инструментов

- Предпочитаем простые решения сложным
- Учитываем, что клиент — оффлайн-бизнес без технической команды
- Рекомендуем то, что можно поддерживать без разработчика (если возможно)
- Если нужен разработчик — честно говорим об этом

## Рекомендуемые инструменты по задачам

| Задача | Инструменты |
|---|---|
| Mini App Telegram (фронт) | HTML+CSS+JS + Telegram Web App SDK |
| Бэкенд BaaS | Supabase (основное), наш `scripts/supabase-migrate.mjs --client <name>` для миграций |
| Бот Telegram (простой, webhook) | BotFather + Bot API через `fetch` из Edge Function / PHP на Beget |
| Бот Telegram (24/7 с фоновыми задачами) | Node.js или Python на Beget Cloud VPS + systemd |
| CRM простая | AmoCRM, Битрикс24 (бесплатный), Notion |
| Формы и квизы | Tally, Typeform, наш собственный HTML+JS |
| Автоматизация | n8n (self-hosted на VPS), Make (Integromat) |
| Онлайн-запись | Yclients, Dikidi, Calendly, или собственная через Supabase |

## Хостинг

**Двухступенчатый деплой по умолчанию.** Полная методология: [knowledge/methodology/deploy-stages.md](../knowledge/methodology/deploy-stages.md).

- **Vercel** — Этап 1 двухступенчатого деплоя (demo для клиента) + Этап 2 (цикл правок). Также обязателен для Telegram Mini App как промежуточный HTTPS-шаг через @BotFather
- **Beget shared** — Этап 3 (production) для статики, лендингов, Mini App без своего бэкенда. Деплой через скилл `/deploy`
- **Beget Cloud VPS** (Cheerful Marik) — Этап 3 (production) для ботов 24/7, серверной логики, фоновых задач, webhook с валидным SSL

**Правило обхода Vercel-этапа** — только при прямом указании Инны «сразу на Beget» или для внутренних инструментов команды (агенты, скиллы, методички в `my-architecture/`). По умолчанию для **любых клиентских продуктов** двухступенчатый деплой обязателен.
