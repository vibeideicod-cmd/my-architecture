## Структура файлов

Каждая страница — один самодостаточный HTML-файл. Все стили пишем внутри `<style>` в том же файле. Никаких внешних CSS-фреймворков, JS-библиотек и CDN — только чистый HTML и CSS.

## Дизайн

- Фон: тёмный (`#0a0a0a` или `#103206`)
- Текст: светлый (`#e8e0d8`)
- Акцентный цвет новых страниц: оранжевый `#e86c3a`
- Акцент подбренда "Нейро Бабки" (Фаза 2): горчичный `#c99700`
- Шрифты: только Google Fonts (Montserrat + Playfair Display) или системные
- Полная палитра: `/knowledge/standards/design-system.md`

## Вёрстка

Адаптивность обязательна. Страница должна хорошо выглядеть на мобильных, планшетах и десктопе. Используем `clamp()`, `flexbox`, `grid` и медиазапросы.

## Логотипы сторонних брендов (мессенджеры, соцсети, сервисы)

Если на странице ссылка на внешний бренд — **Telegram, WhatsApp, VK, MAX, Instagram, YouTube, Дзен, Rutube** и т.д. — используем **оригинальный узнаваемый логотип в фирменных цветах бренда**. Это правило критично: пользователь должен за долю секунды узнать мессенджер/соцсеть и понять, куда ведёт кнопка.

**Нельзя:**
- подменять логотип абстрактной иконкой («конвертик», «человечек», «телефонная трубка»)
- перекрашивать логотип в монохром или в цвет палитры клиента
- использовать только текст «Telegram» без логотипа
- выдавать все мессенджеры одинаковыми кругляшами серого цвета

**Правильно:**
- официальный SVG-логотип бренда
- фирменный цвет бренда сохранён
- логотип не искажён (пропорции, углы, фирменная форма)

**Фирменные цвета основных мессенджеров:**
| Бренд | Цвет | Примечание |
|---|---|---|
| Telegram | `#229ED9` | голубой «плашка» |
| WhatsApp | `#25D366` | зелёный |
| VK | `#0077FF` | синий |
| MAX | фирменный оранжево-красный градиент | мессенджер VK — использовать актуальный логотип |
| Phone / звонок | нейтральный | можно подстроить под палитру — это не бренд |

**Где применяется:** кнопки связи с мастером/клиентом, футер с соцсетями, шеринг, любые внешние ссылки на платформы.

## Сессионная память

**Начало сессии:** Прочитай `MEMORY.md` из папки памяти — там индекс файлов с контекстом о проекте и последних сессиях.

**Дневник сессии:** Веди дневник в `memory/sessions/YYYY-MM-DD.md` (отдельный файл на каждый календарный день, дата сегодняшняя). Создавай файл при первой записи дня. Дописывай в него по ходу сессии — что сделал, какие решения приняли, что осталось открытым. **Не пиши** в старый `memory/session-log.md` — он переведён в архив (хранит историю до 2026-04-14).

**Конец сессии:** Когда Инна говорит "пока", "спокойной ночи", "до завтра" или заканчивает работу — допиши финальную сводку в дневник дня (`memory/sessions/YYYY-MM-DD.md`).

## Правило: обмен данными между агентами — через файлы, не через текст

**Когда одному агенту нужно передать результат другому** — не пересказывай данные в диалоге. Пиши в файл, читай из файла.

**Папки:**
- `agent-runtime/shared/` — промежуточные данные (ресёрч, черновики, структурированные данные)
- `agent-runtime/outputs/` — финальные результаты готовые к передаче Инне
- `agent-runtime/logs/` — короткий лог работы агента (для отладки)

**Как:**
1. Агент завершил работу → сохраняет результат в `agent-runtime/shared/<дата>-<агент>-<тема>.md`
2. Следующий агент → читает этот файл через Read tool
3. Финальный результат → в `agent-runtime/outputs/<дата>-<тема>.md`
4. Постоянное хранение → скопировать из outputs/ в `inna/`, `knowledge/` или `clients/`

**Именование:** `YYYY-MM-DD-<агент>-<тема>.md` — например `2026-04-29-analytics-competitors.md`

**Файлы в agent-runtime/ не коммитятся** — они в .gitignore. Это рабочий стол, не архив.

**Детали:** `agent-runtime/shared/README.md`

## Правило: коммитим и пушим регулярно (без отдельного запроса)

**Инна авторизовала автоматический коммит и пуш для этого репозитория.** Не спрашивай разрешения каждый раз — делай это сам, как только закрыта осмысленная единица работы.

**Когда коммитить:**
- Завершил логическую правку (новый файл, законченное усиление агента, фикс бага, добавленный раздел в документ)
- Перед сменой темы — если переключаемся с одной задачи на другую, сначала коммит того что сделали
- В конце сессии — обязательно, всё что не закоммичено должно уйти в гит

**Как коммитить:**
- Группируй изменения в **осмысленные** коммиты (не один файл = один коммит, и не «всё свалено в кучу»)
- Сообщения по-русски, в стиле существующих коммитов репо: `feat(scope): что сделали`, `fix(scope): ...`, `docs(scope): ...`, `sync(scope): ...`
- Перед коммитом всегда `git status` + `git diff` чтобы увидеть что попадает в коммит
- Стажить файлы по именам, **не** через `git add -A` или `git add .` (защита от случайного попадания .env и других секретов)
- После коммита **сразу пушим** в `origin/main` — мы работаем напрямую в main, без feature-веток

**Когда НЕ коммитить автоматически:**
- Если правка ломает что-то в уже работающем продукте (например, ломает деплой ЮБ или Beauty TMA) — сначала почини, потом коммить
- Если в diff попало что-то подозрительное (`.env`, токены, ключи API) — не коммить, предупреди Инну
- При rebase, force-push, merge-конфликтах — спрашивай Инну
- Если Инна явно сказала «не коммить пока» — слушайся, это краткосрочная блокировка

**Cmd+S напоминание:** прежде чем коммитить, напомни Инне нажать Cmd+S, если файлы открыты в VS Code (см. `feedback-save-reminder.md`).
