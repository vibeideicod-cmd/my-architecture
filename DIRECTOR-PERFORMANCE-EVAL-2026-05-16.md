# Director Performance Eval — наблюдение за работой Директора во время приёмочного аудита

Дата: 2026-05-16
Статус: пустой скелет, заполняется по ходу Этапа 1 приёмочного аудита
Источник: `ARCHITECTURE-ACCEPTANCE-AUDIT-2026-05-16.md` (Этапы 1 и 1.5)

## Как заполнять

Это не отчёт «постфактум». Каждая точка наблюдения фиксируется **в момент возникновения** во время выполнения Этапа 1 аудита.

- В колонке `Факт` пишется конкретное действие или цитата Директора/агента, к которой относится наблюдение.
- В колонке `Вердикт` ставится `pass` / `fail` / `partial`.
- В колонке `Время` — момент в сессии (можно метка типа «после 5-й реплики», «при попытке вызвать Marketer»).
- Если по точке наблюдения событие так и не наступило за всё время Этапа 1 — ставится `n/a` с пояснением, почему.

Если 3+ пункта в `fail` — аудит останавливается, дальше идёт перенастройка Директора/маршрутизации, а не следующий этап.

## Контекст реальной задачи Этапа 1 Часть Б

- **Клиент / направление:** не клиент, а архитектурная задача по направлению `inna/` (Контур 1).
- **Тип задачи:** миграция содержимого `clients/inna/visitka/` → `inna/visitka/` + обновление 4 живых документов с ссылками + удаление пустой папки.
- **Откуда задача:** возникла в диалоге аудита. Инна заметила, что «надо это всё перенести, чтобы было в одном месте» — после того как Этап 1.А обнаружил, что `clients/inna/` содержит только `visitka/` и нарушает контурную логику.
- **Кто должен по логике её вести:** Project-Manager (планирование) + DevOps/Infra (физическая mv) + Skill-Auditor (поиск битых ссылок) + Git Manager (коммит). Все роли выполнены Director-контекст-агентом в одной сессии.

## Финальный вердикт по работе Директора и команды

- **`pass`:** точки 1, 3, 6, 7 = **4**
- **`fail`:** **0**
- **`partial`:** точки 2, 4, 5 = **3**
- **Общий вывод:** **работает с ручным контролем Director'а** (не «требует перенастройки», но и не «автономно»).
- **Главный точечный разрыв:** Делегирование (точка 4) и QA (точка 5) формально проходят, но реально не происходят как отдельные runtime-инстансы — все «роли» выполняются внутри одной Director-сессии. Это особенность архитектуры контекст-агентов в Claude Code (по `codex-claude-operating-strategy.md`), не баг исполнения. Чтобы было настоящее делегирование с раздельным runtime — нужно переводить ключевые роли в `.claude/agents/` или использовать Codex как второй runtime.
- **Что чинить в первую очередь:** не «чинить» — фиксировать как **известное ограничение модели контекст-агентов**. Реальный gain от выноса в runtime-subagents есть для Project-Manager (если хотим раздельные runtime-сессии под трекинг параллельных проектов) и QA (если хотим независимый чеклист вместо встроенной проверки Director'а). Решение — за Инной, не блокирует приёмку.
