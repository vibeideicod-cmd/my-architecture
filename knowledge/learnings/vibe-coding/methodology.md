## Главная идея за 30 секунд

Вайбкодинг — это создание цифровых продуктов **без написания кода**. Вы описываете идею простыми словами, ИИ пишет код. Метод состоит из двух частей: **Vibe** (ваша идея, описанная как объяснение другу) и **Coding** (код, который ИИ создаёт за вас). Базовый алгоритм любого продукта — 5 шагов: **Идея → Исследование → Критика → План → Реализация**. Стек инструментов — VS Code + Claude Code + GitHub (для серьёзных продуктов) и Google AI Studio + Lovable (для быстрых черновиков). Главное правило: **80% работы — думать и описывать, 20% — просить ИИ сделать**.

## Урок 1 — Что такое вайбкодинг и зачем он сейчас

### 1.1. Определение
<!-- LIBRARIAN-PRINCIPLE: vibe-coding-definition -->

**Вайбкодинг** — создание цифровых продуктов без написания кода. Вы управляете процессом, ИИ делает техническую работу. Не нужно быть технарём, программистом или дизайнером.

**Vibe + Coding:**
- **Vibe** (вайб, настроение) — ваша идея, описанная простыми словами
- **Coding** (кодинг) — код, который создаёт ИИ за вас

**Вайбкодер** — человек, который создаёт приложения, описывая что он хочет получить, **а не как это запрограммировать**.

**Аналогия:** заказ в ресторане. Вы не повар, но точно знаете что хотите: «Салат Цезарь, без анчоусов, с куриной грудкой». Вы не объясняете, как нарезать овощи и жарить курицу — вы описываете результат. С вайбкодингом так же.

### 1.2. Классический программист vs Вайбкодер
<!-- LIBRARIAN-PRINCIPLE: programmer-vs-vibecoder -->

| Классический программист | Вайбкодер |
|---|---|
| Пишет код вручную | Описывает идею словами |
| Учится годами | Начинает создавать через час после изучения |
| Думает о технической реализации | Думает о пользовательском опыте |
| Ищет и исправляет баги в коде | ИИ исправляет код автоматически |
| Недели/месяцы на создание | Минуты/часы на создание |

### 1.3. Хронология «почему именно сейчас»
<!-- LIBRARIAN-PRINCIPLE: why-now-vibecoding -->

- **2024** — 70% стартапов Y Combinator используют ИИ для написания кода
- **2025** — создатель ИИ Tesla вводит термин «вайбкодинг»
- **2026** — 40% малых бизнесов начинают использовать ИИ-инструменты
- **2028 (прогноз)** — 80% продуктов создаётся без классического кода

**Аналогия:** «Вайбкодинг в 2026 = App Store в 2008». Площадка пустая, конкуренция минимальна. Через 2-3 года это будет уметь каждый второй.

**3 причины «почему именно СЕЙЧАС»:**
1. **Инструменты доступны** — Claude Code, Lovable, ChatGPT работают для всех
2. **Конкуренция минимальна** — 95% людей не знают, что это возможно
3. **Рынок растёт** — бизнесу нужны цифровые продукты, а программистов не хватает

## Урок 2 — 5-шаговый алгоритм + Личный проект

### 2.1. Почему без системы ничего не работает
<!-- LIBRARIAN-PRINCIPLE: no-system-no-result -->

**Хаотичный путь** (как делают новички): «Хочу приложение» → сразу открывают ИИ → пишут промпт → получают что-то непонятное → переделывают 15 раз → бросают через 3 дня.

**Три причины провала:**
1. **Нет чёткой идеи** — «хочу приложение» это мечта, а не идея
2. **Не изучил, что уже есть** — изобретает велосипед
3. **Не спланировал** — начинает делать всё сразу и тонет в деталях

**Факт из урока:** 70% IT-стартапов закрываются в первый год. Главная причина — не технологии, а отсутствие плана.

**Аналогия:** без системы — купил кирпичи, начал класть стены, забыл фундамент, всё развалилось. С системой — сначала чертёж, потом фундамент, потом стены. Дом стоит.

---

### 2.2. 5-шаговый алгоритм (GPS-навигатор вайбкодера)
<!-- LIBRARIAN-PRINCIPLE: 5-step-algorithm -->

Универсальный процесс создания любого цифрового продукта. Работает для лендинга, бота, приложения. **Всегда в этом порядке.**

| Шаг | Название | Что делаем |
|---|---|---|
| 1 | Идея | Формулируем ЧТО создаём, ДЛЯ КОГО и ЗАЧЕМ |
| 2 | Исследование | Смотрим существующие примеры, берём лучшее |
| 3 | Критика | Просим ИИ честно оценить идею |
| 4 | План | Разбиваем на маленькие шаги |
| 5 | Реализация | Даём задачи ИИ по одной |

**Аналогия:** GPS-навигатор. Ввели адрес — построил маршрут. Без навигатора — блуждаете наугад.

---

### 2.3. Шаг 1 — Идея (формула)
<!-- LIBRARIAN-PRINCIPLE: idea-formula-what-who-why -->

**Шаблон:** `Я создаю ЧТО для КОГО, чтобы решить проблему КАКУЮ`

**Примеры:**
- «Я создаю калькулятор стоимости ремонта для владельцев квартир, чтобы они за 2 минуты узнали примерную стоимость»
- «Я создаю лендинг для психологов, чтобы у них появился сайт для записи клиентов»
- «Я создаю Telegram-бота для владельцев кофеен, чтобы они могли отслеживать расходы»

**Идея без формулировки — это мечта. Мечта без плана — это фантазия.**

---

### 2.4. Шаг 2 — Исследование
<!-- LIBRARIAN-PRINCIPLE: research-step -->

Смотрим, что уже существует. Изучаем 5–10 примеров: что работает, что нет. Выписываем лучшие идеи.

**Зачем:** понять что уже работает, вдохновиться удачными решениями, избежать типичных ошибок, собрать референсы для ИИ.

**3 способа:**
1. Через ИИ (Claude / ChatGPT / Gemini) — попросить найти 5-7 примеров
2. Через Google — искать «лучшие сайты [ниша]»
3. Через галерею Lovable — посмотреть готовые проекты

**Аналогия:** шеф-повар не придумывает борщ заново. Он изучает 10 лучших рецептов и собирает свой.

---

### 2.5. Шаг 3 — Критика (рентген идеи ДО разработки)
<!-- LIBRARIAN-PRINCIPLE: critique-before-build -->

Находим слабые места ДО начала работы. Даём ИИ свою идею + результаты исследования и просим раскритиковать.

**Что спрашиваем:**
- Что может не сработать?
- Чего не хватает для успеха?
- Какие секции/элементы обязательно должны быть?
- Какие типичные ошибки в таких проектах?
- Как улучшить мою идею?

**Принцип:** лучше найти проблему за 3 минуты в чате с ИИ, чем через 5 часов работы обнаружить, что продукт никому не нужен.

**Аналогия:** критика — это не нападение, это рентген идеи. Лучше увидеть трещину ДО стройки, чем когда стены уже стоят.

---

### 2.6. Шаг 4 — План (LEGO-принцип)
<!-- LIBRARIAN-PRINCIPLE: plan-as-lego -->

Разбиваем проект на маленькие шаги. Описываем итоговый результат, делим на 3–5 этапов. **Каждый этап = одна задача для ИИ.**

**Пример: калькулятор ремонта**
1. Создать форму ввода (площадь, комнаты, тип ремонта)
2. Добавить расчёт стоимости
3. Добавить красивый дизайн
4. Добавить кнопку «Получить смету на email»
5. Опубликовать по ссылке

**Аналогия:** собирать LEGO по инструкции — легко. Высыпать все детали на пол и надеяться, что само соберётся — нет.

---

### 2.7. Шаг 5 — Реализация (по одной задаче за раз)
<!-- LIBRARIAN-PRINCIPLE: one-task-one-prompt -->

Берём этап из плана → пишем промпт → проверяем результат → следующий этап.

**Аналогия:** конвейер на заводе. Одна деталь за другой. Не пять одновременно.

---

### 2.8. Личный проект — 3 критерия хорошей идеи
<!-- LIBRARIAN-PRINCIPLE: personal-project-criteria -->

| # | Критерий | Проверка |
|---|---|---|
| 1 | Вам лично интересно | Будете работать над этим 7 недель |
| 2 | Кому-то это нужно | Есть аудитория, которая заплатит или будет пользоваться |
| 3 | Это реально сделать | Не «построить новый Яндекс», а конкретный продукт |

Если все три галочки стоят — идея хорошая. Если хотя бы одной нет — пересмотрите.

## Урок 3 — Настраиваем рабочее место

### 3.1. Стек вайбкодера
<!-- LIBRARIAN-PRINCIPLE: vibecoder-stack -->

| Инструмент | Роль | Стоимость |
|---|---|---|
| **VS Code** | Мастерская — программа, где живут все ваши проекты (как Word для документов, только для сайтов и приложений) | Бесплатно |
| **Claude Code** | Напарник — ИИ-исполнитель, живёт прямо в VS Code, сам создаёт файлы и пишет код | Pro $20/мес — для старта; Max $100–200/мес — для активной работы |
| **GitHub** | Облачное хранилище — код в облаке, история всех изменений, можно откатиться | Бесплатно |

---

### 3.2. ChatGPT vs Claude Code — главное отличие
<!-- LIBRARIAN-PRINCIPLE: chatgpt-vs-claude-code -->

- **ChatGPT в браузере — советчик.** Даёт ответ, дальше делаете сами (копируете, вставляете, разбираетесь).
- **Claude Code — исполнитель.** Сам создаёт файлы, сам пишет код, сам сохраняет.

### 3.3. Правила именования папок проекта
<!-- LIBRARIAN-PRINCIPLE: project-folder-naming -->

Каждый проект живёт в своей папке. Без папки Claude не знает, куда сохранять файлы.

**Структура:**
```
Documents/
└── Projects/
    └── my-first-project/  ← папка проекта
```

**Правила именования:**
- ✅ Только латиница: `my-project`, не `мой-проект`
- ✅ Без пробелов: дефис вместо пробела
- ✅ Без спецсимволов: никаких `!`, `#`, `@`, скобок
- ✅ Всё строчными: `my-first-project`, не `My First Project`

<!-- LIBRARIAN-PRINCIPLE: claude-code-modes -->

| Режим | Что делает | Когда использовать |

### 3.6. GitHub: commit и push (ритм сохранений)
<!-- LIBRARIAN-PRINCIPLE: commit-push-rhythm -->

**Два слова:**
- **Commit (коммит)** — сохранился на своём компьютере
- **Push (пуш)** — загрузил сохранение в облако

**Аналогия:** Git — как сохранение в видеоигре. Прошёл уровень — сохранился. Погиб — загрузил сохранение.

| Когда | Что сказать Claude |

**Если что-то сломалось:** «Покажи историю коммитов» → «Верни проект к коммиту [название]» → проект вернётся к рабочему состоянию.

### 4.2. Google AI Studio — бесплатный тренажёр
<!-- LIBRARIAN-PRINCIPLE: google-ai-studio-as-sandbox -->

Google AI Studio — это AI-чат от Google. Работает в браузере, ничего устанавливать не нужно. **Бесплатный.** На базе модели Gemini.

**Зачем нужен, если есть Claude Code:** Claude Code платный, каждое сообщение стоит денег. В AI Studio можно тренироваться, экспериментировать и ошибаться сколько угодно — бесплатно.

**Метафора:** Claude Code — рабочая стройка, где каждый кирпич стоит денег. AI Studio — песочница, где можно пробовать и учиться без последствий.

**Два режима:**
- **Playground** — обычный чат (как ChatGPT, только бесплатно)
- **Build** — конструктор приложений: AI рисует работающее приложение прямо в браузере

**Что можно делать в Playground:**
- Написать текст (пост, описание, письмо клиенту)
- Составить план (ТЗ, структура сайта)
- Разобраться в теме
- Проанализировать файл (фото, документ, таблицу)
- Диктовать голосом

**Что можно делать в Build:**
- Калькулятор (стоимости услуг, калорий, ипотеки)
- Квиз или тест
- Простая форма (заявка, обратная связь)
- Набросок будущего сайта (бесплатно проверить идею до Lovable)

**Доступ из России:** напрямую не открывается, но решение находят за один день — спрашивайте в сообществе.

### 4.3. Lovable — конструктор сайтов
<!-- LIBRARIAN-PRINCIPLE: lovable-as-lego -->

Lovable — сервис, собирающий красивые сайты в браузере. Описываете словами — через минуту видите готовый результат.

**Метафора:** если Claude Code — стройка, где вы контролируете каждый кирпич, то Lovable — конструктор LEGO. Быстро, красиво, но только из тех деталей, что есть в наборе.

**Что умеет Lovable:**

- Сразу публиковать в интернет (без настройки)
- Подключать базу данных
- Подключать оплату

**Чего Lovable НЕ умеет:**
- Создавать Telegram-боты (только сайты)
- Работать с файлами на вашем компьютере
- Давать полный контроль над кодом

**Стоимость:** бесплатно 5 генераций в день (попробовать), Pro — $25/мес.

**Что можно подключить:**

**Галерея Lovable** — тысячи готовых проектов. Прежде чем создавать что-то с нуля, **всегда** ищите похожее в галерее. Нашли — нажимаете кнопку **Remix** и получаете копию для переделки. Быстрее и дешевле.

## Урок 5 — Учим ИИ понимать вас + Паспорт проекта

### 5.1. Три главные ошибки в запросах к ИИ
<!-- LIBRARIAN-PRINCIPLE: three-prompt-errors-vibecoding -->

| Ошибка | Что не так | Аналогия |
|---|---|---|
| **Слишком размыто** | «Сделай мне сайт» — без контекста | Сказать таксисту «отвези меня куда-нибудь» |
| **Слишком много сразу** | «Сделай сайт с регистрацией, каталогом, корзиной, оплатой и личным кабинетом» одним промптом | Попросить повара одновременно приготовить завтрак, обед и ужин |
| **Нет контекста** | «Напиши текст для главной страницы» — какой страницы? Для кого? О чём? | ИИ не умеет читать мысли |

**Аналогия:** запрос к ИИ — как заказ в ресторане. «Принесите что-нибудь вкусное» — несут стейк, а вы вегетарианец. «Салат Цезарь, без анчоусов, соус отдельно» — именно то, что хотели.

---

### 5.2. Формула 5 элементов хорошего запроса
<!-- LIBRARIAN-PRINCIPLE: five-elements-prompt -->

Не обязательно использовать все пять каждый раз. Но чем больше элементов — тем точнее результат.

| # | Элемент | Что это | Пример |
|---|---|---|---|
| 1 | **Контекст** | Что за проект, какая ситуация | «Я делаю сайт для кофейни» |
| 2 | **Роль** | Кем должен быть ИИ | «Ты опытный веб-дизайнер» |
| 3 | **Задача** | Что именно нужно сделать | «Создай главную страницу» |
| 4 | **Формат** | В каком виде получить результат | «Один HTML-файл, адаптивный» |
| 5 | **Ограничения** | Чего делать не нужно | «Без внешних библиотек» |

**Главный принцип:** опишите **образ конечного результата**. Что вы хотите увидеть, когда откроете файл? Если не знаете — попросите Claude: «Подготовь подробный промпт для самого себя и выполни его».

---

### 5.3. Техника «Раскритикуй»
<!-- LIBRARIAN-PRINCIPLE: critique-technique -->

Получили результат, выглядит красиво. Но вы делаете не для себя — для пользователей. Посмотрим на результат их глазами.

**Claude может посмотреть на результат с разных сторон:**
- Глазами пользователя (что непонятно? что неудобно?)
- Глазами UI/UX-дизайнера (что выглядит непрофессионально?)
- Глазами тестировщика (что может пойти не так?)
- Глазами владельца конкурирующего бизнеса (что бы сделал лучше?)

**Не стесняйтесь спорить с ИИ. Он не обидится. Зато получите лучший результат.**

---

### 5.4. Техника «Итерация»
<!-- LIBRARIAN-PRINCIPLE: iteration-technique -->

Не нужно писать идеальный запрос с первого раза. Это миф. Работайте короткими уточнениями — шаг за шагом.

**Пример:**
1. «Добавь кнопку «Новая неделя» для сброса прогресса»
2. «Сделай чекбоксы крупнее для мобильных»
3. «Добавь подсказки при наведении на каждую привычку»

Три коротких запроса. Каждый — по одному замечанию из критики. Результат стал в разы лучше первой версии.

**Аналогия:** итерация — как лепить из глины. Сначала форма, потом детали, потом полировка. Не пытайтесь вылепить шедевр одним движением.

---

### 5.5. Цикл вайбкодера (6 шагов)
<!-- LIBRARIAN-PRINCIPLE: vibecoder-cycle-6-steps -->

| Шаг | Что делаем | Пример |
|---|---|---|
| 1 | Представьте образ конечного результата | «Хочу трекер привычек с прогресс-баром» |
| 2 | Опишите его словами (или попросите Claude помочь) | Пишете подробный промпт |
| 3 | Посмотрите результат в браузере | Открыли файл, проверили |
| 4 | Раскритикуйте глазами пользователя | «Что неудобно? Чего не хватает?» |
| 5 | Улучшите итерациями | Короткие уточнения по одному |
| 6 | Повторяйте до идеала | Снова критика → снова улучшение |

**Формула:** Создал → Раскритиковал → Улучшил → Снова раскритиковал.

---

### 5.6. Техника «Ты уверен?»
<!-- LIBRARIAN-PRINCIPLE: are-you-sure-check -->

Главный инструмент проверки. Claude иногда уверенно говорит вещи, которые не совсем точны. Не нужно проверять каждое слово, но **ключевые утверждения** — спрашивайте: «Ты уверен?»

---

### 5.7. Документация — память вашего проекта
<!-- LIBRARIAN-PRINCIPLE: documentation-as-memory -->

Вы закрыли чат — Claude не помнит, над чем вы работали. Не знает, что за проект, какие файлы, какой дизайн вы выбрали. **Каждый новый разговор — это как новый сотрудник на первый день.** Талантливый, но ничего не знает о вашем проекте.

**Решение:** записка на холодильнике. Любой новый чат с Claude — сразу в курсе дела.

**Главное правило:** **80% работы вайбкодера — думать и описывать. 20% — просить ИИ сделать.** Без документации каждый новый чат — объяснение с нуля. С ней — Claude сразу в курсе.

**Два типа документации:**
- **Проектная** — описание всего проекта целиком (что, для кого, зачем)
- **Техническая** — описание конкретной функции перед реализацией

---

### 5.8. Три файла, которые решают проблему памяти
<!-- LIBRARIAN-PRINCIPLE: three-memory-files -->

| Файл | Что внутри | Зачем |
|---|---|---|
| **`project.md`** | Паспорт проекта: что это, для кого, какие функции, чего НЕ будет на старте | База знаний проекта, к которой вы и Claude возвращаетесь |
| **`CLAUDE.md`** | Правила работы с проектом: дизайн, технологии, стиль, что нельзя | Claude **читает автоматически** при каждом запуске |
| **`PLAN.md`** | План работ как чек-лист с галочками | Что сделано, что в процессе, что дальше |

**Как проверить, что CLAUDE.md работает:** откройте новый чат, дайте короткий запрос, в котором специально НЕ упоминайте стиль («Создай страницу `pricing.html`»). Если Claude сделал тёмный фон и оранжевый акцент без подсказки — файл работает.

---

### 5.9. Структура `project.md`
<!-- LIBRARIAN-PRINCIPLE: project-md-structure -->

| Раздел | Что писать |
|---|---|
| Что это | Краткое описание в 1-2 предложения |
| Для кого | Целевая аудитория |
| Проблема | Какую боль решает |
| Основные функции | Что проект умеет (MVP — что делаем первым) |
| Страницы / экраны | Что увидит пользователь |
| Чего НЕ будет (пока) | Что сознательно не делаем — важно для фокуса |
| Конкуренты | Краткий обзор 3-5 штук |
| Технологии | Что будем использовать |
| Первые шаги | С чего начать разработку |

**Совет:** раздел «Чего НЕ будет» — один из самых важных. Без него вы будете бесконечно добавлять «ещё одну фичу» и никогда не закончите.

---

### 5.10. Цикл улучшения документации
<!-- LIBRARIAN-PRINCIPLE: documentation-improvement-cycle -->

Документация готова, но не идеальна — это нормально. Прогоните её через цикл 2–3 раза:

| Шаг | Действие |
|---|---|
| 1 | **Перечитайте** как будто видите впервые. Понятно ли человеку со стороны? Нет ли противоречий? |
| 2 | **«Ты уверен?»** — проверяйте ключевые утверждения. Claude иногда уверенно говорит неточные вещи. |
| 3 | **Критический взгляд** — попросите Claude найти слабые места: «Какие слабые места? Что может пойти не так? Какие функции обычно забывают?» |
| 4 | **Пересмотрите** — обновите документацию с учётом найденного. |

**Когда остановиться:** новые правки мелкие и косметические. Нет явных дыр. По документации можно начать работу — и не переспрашивать каждые 5 минут «а что тут имелось в виду?»

---

### 5.11. Лайфхак: попросите Claude составить план реализации сам для себя
<!-- LIBRARIAN-PRINCIPLE: claude-writes-plan-for-itself -->

У вас есть паспорт проекта, Claude его видит. Теперь можно попросить его поставить задачу **самому себе:**

```
Изучи паспорт проекта (project.md) и исследование.
На основе этого составь подробный промпт для самого себя — пошаговый план реализации MVP.

Учти:
- Код пишешь ты, не человек
- Начни с самого простого — чтобы через 10 минут уже был результат
- Каждый шаг должен давать видимый результат
- После каждого шага — проверка
- Комментируй, что делаешь — я новичок и хочу учиться

Составь план и выполни первый шаг.
```

**Принцип:** вы не придумываете задачу для ИИ — вы просите ИИ самому себе поставить задачу, опираясь на готовый паспорт. Результат точнее.

## Урок 6 — Исследование ниши и первый сайт

### 6.1. Сайт = ваша земля, соцсети = арендованная
<!-- LIBRARIAN-PRINCIPLE: site-is-your-land -->

**Главный принцип:** «Сайт — единственное, что вы реально контролируете. Социальные сети — это арендованная земля. Одно обновление алгоритма — и мы невидимы.»

**Что не работает в 2026:**
- Instagram (запрещён в РФ, охваты упали в 10 раз)
- Telegram-канал — «молитва и надежда» (подписчики листают не читая)
- ВКонтакте — живы только мамы и школьники

**Где реальные клиенты:**
- Яндекс / Google (поиск)
- Сами сайты (ссылки из статей, поисковая выдача)
- Рекомендации (нужна репутация — а у репутации должен быть адрес)

**Цифры из урока:**
- 87% B2B клиентов изучают сайт компании перед покупкой

- Стартапы с сайтом привлекают инвестиции в 3 раза чаще

### 6.2. Три технологии, из которых состоит каждый сайт
<!-- LIBRARIAN-PRINCIPLE: html-css-js-metaphor -->

| Технология | Метафора | Что делает |
|---|---|---|
| **HTML** | Скелет | Кости, структура: заголовки, абзацы, кнопки, картинки. Без HTML — нет страницы |

### 6.3. Фронтенд vs бэкенд (для первого проекта — только фронтенд)
<!-- LIBRARIAN-PRINCIPLE: frontend-only-for-first -->

**Метафора ресторана:**
- **Фронтенд (зал для гостей)** — столики, меню, официант, красивая подача. То, что видит клиент.
- **Бэкенд (кухня)** — повара, холодильники, плита, склад. То, чего клиент не видит.

**Что входит во фронтенд:** кнопки, тексты, картинки, формы ввода, меню, цвета и дизайн, анимации.

**Что входит в бэкенд:** база данных, логика обработки, сервер, авторизация, платежи, отправка email.

**Для первого проекта — только фронтенд.** Красивая страница в браузере, без базы данных и авторизации. Этого достаточно для лендинга и портфолио.

## Урок 10 — Исследование ниши для Mini App

### 10.1. Telegram Mini App — определение
<!-- LIBRARIAN-PRINCIPLE: mini-app-definition -->

**Telegram Mini App** — приложение (по сути сайт), которое открывается внутри мессенджера Telegram. Технически — обычный сайт на HTML + CSS + JavaScript, упакованный для мессенджера. Адреса в браузерной строке не видно, всё работает прямо в Telegram, без перехода в браузер.

**Главный принцип:** «Это навык, не привязанный к Telegram». Мессенджеры меняются (WhatsApp, MAX, WeChat) — принцип остаётся. Фундамент — умение создавать сайты — вечен. То, что вы здесь получите, работает в любом мессенджере, который запустит Mini App.

**Контекст рынка:**
- Telegram — 2,5 млрд людей в мессенджерах каждый день. Средний человек проверяет мессенджер 23 раза в день
- MAX (русский мессенджер VK, 75 млн пользователей) запустил Mini App по тому же принципу
- WeChat (Китай) — миллиарды пользователей в Mini App
- WhatsApp — тестирует похожий функционал

---

### 10.2. Когда Mini App нужен, а когда нет
<!-- LIBRARIAN-PRINCIPLE: mini-app-when-needed -->

**Когда Mini App имеет реальное преимущество:**
- Приложение должно узнавать пользователя без логина (запись, личный кабинет)
- Нужна обратная связь с ботом (уведомления, подтверждения, напоминания)
- Важно нативное ощущение — повышает доверие и конверсию
- Планируется оплата — у Telegram есть встроенные механизмы

**Когда Mini App НЕ нужен:**
- Простая визитка → ссылка на сайт работает
- Mini App — не волшебная таблетка

**Принцип:** «Не усложняйте ради усложнения. Если сейчас не нужно полноценное Mini App — не делайте. Простая ссылка работает для простых задач».

### 10.3. Исследование ниши под Mini App (5 примеров)

```

Добавь блок с отзывами — 3 коротких отзыва.
```

### 11.5. Безопасное хранение токена — `.env` + `.gitignore`
<!-- LIBRARIAN-PRINCIPLE: env-gitignore-pattern -->

**Три файла, которые встречаются в каждом проекте:**

| Файл | Что внутри | Публиковать? |
|---|---|---|
| `.env` | Настоящие токены и пароли | ❌ Нет, никогда |
| `.env.example` | Подсказка: какие ключи нужны | ✅ Да, безопасно |
| `.gitignore` | Список файлов, которые Git не отправляет | ✅ Да, нужно |

**Метафоры:**
- `.env` — сейф в офисе. Внутри ключи от всех замков. Каждый проект, в котором есть хоть один секрет — должен иметь свой `.env`. Без исключений
- `.env.example` — витрина сейфа в магазине. Показывает, какие полки внутри, но без ценностей
- `.gitignore` — охранник на проходной. У него список «не трогать. Сейф — не выносить. Личные вещи — не выносить»

**Правило:** в любом проекте, где есть токены, пароли или ключи — должны быть эти три файла. Claude Code знает это и создаст их автоматически. Но теперь вы понимаете, зачем они нужны.

**Где НЕЛЬЗЯ отправлять токены — никогда:**
- В публичные чаты и каналы Telegram
- В комментарии на GitHub (даже если потом удалите — история сохраняет всё)
- В сообщения на форумах и в соцсетях
- В код напрямую (вместо `.env`)

**Совет автора:** «Привычка беречь секреты — полезная привычка. Для серьёзных ключей (платёжные системы, базы с реальными клиентами) лучше использовать второй способ — заполнить `.env` руками в VS Code, чтобы токен вообще никуда не уходил».

### 11.6. Публикация Mini App на Vercel
<!-- LIBRARIAN-PRINCIPLE: mini-app-vercel-deploy -->

Без публикации — цепочка рвётся: Telegram не может показать приложение, пока файлы не окажутся в интернете. Mini App — это веб-страница. Когда пользователь нажимает кнопку в боте, Telegram загружает страницу по адресу `https://...`. Если адреса нет — нечего загружать.

**Цепочка:**
```
Файлы на компьютере
    ↓ публикуем в интернет
Ссылка https://my-app.vercel.app
    ↓ даём BotFather
Telegram загружает страницу
    ↓
Пользователь видит приложение
```

**Промпт:** «Помоги опубликовать папку `tg-app/` через Vercel. Мне нужна HTTPS-ссылка для Mini App. Важно: публиковать именно `tg-app/`, а не весь проект. Веди меня пошагово как новичка».

**Root Directory = `tg-app/`** — если Vercel спросит про Root Directory, указать `tg-app/` (папку, где лежит `index.html`).

**Telegram принимает только HTTPS-ссылки** (с замочком) — Vercel даёт их автоматически.

**Проверка перед следующим шагом:** открыть полученную ссылку в браузере телефона. Видите своё приложение — отлично. Видите пустую страницу или ошибку — деплой не удался, описать проблему Claude.

### 11.7. Подключение Mini App к боту — 2 способа
<!-- LIBRARIAN-PRINCIPLE: bot-connection-2-ways -->

После публикации нужно подключить Mini App к боту. Есть **два разных способа** — они делают **разные вещи**.

#### Способ 1. Menu Button — кнопка внутри бота

| Шаг | Действие |
|---|---|
| 1 | Вернуться к `@BotFather` |
| 2 | `/mybots` → выбрать бота |
| 3 | `Bot Settings` → `Menu Button` → `Configure` |
| 4 | Вставить ссылку Vercel |
| 5 | Текст кнопки: например «Открыть каталог» |

**Что даёт:** внизу чата с ботом, слева от поля ввода, появляется кнопка. Человек **уже зашёл в бот**, видит кнопку, нажимает — открывается приложение. Это вход «изнутри» бота.

#### Способ 2. Mini App — прямая ссылка

| Шаг | Действие |
|---|---|
| 1 | `/mybots` → бот → `Bot Settings` → `Mini Apps` |
| 2 | `Enable Mini Apps` → вставить ту же ссылку |

**Что даёт:** появляется **прямая ссылка** вида `t.me/ваш_бот/app`. Эту ссылку можно отправить кому угодно — в чат, в соцсети, на визитку. Человек нажимает — и сразу попадает в приложение на полный экран.

**Важный нюанс:** даже через прямую ссылку Telegram передаёт приложению данные пользователя (имя и ID). Приложение «видит», кто его открыл, и может показывать персональный контент («Привет, Анна!»). Не нужно регистрироваться или логиниться — Telegram делает это автоматически.

**Но есть разница:** если человек зашёл через прямую ссылку, **не нажав `/start`** в боте — бот не сможет потом отправить ему сообщение. Ни подтверждение записи, ни напоминание, ни акцию. Для этого человек должен хотя бы раз нажать `/start` — тогда бот «знает» о нём и может писать первым.

**Метафора:** прямая ссылка — адрес магазина на визитке (заходят, смотрят витрину, могут что-то выбрать; но если не оставили номер на входе — вы не позвоните с напоминанием). Menu Button — вывеска внутри ТЦ (чтобы её увидеть, нужно уже быть «внутри» — нажать `/start` — зато у вас сразу есть контакт).

**Нужны оба?** Да, настройте оба:
- **Menu Button** — для тех, кто уже в боте (у вас есть их контакт)
- **Прямая ссылка** — для привлечения новых людей (быстрый вход без обязательств)

### 11.8. Лайфхак — экран-оффер для сбора контактов
<!-- LIBRARIAN-PRINCIPLE: offer-screen-lead-capture -->

Если человек зашёл через `t.me/бот/app` — он видит приложение, но бот о нём не знает. Решение: добавить ненавязчивый экран-оффер при первом открытии (показывать один раз — запоминать в `localStorage`).

**Структура экрана-оффера:**

1. **Эмодзи** — крупная, по центру (`🎁`)
2. **Заголовок** — главная ценность (`Скидка 15% на первую запись`)
3. **Подзаголовок** — что нужно сделать (`Подпишитесь на бота — получите промокод в личное сообщение`)
4. **Буллеты** — выгоды подписки (Напомним о записи за день / Первыми узнаёте о свободных окошках / Эксклюзивные акции для подписчиков)
5. **Кнопка** — яркая, на всю ширину, текст `Получить скидку 15%`, ссылка `https://t.me/{username_бота}?start=from_app`
6. **Под кнопкой** — «Пропустить»

**Что произойдёт:** человек нажимает кнопку → открывается бот → срабатывает `/start` → бот теперь «знает» этого человека и может писать первым. Параметр `?start=from_app` — это метка, она говорит боту: «этот человек пришёл из приложения».

**Принцип:** «Человек нажимает не на "подписаться", а на "получить ценность". Подписка — побочный эффект, а не цель. Человек хочет скидку — и попутно становится подписчиком».

**Конверсия:** оффер с ценностью даёт конверсию в **3-5 раз выше**, чем просто «подпишитесь».

**Шаблон под разные ниши:**

| Ниша | Оффер (заголовок) | CTA (кнопка) |
|---|---|---|
| Бьюти-мастер | Скидка 15% на первую запись | Получить скидку 15% |
| Ресторан | Десерт в подарок к первому заказу | Получить подарок |
| Фотограф | Бесплатная мини-консультация | Записаться бесплатно |
| Репетитор | Первый урок — 0 ₽ | Забрать бесплатный урок |

### 11.9. Три типа кнопок в Telegram-боте
<!-- LIBRARIAN-PRINCIPLE: telegram-bot-button-types -->

Знать про возможности — не обязательно использовать сразу.

| Тип | Что делает | Когда использовать |
|---|---|---|
| **Menu Button** | Постоянная кнопка слева внизу, рядом с полем ввода. Один клик — открыто приложение | У бота одна главная функция (каталог, калькулятор, портфолио) |
| **Inline-кнопки** | Кнопки под сообщением бота. Бот отправляет сообщение, под ним кнопки с вариантами («Маникюр / Брови / Макияж»). Логика заложена при формировании идеи и анализе конкурентов | Нужно предложить выбор. Квиз, навигация по разделам, подтверждение действия |
| **Reply Keyboard** | Большие кнопки вместо клавиатуры. Появляются крупные кнопки `[Каталог] [О нас] [Контакты]`. Пользователь нажимает — бот получает текст | У бота несколько разделов, нужна постоянная навигация |

Все три типа можно **комбинировать** в одном боте: Menu Button открывает Mini App + Inline-кнопки помогают с выбором внутри чата + Reply Keyboard для быстрой навигации.

### 11.10. Оформление бота — один запрос
<!-- LIBRARIAN-PRINCIPLE: bot-setup-one-request -->

Когда кто-то находит вашего бота в Telegram — он видит описание. Это первое впечатление, оно решает: нажмёт человек Start или пройдёт мимо.

**Плохое описание:** «Привет! Я бот.» Человек не понимает, зачем он здесь.

**Хорошее описание отвечает на три вопроса:**
- **Что** — что делает этот бот
- **Зачем** — какую пользу получит человек
- **Куда нажать** — чёткое следующее действие

**Лайфхак — один запрос и бот полностью оформлен:** Claude Code знает токен (он в `.env`). Через токен он может настроить бота напрямую — без ручной работы в BotFather:

```
Настрой моего Telegram-бота полностью.
Токен бота в файле .env.

Бот открывает Mini App — [опиши в 1-2 предложениях что делает приложение].

Через Telegram Bot API:
1. Установи описание бота (setMyDescription) —
   2-3 предложения: что, зачем, куда нажать
2. Установи короткое описание (setMyShortDescription) — одно предложение
3. Установи команды (setMyCommands) — /start, /help и как связаться с автором

В приложении (Mini App):
4. Экран приветствия (онбординг):
   - Обращение по имени (из Telegram)
   - Что здесь можно сделать (2-3 пункта)
   - Кнопка «Начать»
5. На каждом экране — понятные кнопки
6. Кнопку «Поделиться с другом» — отправляет ссылку на бота

Русский язык, дружелюбный тон, без пафоса.
```

Claude сделает всё сам: зайдёт в Telegram через токен, настроит описание и команды бота, доработает приложение. Вам ничего не нужно копировать и настраивать вручную.

### 12.2. SaaS-архитектура (один сервер — много мастеров)
<!-- LIBRARIAN-PRINCIPLE: saas-multi-tenant -->

**Принцип SaaS:** один сервер, одно приложение, но **каждый мастер видит только своё** — как Google Docs (одна платформа, у каждого свой документ).

**Метафоры:**

- Бэкенд — склад и кладовщик (то, чего клиент не видит)
- API — посредник-кладовщик (фронтенд никогда не работает с базой напрямую — только через API)

**Принцип API-посредника:** API проверяет, кто это, берёт нужное из базы, возвращает обратно. Фронтенд не видит склад — видит только результат.

**Результат после Урока 12:** «Любой мастер даёт токен своего бота и через 2 минуты получает личный каталог. Именно так работает SaaS. Именно это и было задумано с самого начала».

### 12.9. Структура промпта для точечных правок (Урок 12)

## Дополнение к Уроку 6 — Agent Skills для Claude Code

> Бонус-видео автора. Тема не связана напрямую с Уроком 6 «Исследование ниши» — это новая фича Anthropic, которую Дмитрий показывает отдельным дополнением. Источник: `vibe-coding-6-dop-skills.pdf` (транскрипт видео).

### 6-Д.1. Что такое скиллы (Agent Skills)
<!-- LIBRARIAN-PRINCIPLE: claude-code-skills -->

**Скилл** — это расширение функционала Claude Code (как плагин для браузера или приложение для смартфона). Устанавливается в проект или глобально, расширяет возможности Claude практически до бесконечности.

**Зачем нужны:**
- Получать внешние инструкции
- Создавать дизайны, красивые таблицы, PDF-ки, презентации
- Решать любые специализированные задачи
- Проводить глубокие исследования и интервью
- В разы упростить работу, которую сейчас делаете вручную

**Метафора:** скилл — как покупка пустого смартфона и наполнение его приложениями. Сам по себе работает, но с приложениями — кратно мощнее.

### 6-Д.2. Где брать скиллы — Marketplace
<!-- LIBRARIAN-PRINCIPLE: skills-marketplace -->

**Agents Skills Marketplace** — основной сайт каталога скиллов. На английском (можно использовать переводчик Google Chrome).

**Что видно по каждому скиллу:**
- Кто разработчик (например, разработчики Anthropic делают свои скиллы)
- Количество скачиваний и оценок (например — 337 тысяч человек оценили скилл от OpenGlow → значит, в целом неплохо)
- Описание (стоит прочитать, что за скилл)
- Ссылка на репозиторий — её копируем для установки

**Принцип отбора:**
- Не устанавливайте сотню скиллов — это распыление
- 2-4 нужных под текущие задачи — оптимально
- Можно обмениваться найденными скиллами в чате обучения / сообществе

### 6-Д.3. Три ключевых скилла (рекомендация автора)
<!-- LIBRARIAN-PRINCIPLE: top-3-skills -->

| Скилл | Что делает | Когда вызывать |
|---|---|---|
| **Discovery Interview** | Углублённый процесс интервьюирования. Превращает размытую идею (которая есть в голове) в детальную спецификацию через 7 фаз вопросов. Подходит и для технических, и для нетехнических пользователей. Уходит ~10–30 минут на проведение интервью | Всегда, когда приходит новая идея — сайт, функция, платёжный модуль, юр.документация. «Любую идею всегда разрабатывайте с помощью этого скилла» |
| **Full-Stack Developer** | Разработчик «полного года», разбирается во фронтенде, бэкенде. Использует готовую спецификацию для написания кода | Когда есть готовая спецификация и нужно её реализовать |
| **Front-End Design** | Даёт навыки фронт-энд разработчика: красивые элементы, анимации, движения. Без него «всё выдаётся одинаково — фиолетовая расцветка, одинаковый шрифт» | Когда работаете с дизайном сайта/приложения, нужны крутые анимации и интерфейс |

**Дополнительно упомянутый скилл (бонус):**
- **CodeReview** — рекомендуемый автором, очень крутой скилл для ревью кода

**Рабочий расклад:** Discovery Interview (всегда первым на новую идею) → спецификация → Full-Stack Developer (пишет код по спеке) → Front-End Design (доводит дизайн).

### 6-Д.4. Два способа установки скилла
<!-- LIBRARIAN-PRINCIPLE: skills-install-methods -->

**Способ 1 — через просьбу к Claude (простой):**

1. На Marketplace копируем ссылку на репозиторий скилла
2. В Claude Code говорим:
   ```
   Установи скилл из этого репозитория: {ссылка}
   ```
3. Claude разрешения спросит — отвечаете «да»
4. Claude сам клонирует и устанавливает скилл

**Способ 2 — через терминал (более прозрачный):**

1. В VS Code открываем терминал (вкладка «Terminal» сверху → Create Terminal)
2. На Marketplace на странице скилла видим ссылку `install global` — копируем
3. Вставляем команду в терминал → нажимаем Enter
4. Происходит клонирование репозитория → подтягиваются все скиллы разработчика
5. Терминал показывает: «Select skills for install» — список скиллов разработчика
6. Стрелочками листаем, **пробелом** выделяем нужные (например — `CodeReview` + `FullStackDeveloper`)
7. **Enter** — переходим к следующему шагу выбора места установки

### 6-Д.5. Куда устанавливать — Global vs Текущий проект
<!-- LIBRARIAN-PRINCIPLE: skills-global-vs-project -->

После выбора скиллов терминал спрашивает: куда устанавливать?

| Вариант | Где лежит | Когда использовать |
|---|---|---|
| **Global** | В общей папке, доступной всем проектам | Поставил один раз — работает везде, не нужно переустанавливать в каждый проект |
| **Текущий проект** | Внутри папки конкретного проекта | Скилл лежит ближе к проекту, проект «не путается». Автор предпочитает этот вариант |

**Стратегия автора:** Discovery Interview можно установить **глобально** (нужен в каждом новом проекте). Узкоспециализированные (FullStackDeveloper, FrontEndDesign) — **в текущий проект**, чтобы они «лежали ближе и проект не путался».

**После установки в проект** появляется:
- Папка `Claude/Skills/` со всеми установленными скиллами
- JSON-файл (типа `skills.lock`) — список того, что установлено

### 6-Д.6. Как использовать скилл в работе
<!-- LIBRARIAN-PRINCIPLE: skills-usage -->

**Вариант 1 — явно указать в запросе:**
```
Я сейчас буду разрабатывать сайт-визитку с описанием меня как эксперта.
Помоги описать спецификацию для разработки этого сайта,
чтобы в будущем с помощью вайбкодинга я мог его разработать.
Используй скилл Discovery Interview.
```

**Вариант 2 — Claude сам подтянет по контексту:**
Если скилл установлен, Claude активирует его автоматически, когда задача подходит.

**Вариант 3 — посмотреть установленные скиллы перед запросом:**
В чате Claude Code пишете команду или запрос типа «покажи скиллы» — выходит список установленных. Копируете нужные названия в свой запрос «чтобы не забыть».

### 6-Д.7. Демо из урока (как это работает на практике)
<!-- LIBRARIAN-PRINCIPLE: skills-demo-flow -->

Автор делает живое демо — сайт-визитку для себя:

1. **Запрос с скиллом:** «Помоги описать спецификацию для разработки сайта-визитки. Используй Discovery Interview»
2. **Discovery Interview задаёт вопросы:**
   - Кто ты? (наставник, предприниматель)
   - Кто целевой посетитель? (откуда придёт — рекомендации, ТГ-канал, поиск, после выступления)
   - Что должно считываться за 3 секунды на сайте?
   - Что должен сделать посетитель в итоге? (написать в Telegram / оставить заявку / подписаться)
   - У тебя сейчас сколько продуктов? (Ай-эксперт, нейроклуб, профессия вайб-кодер, разработка на заказ)
   - Это всё на одном сайте или сайт только про тебя как личность? («только про меня как личность, продукты — на отдельных сайтах»)
3. **Claude формирует спецификацию** на основе ответов
4. **Запуск разработки:** «Теперь начинай полную разработку по всем шагам, по фазам, которые ты написал, при необходимости используй скиллы FullStackDeveloper и FrontEndDesign»
5. **Claude сам разрабатывает по плану** — вызывает скиллы по необходимости. Создаёт сайт-визитку с 3D-нейронной сетью, частицами, ссылкой на TG, права защищены — всё за минуты

### 6-Д.8. Принципы работы со скиллами
<!-- LIBRARIAN-PRINCIPLE: skills-principles -->

- ✅ **Без скиллов всё работает.** Скиллы — это ускоритель, а не обязательное условие
- ✅ **Discovery Interview вызывайте на любую новую идею** — даже на маленькую функцию (платёжный модуль, юр.оферта)
- ✅ **Не распыляйтесь** — установите 2-4 нужных, не сотню
- ✅ **Обменивайтесь** найденными скиллами в сообществе — ускоряет всех
- ✅ **Read description перед установкой** — на странице скилла читайте, что он делает
- ❌ **Не путать с самописными скиллами** — на курсе используются готовые из Marketplace; своё писать — отдельная тема
- ❌ **Не игнорируйте Front-End Design** — без него сайты «фиолетовые и однообразные»

## Антипримеры / типичные ошибки

❌ **Сразу открыть ИИ и написать «сделай мне приложение»** — без идеи, без плана, без контекста
❌ **Пытаться сделать всё за один промпт** — вместо разбивки на 3-5 шагов
❌ **Не делать commit/push** — потеряете работу при первой же поломке
❌ **Использовать кириллицу или пробелы в названиях папок** — Claude не сможет работать
❌ **Бросаться сразу в Lovable или Claude Code** — лимиты сгорают, деньги уходят. Сначала бесплатный AI Studio для плана
❌ **Работать в ChatGPT и копировать код в файлы вручную** — хаос: 20 вкладок, 15 файлов на рабочем столе
❌ **Не создавать `CLAUDE.md`** — каждый новый чат начинается с объяснения «у нас тёмный фон, акцент оранжевый…» с нуля
❌ **Раздел «Чего НЕ будет» в `project.md` забыть** — будете бесконечно добавлять «ещё одну фичу»
❌ **Не делать «Раскритикуй» перед публикацией** — обнаружите проблемы после клиентов, а не до
❌ **Пропускать «Ты уверен?»** — Claude иногда уверенно говорит неточные вещи
❌ **Стартовать без насмотренности** — не посмотрев 15-20 лендингов в нише
❌ **Делать ТЗ без research.md и критики экспертов** — потом переделывать всё с нуля
❌ **Использовать внешние библиотеки на первом проекте** — теряется контроль и адаптивность
❌ **Забыть проверить адаптивность на мобильных** до деплоя
❌ **Менять код в Claude Code, но не пушить в GitHub** — Vercel ничего не обновит
❌ **Устанавливать сотню Agent Skills** — распыление, нужны 2-4 ключевых
❌ **Игнорировать Front-End Design скилл** — все сайты выходят одинаковыми, фиолетовыми
❌ **Полагаться на соцсети без сайта** — один апдейт алгоритма и вы невидимы
❌ **Делать Mini App там, где достаточно ссылки на сайт** — для простой визитки Mini App избыточен
❌ **Усложнять интерфейс Mini App** — все функции подряд = пользователь теряется
❌ **Хранить токен бота в коде вместо `.env`** — компрометация бота, потеря контроля
❌ **Отправлять токены в публичные чаты, GitHub-комментарии, на форумы** — даже после удаления история сохраняет всё
❌ **Писать в фронтенде запросы прямо в базу** — только через API-посредник
❌ **Делать одну витрину для всех мастеров вместо SaaS-модели** — каждый мастер должен видеть только своё (по ID бота)
❌ **Без `BACKEND-PLAN.md` сразу прыгать в код бэкенда** — каждый шаг с нуля
❌ **Деплоить бота на Vercel вместо VPS** — Vercel не для 24/7-задач (не отправит уведомление в нужное время)
❌ **Большой промпт «переделай всё»** — каша. 3 коротких на 3 правки лучше
❌ **Не делать `DEPLOYMENT.md`** — через полгода не вспомните как деплоили

| Agent Skills marketplace на английском | Часть скиллов узкоспециализирована под западные кейсы | Отбираем те, что универсальны (Discovery Interview, Full-Stack Developer, Code Review). Создание собственных скиллов под наш стек — отдельный проект |

**Юридические корректировки:**
- Если упаковываем услуги вайбкодинга для клиентов через Ирину (самозанятая) — называем «изготовление дизайна цифрового продукта» / «разработка веб-страницы» / «изготовление макета», а не «аудит сайта», «консультация по продукту», «стратегия запуска» (запретные для самозанятой формулировки).
- Через Инну (ИП, ОКВЭД 70.22) можно использовать слова «аудит», «сопровождение», «экспертиза», «архитектура».

**Контекст 8 ниш (инфобизнес, финансы, бьюти, коучи, фитнес, торговля, риэлторы, санатории/отели):**
Категории продуктов вайбкодера ложатся почти на все ниши:
- **Инфобизнес/коучи/наставники** — лендинги, квизы, ИИ-помощники, автоматизации рассылок
- **Финансы/брокеры** — калькуляторы, дашборды, ИИ-консультанты
