# Шаблонные промпты: Лендинги, калькуляторы, квизы, сайты

**Источник:** Урок «CRAFT + три файла + цикл вайбкодера» + наш опыт ЮБ/Нейро Бабки СССР.
**Адаптировано под наш стек:** один самодостаточный HTML-файл, стили внутри `<style>`, Montserrat + Playfair Display, брендбук Инны для внутренних продуктов, деплой через Beget.

Применять в связке с [product-creation.md](../playbooks/product-creation.md) (раздел 3.4) и [craft-formula.md](./craft-formula.md).

## 1. Стартовый one-shot — лендинг из brief.md + research.md

**Когда:** Фаза 4 (Build), есть `brief.md` и `research.md`.

**Перед применением:** Директор задал вопрос «Beget или Vercel?». Для лендинга почти всегда — Beget сразу.

```
Ты — frontend-разработчик и UX-специалист. 10+ лет опыта с HTML, CSS, адаптивом.

Прочитай файлы в проекте:
- brief.md — чертёж страницы (экраны, секции, CTA)
- research.md — что работает у конкурентов
- knowledge/standards/design-system.md — наш брендбук

Создай лендинг clients/<project>/mvp/<page>.html.

Технические требования:
- Один самодостаточный HTML-файл, все стили внутри <style>
- Никаких внешних библиотек, CDN, фреймворков
- Google Fonts (Montserrat + Playfair Display) через preconnect
- Адаптив: clamp() для размеров текста, flexbox + grid, @media
- Семантический HTML: header, main, section, footer

Структура лендинга (адаптировать под brief.md):
1. Hero — заголовок с УТП + подзаголовок + главный CTA
2. Блок проблемы — какую боль решаем
3. 3 преимущества / фичи с иконками или SVG
4. Отзывы / кейсы (если есть в brief.md)
5. Тарифы или ценность
6. FAQ (если нужно)
7. Финальный CTA

Дизайн:
- Для внутренних продуктов Инны — брендбук из design-system.md:
  Фон #ffffff (или #d3bfb1 для тёплых секций)
  Текст #4c4651
  Главный акцент #d4af37 (золотой)
  Hover #c99700 (горчичный)
  Поддержка #4b7850 (зелёный) или #103206 (deep)
  Заголовки Playfair Display, текст Montserrat

- Для клиентских лендингов — палитра клиента из brief.md

- Адаптивность обязательна: проверить на 375px, 768px, 1440px
- Высота кнопок минимум 44px
- Контраст текст/фон достаточный для чтения (WCAG AA)

Контент:
- Простой язык, без канцелярита
- «Вы» с посетителем
- Заголовки — конкретные, не абстрактные («Три шага к результату» > «Наш подход»)
- Кнопки CTA — глагол + выгода («Записаться на консультацию»)
- Реальный контент из brief.md; если данных нет — placeholder «TBD» и пометка в PLAN.md

НЕ делай:
- Не используй Bootstrap, Tailwind, React, Vue
- Не подключай внешние иконочные шрифты (Font Awesome) — используй inline SVG или emoji
- Не придумывай факты о клиенте

Все комментарии в коде — на русском.
Задавай вопросы если в brief.md не хватает данных.
```

## 2. Стартовый one-shot — калькулятор

```
Ты — frontend-разработчик, специалист по интерактивным формам.

Прочитай brief.md — там описаны параметры калькулятора и формула расчёта.

Создай calculator.html — калькулятор стоимости [услуги/работы/проекта].

Технические требования:
- Один HTML-файл, стили внутри <style>, без фреймворков
- Адаптив: мобильный, планшет, десктоп

Структура:
1. Заголовок + объяснение в 1-2 предложения
2. Параметры ввода:
   - [из brief.md: что за параметры, какого типа — слайдеры, select, input]
   - Подписи к каждому параметру
   - Подсказки где нужно
3. Итоговая сумма — крупно, обновляется мгновенно при изменении
4. Расшифровка расчёта (по пунктам: какой параметр + вклад в сумму)
5. Кнопка «Оставить заявку» под результатом
6. Форма заявки (имя + контакт) — опционально, если в brief.md

Формула расчёта — из brief.md. Если не описана — задай вопрос.

Дизайн:
- Брендбук Инны (или палитра клиента из brief.md)
- Поля ввода крупные, на мобильном удобные для пальца
- Результат выделен цветом (золотой для внутренних, акцент клиента для клиентских)
- Анимация обновления результата (плавная смена числа)

НЕ делай:
- Не сохраняй данные в localStorage без разрешения в brief.md
- Не отправляй данные на сервер без указания backend в brief.md
- Не придумывай параметры которых нет в brief.md

Задавай вопросы если формула расчёта неоднозначна.
```

## 3. Стартовый one-shot — квиз

```
Ты — frontend-разработчик и UX-специалист по интерактивным квизам.

Прочитай brief.md — там перечислены вопросы, варианты ответов, ветвления и результаты.

Создай quiz.html — интерактивный квиз на [тема из brief.md].

Технические требования:
- Один HTML-файл, стили внутри <style>
- Один вопрос на экране, навигация «Далее / Назад»
- Прогресс-бар сверху (% пройдено)
- Сохранение прогресса в sessionStorage (чтобы обновление страницы не сбрасывало)

Структура:
1. Стартовый экран — заголовок + «Начать»
2. Экран вопроса:
   - Номер вопроса и прогресс
   - Текст вопроса
   - Варианты ответов (2-4 кнопки)
   - Навигация «Назад»
3. Экран результата:
   - Персонализированный результат на основе ответов
   - Рекомендация / следующий шаг
   - CTA-кнопка (из brief.md)
   - Кнопка «Пройти ещё раз»

Логика:
- Вопросы из brief.md, в том же порядке
- Ветвления (если есть) — из brief.md
- Результаты — по правилам из brief.md. Не придумывай новые.

Дизайн:
- Брендбук Инны / палитра клиента
- Анимация появления каждого вопроса (fade + slide)
- Активный вариант ответа выделен
- На результате — confetti или другой эмоциональный акцент (жёлтый #ffc403 — эпизодически)
- Адаптив: всё работает на мобильном

НЕ делай:
- Не добавляй вопросы вне brief.md
- Не меняй формулировку результатов
- Не делай backend-интеграцию без указания в brief.md
```

## 4. Стартовый one-shot — многостраничный сайт

```
Ты — frontend-разработчик, специалист по многостраничным сайтам.

Прочитай brief.md — там перечислены страницы, структура каждой, навигация.

Создай сайт для [проекта] из N страниц:
1. index.html — главная (представление + УТП + CTA)
2. about.html — о нас / о мастере
3. services.html — услуги с ценами
4. contacts.html — форма + карта
5. [другие из brief.md]

Технические требования:
- Каждая страница — отдельный HTML-файл
- Общая навигация на всех страницах (identical header + footer)
- Стили — либо одним общим <style> блоком, повторяющимся в каждом файле,
  либо одним внешним css/style.css (ОДНО исключение из правила «всё inline»)
- Без фреймворков
- Google Fonts: Montserrat + Playfair Display
- Адаптив обязателен

Дизайн:
- Брендбук Инны (если внутренний) или палитра клиента
- Единый стиль всех страниц
- Активная страница подсвечена в меню

Контент:
- Только из brief.md, не придумывай
- Тексты простые, без канцелярита
```

## 5. Точечные правки

Как в templates-mini-app — формула «Что / Где / Что не трогать».

```
Сделай заголовок Hero крупнее и добавь больше воздуха снизу.
Остальные секции не трогай.
```

```
На странице тарифов выделить средний тариф (рекомендуемый)
рамкой и плашкой «Выбор большинства».
Не меняй цены и текст, только визуал.
```

```
Форма обратной связи слишком узкая на десктопе.
Сделай её шире, но не больше 600px.
Остальное оставь как есть.
```

## 6. Итерации после «раскритикуй»

После того, как критика через [critique-techniques.md](./critique-techniques.md) дала список проблем:

```
Список проблем из критики:
1. [проблема 1 — что и где]
2. [проблема 2]
3. [проблема 3]

Исправь проблему №1 точечно. Только её. Остальные не трогай.
Покажи что изменил и почему.
```

После исправления — проверка и следующая итерация. **Не исправляй все проблемы одним промптом.**

## 7. Адаптивность — проверка на мобильных

```
Проверь страницу на разных экранах: 320px, 375px, 390px, 430px, 768px, 1024px, 1440px.

Для каждого размера — что ломается? Что обрезается? Что налезает?
Есть ли горизонтальная прокрутка?
Кнопки удобные для пальца (минимум 44px)?
Текст читаемый (минимум 14px, контраст AA)?

Исправь все найденные проблемы. Объясни что изменил.
```

## 8. Ускорение / оптимизация

```
Посмотри на страницу как инженер производительности.
Что замедляет загрузку? Что тяжёлого?
- Большие картинки — оптимизируй размер и формат (webp если возможно)
- Лишние шрифты — убери неиспользуемые начертания
- Неоптимизированный CSS — убери дубли
- Блокирующие скрипты — defer или async

Цель: страница загружается за 1 секунду на 3G.
Не ломай дизайн.
```
