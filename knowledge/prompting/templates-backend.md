## 0. Фронтенд vs бэкенд — метафора, которую напомним Инне

Перед тем как писать код бэкенда, Директор напоминает:

- **Фронтенд** — то, что видно: экраны, кнопки, анимации. Всё, что открывается на телефоне.
- **Бэкенд** — то, что не видно: база данных, API, фоновые процессы. Как кухня ресторана — без неё зал пустой, даже если красиво накрыт.
- **База данных (Supabase)** — склад. Там хранятся все мастера, услуги, записи, фотографии.
- **API** — кладовщик. Фронтенд говорит «дай список услуг» — API идёт в базу, берёт, возвращает. Фронтенд никогда не работает с базой напрямую.

Бэкенд добавляется, когда:
- Данные должны сохраняться между сессиями (запись у мастера, корзина, прогресс квиза с историей)
- Данные должны быть одинаковыми на всех устройствах
- Один сервер — много клиентов (мульти-tenancy)
- Нужны уведомления, напоминания, фоновые задачи

Без этого — чистый фронт на `localStorage` достаточен.

## 1. BACKEND-PLAN.md — чертёж бэкенда (Шаг 0 из Урока 12)

**Когда:** Фаза 3 (Plan) для продукта с бэкендом. Обязательно ДО написания кода.

```
Ты — бэкенд-архитектор с опытом SaaS и мульти-tenant систем.

Прочитай файлы в проекте:
- idea.md — формулировка продукта
- research.md — решения конкурентов
- brief.md — чертёж фронтенда

Ключевая задача бизнеса:
[Вставить из idea.md — что должно произойти у пользователей.
Пример: «Любой мастер может запустить свою мастер-группу через платформу,
получить витрину и кабинет, собирать участников. Мой сервис видит всех мастеров
и всех участников в одной админке»]

Задавай мне уточняющие вопросы — не больше 7 за раз, собери их в один заход:
- Сколько пользователей ожидается (десятки, сотни, тысячи)?
- Какие роли (админ, мастер, участник, гость)?
- Кто что видит и что редактирует?
- Нужна ли мульти-tenancy (разделение данных между клиентами)?
- Какие данные критичны (не терять), какие можно перегенерить?
- Нужны ли realtime-обновления (новые заявки появляются сразу)?
- Нужны ли фоновые задачи (напоминания, уведомления)?

После получения ответов — составь BACKEND-PLAN.md со структурой:

# Backend Plan: <название>

## Ключевая задача
...

## Сущности данных
Перечисли таблицы: название, поля, типы, связи.
Отдельно — индексы и уникальные ограничения.

## Роли и доступ
Таблица: роль × таблица × операции (R/W/U/D).

## RLS политики (Supabase)
Для каждой таблицы — какие политики public_read, insert, update, delete.
Кто что может сделать через anon-ключ, что — только через service_role.

## API / RPC функции
Если нужны server-side функции (для RLS-обхода, агрегаций, транзакций).

## Realtime
Какие таблицы подключаем к supabase_realtime publication.

## Storage
Нужны ли бакеты (фото, документы). Какие политики доступа.

## Схема мульти-tenancy (если применимо)
Как разделяем данные между клиентами: поле tenant_id? отдельная схема?

## Чего НЕ делаем в MVP
Важный раздел — что отложено.

## Границы безопасности
Что кладём в публичный ключ, что в секрет. Где проверяем токены.

Только то, что реально нужно этому приложению. Без over-engineering.
```

## 2. Создать Supabase-проект (ручной шаг Инны)

Чеклист, который Директор показывает Инне:

```
1. supabase.com → Sign in (GitHub)
2. New project → название (обычно my-architecture или <проект>)
3. Database password — СЛОЖНЫЙ, сохранить в 1Password/заметках
4. Region — eu-west-2 (Лондон) или ближайший к ЦА
5. Pricing — Free tier, хватит до 500 MB
6. Ждать 2-3 минуты пока провизионится
7. Project Settings → API:
   - Project URL → скопировать в .env как SUPABASE_URL
   - Publishable key (раньше anon) → скопировать в .env как SUPABASE_PUBLISHABLE_KEY
   - Secret key (раньше service_role) → скопировать в .env как SUPABASE_SECRET_KEY
8. Project Settings → Database → Connection string → URI:
   - ВАЖНО: взять строку из раздела "Transaction pooler"
     (прямое подключение db.*.supabase.co теперь только IPv6!)
   - Формат: postgresql://postgres.XXX:ПАРОЛЬ@aws-X-XX.pooler.supabase.com:6543/postgres
   - Вставить в .env как SUPABASE_DB_URL
9. Cmd+S в VS Code
```

## 3. Создать SQL-миграции (Шаг 1 из Урока 12)

**Когда:** после `BACKEND-PLAN.md`, проект Supabase создан, ключи в `.env`.

```
Ты — бэкенд-разработчик Postgres / Supabase, 10+ лет опыта.

Прочитай BACKEND-PLAN.md.

Создай SQL-миграции в папке clients/<project>/supabase/migrations/:

1. 001_schema.sql — создание таблиц
   - create table if not exists (идемпотентность)
   - bigserial для id (кроме тех случаев где text PK = slug)
   - timestamptz default now() для created_at
   - индексы на foreign keys и фильтруемые поля
   - create extension if not exists pgcrypto; если нужны SHA-256 хэши

2. 002_policies.sql — RLS
   - alter table <name> enable row level security; для каждой таблицы
   - политики SELECT, INSERT, UPDATE, DELETE по BACKEND-PLAN
   - drop policy if exists перед create policy (идемпотентность)
   - alter publication supabase_realtime add table <name>; для realtime

3. 003_seed_<dataset>.sql — тестовые данные
   - on conflict do nothing / do update для идемпотентности
   - Реальные данные, не «Мастер Анна» и «Услуга 1»
   - Если секретные токены — хранить как SHA-256 хэш (encode(digest(...,'sha256'),'hex'))

Применяй миграции через наш скрипт:
  node scripts/supabase-migrate.mjs --client <project>

Он читает .env, подключается через Transaction pooler,
ведёт таблицу schema_migrations (чтобы не применять дважды),
в конце показывает счётчики по таблицам.

Не применяй руками через Supabase SQL Editor — только через скрипт.

Объясняй каждый шаг простым языком через метафоры, я новичок в Postgres.
```

**Важно:** `supabase-migrate.mjs` поддерживает `--client <name>` с 2026-04-14. Если этого аргумента нет — обновить скрипт.

## 4. API / слой работы с данными (Шаг 2 из Урока 12)

**Подход №1: прямые вызовы через supabase-js (простой MVP)**

```
Ты — fullstack-разработчик, специалист по Supabase JS SDK.

Прочитай BACKEND-PLAN.md и текущий код фронтенда
(clients/<project>/tg-app/ или clients/<project>/mvp/).

Подключи Supabase напрямую из фронтенда:

1. Добавь CDN-скрипт в index.html:
   <script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2/dist/umd/supabase.min.js"></script>

2. Создай js/supabase-config.js с SUPABASE_URL и SUPABASE_PUBLISHABLE_KEY
   (эти ключи БЕЗОПАСНО класть в код — они защищены RLS)

3. Перепиши js/data.js — вместо констант сделай async-функции:
   - loadMaster(id) → select from masters
   - loadCategories() → select from categories
   - loadServices(categoryId) → select from services
   - createBooking(data) → insert into bookings
   - ... по BACKEND-PLAN

4. Обнови js/app.js:
   - loading state при запросах (spinner)
   - обработка ошибок (показ пользователю)
   - замена синхронных циклов на await

5. Для realtime-обновлений используй supabase.channel():
   const ch = supabase.channel('<project>-<entity>')
     .on('postgres_changes', {
       event: 'INSERT', schema: 'public', table: '<name>',
       filter: 'master_id=eq.' + masterId
     }, (payload) => { /* обработка */ })
     .subscribe();

Не ломай существующие экраны. Добавляй постепенно, после каждого шага — проверка что работает.
Объясняй каждый шаг простым языком, я новичок.
```

**Подход №2: API-слой через Supabase Edge Functions (если нужна серверная логика)**

```
Создай Edge Function в supabase/functions/<name>/ для операций,
которые нельзя открыть анонимному пользователю:
- проверка токена мастера (SHA-256 матч)
- агрегации (статистика, отчёты)
- операции с service_role (обход RLS)

Используй Deno runtime, стандартные Supabase helpers.
Тестируй через supabase functions serve <name> локально.
Деплой: supabase functions deploy <name>.
```

## 5. Мультибот / мульти-tenancy (Шаг 3 из Урока 12)

**Когда:** если продукт — платформа, где каждый клиент получает свой бот/витрину.

```
Ты — бэкенд-разработчик с опытом мульти-tenant систем.

Прочитай BACKEND-PLAN.md и текущий webhook-обработчик.

Задача: любой мастер может подключить своего бота через BotFather
и получить личный каталог. Платформа по ID бота в webhook определяет
чей каталог показывать.

Реализация:

1. Таблица bots:
   - id bigserial pk
   - token_hash text unique (sha256 от настоящего токена)
   - master_id text fk → masters.id
   - username text
   - created_at timestamptz

2. Webhook-обработчик:
   - Принимает update от Telegram
   - Из update.message.from или update.callback_query.from — user_id
   - Из заголовка X-Telegram-Bot-Api-Secret-Token или из URL pathname — идентифицирует бота
   - lookup в bots → получает master_id
   - Дальше обрабатывает в контексте этого мастера

3. Регистрация нового мастера:
   - Форма на витрине платформы: «Подключи свой бот»
   - Мастер вводит токен → хэш сохраняется в bots
   - Платформа вызывает setWebhook для этого бота
   - Мастер получает свой deep link t.me/<bot_username>/app

4. RLS для мульти-tenancy:
   - Политики используют master_id из JWT claims
   - Каждый запрос фильтруется по master_id

Объясняй каждый шаг. Я новичок в webhook-архитектуре.
```

## 9. Деплой на Beget Cloud VPS (для ботов 24/7)

**Когда:** серверная логика, webhook, фоновые задачи, боты с сообщениями по расписанию.

```
Ты — DevOps-инженер, специалист по Ubuntu Server и systemd.

Задеплой моего бота на Beget Cloud VPS.
IP сервера: <вставь IP>
Логин: root
Пароль: <пусто — Инна введёт>

Веди меня пошагово. Я новичок в Linux и SSH.

Шаги:

1. SSH на сервер: ssh root@<IP>
2. Обновить систему: apt update && apt upgrade -y
3. Установить Node.js (или Python, в зависимости от языка бота)
4. Создать непривилегированного пользователя для запуска бота
5. Склонировать репо или rsync код
6. Скопировать .env (безопасно — через scp или ручной ввод на сервере)
7. Установить зависимости: npm install / pip install
8. Создать systemd-сервис:
   /etc/systemd/system/<project>-bot.service
   - ExecStart=/usr/bin/node /path/to/bot.js
   - Restart=always
   - User=<nonroot>
9. systemctl enable <project>-bot && systemctl start <project>-bot
10. Проверить: systemctl status / journalctl -u <project>-bot -f
11. Настроить webhook у Telegram:
    curl -F "url=https://<domain>/webhook" https://api.telegram.org/bot<TOKEN>/setWebhook

Запрашивай у меня только то, что не можешь сделать сам
(ввод пароля SSH, выбор параметров).

После деплоя — финальная проверка:
1. Открыть t.me/<bot>/app — Mini App загружается
2. Нажать «Записаться» — запись появилась в Supabase
3. Выключить мой компьютер → через час написать боту /start → должен ответить
```
