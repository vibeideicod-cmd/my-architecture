# graphify — индексация проекта в knowledge graph

**Статус:** инструмент **точечного** применения. Не ставим глобально, не индексируем всё подряд.

**Источник:** [safishamsi/graphify](https://github.com/safishamsi/graphify), PyPI-пакет `graphifyy`.

---

## Что делает

Превращает папку с файлами (код + доки + PDF + картинки + видео) в knowledge graph. На выходе:

- `graph.html` — интерактивный граф в браузере
- `GRAPH_REPORT.md` — «god-nodes» (центральные концепции), surprising connections, вопросы которые граф уникально может ответить
- `graph.json` — машинно-читаемый граф для запросов
- Опционально MCP-сервер — структурный доступ к графу через tool calls

Работает в 3 прохода: AST через tree-sitter (локально, без LLM), Whisper для видео/аудио (локально), Claude-субагенты параллельно для доков/картинок.

---

## Зачем нам это

Наша **фишка — до-продажи**. Каждому клиенту мы постоянно продаём новые инструменты, услуги, архитектуру. Чтобы до-продавать точно, надо **видеть что у клиента уже есть** и **где дырки**. Когда у клиента накапливается 20+ артефактов (brief, research, MVP, backend-plan, tg-app/, supabase/, заметки), держать это всё в голове — невозможно.

**Граф решает три конкретные задачи:**

1. **Быстро возвращаться в контекст клиента** после перерыва — `GRAPH_REPORT.md` даёт карту за 1 минуту вместо перечитывания всех файлов.
2. **Видеть возможности для до-продажи** — god-nodes показывают центральные концепции клиента (вокруг них можно строить новые услуги), а «висящие» темы (упомянутые, но не развитые) — это прямые зацепки для up-sell.
3. **Экономить токены Claude** — когда Claude работает по большой папке, чтение `GRAPH_REPORT.md` + точечные запросы к графу дают в **5–70 раз** меньше токенов чем grep по всем файлам (масштабируется с размером корпуса).

## Когда ставим (чек-лист)

Применяем graphify в папке **только если выполнены ВСЕ условия:**

- [ ] В папке **15+ файлов** с реальным содержимым (не считая пустых, ассетов, генерируемых)
- [ ] **Смешанный корпус**: brief + research + код + доки (минимум 2 типа, лучше 3+)
- [ ] Папка **живая на месяцы** — долгосрочный клиент, не одноразовый артефакт
- [ ] Мы возвращаемся туда **регулярно** — раз в неделю или чаще
- [ ] Клиент на **фазе 3+** (Design/Plan и дальше) — на Discovery ещё нечего индексировать

Если хотя бы один пункт не выполнен — **не ставим**. Лишний инструмент без нужды — это усложнение, не усиление.

---

## Где НЕ ставим

- Папки с 1–5 файлами (одностраничные сайты типа `Irina/`, `hub/`)
- Клиенты на фазе 1 Discovery — пока нет корпуса, граф пустой
- Одноразовые прототипы которые будут выброшены
- Папки где весь контент помещается в контекст Claude за один заход

## Как ставить (в конкретную папку)

**Один раз на машину:**
```bash
pip install graphifyy
```
Если `command not found` — на Mac через pipx: `pipx install graphifyy && pipx ensurepath`.

**В конкретную папку клиента (например Beauty):**

1. `cd clients/beauty/`
2. Создать `.graphifyignore` — исключить мусор:
   ```
   node_modules/
   dist/
   build/
   supabase/.temp/
   *.log
   ```
3. Первый билд: `/graphify .` (внутри Claude Code) или `graphify .` (из терминала)
4. Посмотреть `graphify-out/GRAPH_REPORT.md` — это карта клиента
5. Закоммитить `graphify-out/` (но НЕ `graphify-out/cache/` — уже в `.gitignore` проекта добавить)

**Опционально — всегда использовать граф:**
```bash
graphify claude install
```
Устанавливает PreToolUse hook в `.claude/settings.json` — перед каждым Grep/Glob Claude сначала читает `GRAPH_REPORT.md`. Хороший вариант для долгих клиентов.

**Обновление при изменениях:**
- Код поменялся → `graphify update .` (локально, без токенов)
- Доки/research поменялись → `/graphify . --update` (новые LLM-проходы только для изменённых файлов)
- Или поставить git-хук: `graphify hook install` — автоматом после каждого коммита

## Паттерн до-продаж (наш бизнес-процесс)

После каждой существенной вехи клиента (закончили фазу, запустили MVP, получили обратную связь):

1. Пересобрать граф: `/graphify . --update`
2. Открыть свежий `GRAPH_REPORT.md`
3. Посмотреть три вещи:
   - **God-nodes** — что стало центральным? Вокруг этого можно предложить новые услуги. Пример: если у Beauty god-node «расписание мастеров» — можно допродать систему напоминаний, систему отмен, аналитику по загрузке.
   - **Surprising connections** — неочевидные связи между документами клиента. Часто это подсказка для пакетных предложений. Пример: если research по конкурентам связан с brief про Telegram — возможно стоит предложить аналитику конкурентных Telegram-каналов.
   - **Висящие узлы** — темы, которые упомянуты, но не развиты. Это дырки, которые стоит предложить закрыть.
4. Зафиксировать идеи в `ideas.md` или `clients/{имя}/up-sell.md`

**Правило:** этот проход — часть работы Директора перед каждой плановой встречей с клиентом. Не чаще чем раз в 2–3 недели на одного клиента, иначе токены жжём зря.

## Стоимость

- **Первый build крупной папки** (20+ файлов с доками) — расход токенов на extraction. Ориентир: 50–200k токенов на один билд (зависит от объёма текста и картинок).
- **Обновления** (`--update`) — только для изменённых файлов, дёшево.
- **Код-only изменения** — через `graphify update` или git-hook, **0 токенов** (локальный AST).
- **Запросы к готовому графу** — работают на `graph.json`, не перестраивают, экономят в разы.

**Правило бюджета:** первый билд окупается за ~5–10 запросов к графу. Если мы в эту папку возвращаемся реже — не ставим graphify вообще, смысла нет.

---

## Приватность

- Код обрабатывается **локально** через tree-sitter — не уходит в API
- Доки, PDF, картинки **отправляются в Claude API** для extraction (тот же ключ что у Claude Code)
- Видео/аудио **транскрибируется локально** через faster-whisper — не уходит никуда
- Никакой телеметрии, никаких внешних сервисов кроме Anthropic
