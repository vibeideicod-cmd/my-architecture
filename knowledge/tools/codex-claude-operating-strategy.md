# Codex + Claude Code — стратегия совместной работы

Дата: 2026-05-16  
Статус: рабочая стратегия перед аудитом архитектуры

## Зачем нужен этот документ

Проект теперь живёт в двух средах:

- **Claude Code / Кодыч** — среда, где уже описаны `.claude/skills/`, `.claude/agents/`, модельные frontmatter и привычная команда агентов Инны.
- **Codex** — среда, где можно быстро читать репозиторий, делать точные правки, запускать проверки, собирать аудит и синхронизировать документацию.

Главное правило: **агенты команды Инны — это роли и ответственность, а не одна конкретная платформа.**  
Claude Code и Codex могут использовать одни и те же документы `agents/*.md`, но технически исполняют их по-разному.

## Что важно понять про модели

Нельзя просто сказать: “таблица = истина” или “агент-файл = истина”. Нужно различать **рекомендованную модель роли** и **фактическую модель исполнения**.

### 1. `knowledge/tools/model-map.md`

Это стратегическая карта: какая модель **должна быть экономически и качественно оправдана** для роли.

Используется для решений:
- где достаточно Haiku;
- где нужен Sonnet;
- где оправдан Opus;
- стоит ли повышать/понижать модель.

### 2. `knowledge/tools/team-overview.md`

Это справочник команды: кто есть, где файл, к какому департаменту относится, какая рекомендуемая модель и почему.

Используется для навигации Директора и сверки состава команды.

### 3. `agents/*.md`

Это контекст-агенты: роли, инструкции, ответственность.

В Claude Code по текущему правилу из `CLAUDE.md` большинство `agents/*.md` **не являются отдельными runtime-агентами со своей моделью**. Директор читает их как контекст. Значит фактически они работают на модели текущей сессии Директора, а строка “Модель” внутри файла является справочной, если агент не вынесен в `.claude/agents/` или skill.

Вывод: рассинхрон в заголовке `agents/*.md` важен как документационный долг, но он не всегда означает, что агент реально сейчас сжигает более дорогую модель.

### 4. `.claude/agents/*.md`

Это настоящие Claude Code subagents с frontmatter `model`. Сейчас там реализованы рутинные роли:

- `git-manager` — Haiku;
- `deployer-beget` — Haiku.

Их модель реально влияет на экономию Claude Code.

### 5. `.claude/skills/*/SKILL.md`

Это Claude Code skills с frontmatter `model`. Их модели реально влияют на то, какая модель используется при запуске skill.

### 6. Codex

Codex не исполняет `.claude/skills/` и `.claude/agents/` как Claude Code runtime. Для Codex эти файлы — инструкции и источники контекста. Codex может:

- читать агентные роли;
- выполнять аудит;
- править документы;
- запускать shell-проверки;
- сверять карты, модели, файлы и статусы;
- писать новые регламенты и отчёты.

Но модельные метки `opus/sonnet/haiku` в Claude-документах не переключают автоматически модель Codex. Поэтому для Codex нужна отдельная политика: **какие задачи отдаём Codex, а какие Claude Code**.

## Принцип выбора модели: качество + экономия

Экономия важна, но не как “всё удешевить”. Правильная формула:

> Самая дешёвая модель, которая стабильно держит нужное качество без ручной переделки Инной.

Если дешёвая модель:
- теряет контекст;
- принимает поверхностные решения;
- создаёт больше правок;
- заставляет Инну перепроверять руками;
- ломает архитектуру;

то это не экономия, а перенос стоимости на Инну и команду.

## Решение по ключевым ролям перед аудитом

Эти решения не финальные навсегда. Их нужно проверить в аудите.

| Роль | Базовая рекомендация | Почему |
|---|---|---|
| Director | Sonnet или Opus по сложности задачи | Операционная маршрутизация обычно Sonnet. Архитектурный аудит, где надо держать всю систему и принимать структуру проверки, может временно идти на более сильной модели |
| Project-Manager | Sonnet | Параллельные статусы, блокеры и эскалации требуют суждения |
| QA | Sonnet | Проверяет не только чек-лист, но смысл, бренд, ЦА и риски |
| Git Manager | Haiku | Детерминированные git-команды |
| Deployer-Beget | Haiku | Деплой по сценарию |
| Product Builder | Opus | Discovery/Critique/Plan продукта с долгосрочными последствиями |
| Systems | Opus для архитектуры, Sonnet/Codex для ограниченной реализации | Архитектура бота/CRM/интеграций высокая ставка; точечные правки можно дешевле |
| AI Builder | Opus для архитектуры продукта, Sonnet/Codex для правок базы/промпта по готовой схеме | Высокая цена ошибки на старте, ниже при обслуживании |
| Marketer | Opus для стратегии, Sonnet для текущих кампаний/контента | Не вся маркетинговая работа одинаково стратегическая |

## Как делим работу между Codex и Claude Code

### Codex лучше использовать для

- аудита репозитория и структуры файлов;
- поиска рассинхронов между `team-overview.md`, `model-map.md`, `agents/*.md`, `.claude/skills/`;
- создания и обновления Markdown-документации;
- точечных правок в коде и HTML;
- запуска shell-проверок, `rg`, `git diff`, smoke-проверок;
- подготовки отчётов: `PROJECT-STATUS`, `CLEANUP-CANDIDATES`, acceptance audit;
- синхронизации карт `FILES.md`, `README.md`, `PLAN.md`.

Codex хорош как “технический аудитор и редактор репозитория”: быстро видит файлы, аккуратно правит, проверяет фактами.

### Claude Code лучше использовать для

- задач, где уже есть готовый Claude skill;
- работы через `.claude/agents/` subagents;
- сценариев, где важна привычная команда Кодыча и её skill-механика;
- генерации/исполнения Claude-native пайплайнов;
- задач, где нужно проверить, что Claude Code runtime реально экономит модель через `model:` frontmatter.

### Дублировать в обеих средах стоит только когда

- меняем модельную политику агентов;
- делаем приёмочный аудит всей архитектуры;
- принимаем решение, которое влияет на клиентский продукт или деньги;
- есть риск, что одна среда “видит” процесс иначе.

Формат: Codex делает фактический repo-аудит, Claude Code/Директор проходит операционный сценарий, затем сверяем выводы.

## Что делать с текущим рассинхроном моделей

Не считать автоматически, что `team-overview.md` прав или что `agents/*.md` прав.

Правильный порядок:

1. Снять фактическую таблицу:
   - `model-map.md`;
   - `team-overview.md`;
   - заголовки `agents/*.md`;
   - `.claude/agents/*.md` frontmatter;
   - `.claude/skills/*/SKILL.md` frontmatter.
2. Для каждого расхождения определить тип:
   - **runtime-критично** — реально влияет на расход модели;
   - **документационный долг** — вводит команду в заблуждение, но модель не переключает;
   - **осознанное повышение** — качество важнее экономии;
   - **осознанное понижение** — качество не теряется, экономия оправдана.
3. Принять решение по каждой спорной роли.
4. Только после решения синхронизировать документы.

## Спорные роли на проверку

На 2026-05-16 уже виден рассинхрон:

| Роль | В `model-map/team-overview` | В `agents/*.md` | Что проверить |
|---|---|---|---|
| Director | Sonnet | Opus | Нужен ли Opus всегда или только на архитектурных аудитах |
| Marketer | Opus | Sonnet | Разделить стратегию Opus и текущий маркетинг Sonnet или оставить руководителя на Opus |
| Systems | Opus | Sonnet | Разделить архитектуру Systems Opus и реализацию/правки Sonnet/Codex |
| AI Builder | Opus | Sonnet | Разделить старт AI-продукта Opus и обслуживание Sonnet |

Нельзя править эти строки механически. Нужно решить, какая модель даёт нужное качество при разумной стоимости.

## Правило для приёмочного аудита

Перед запуском полного архитектурного аудита добавить шаг:

**Этап M — Runtime/model strategy preflight**

Проверить:
- понимаем ли, какая среда ведёт аудит: Codex, Claude Code или обе;
- где модель реально исполняется, а где она только справочная;
- какие роли требуют Opus на время аудита;
- какие проверки Codex делает дешевле и точнее;
- какие Claude skills/subagents нужно проверять именно в Claude Code.

Если этот шаг пропустить, аудит может перепутать документационный рассинхрон с реальным расходом моделей.
