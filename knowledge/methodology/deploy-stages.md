# Двухступенчатый деплой клиентских продуктов

**Принцип:** **никогда не деплоим первый раз сразу на Beget или сервер клиента.**

Сначала клиент видит работающий продукт на Vercel (быстрый demo-хостинг), даёт правки, потом — финальный prod-деплой на Beget или его инфру.

---

## Этап 1 — Demo на Vercel

**Цель:** клиент быстро видит работающий продукт, может пощёлкать, дать правки. Без ожиданий «настроим домен / SSL / мониторинг» — сразу рабочая ссылка.

**Когда:** после Build (Фаза 4 Playbook), перед финальным согласованием.

**Что делаем:**
- Деплой на Vercel за 5-10 минут (через `vercel --prod` CLI или drag-n-drop ZIP в панели)
- Короткая ссылка `<project>.vercel.app` отдаётся клиенту
- Клиент проверяет 1-3 дня

**Что НЕ делаем на этом этапе:**
- Свой домен клиента — рано
- SEO-оптимизация финальная — рано
- Подключение аналитики (Метрика, GA) — рано
- Мониторинг через `/monitoring-check` — рано

**Артефакт:** Vercel-URL включается в коммерческое предложение от [agents/sales.md](../../agents/sales.md) как «вот ссылка для проверки до подписания».

---

## Этап 2 — Сбор обратной связи и правки

**Цель:** довести до состояния «клиент сказал ОК».

- Клиент даёт правки → правки вносятся командой → передеплой на тот же Vercel-URL
- Цикл «правки → передеплой» столько раз, сколько нужно
- Финальный QA через [agents/qa.md](../../agents/qa.md) на Vercel-версии перед переходом к Этапу 3

---

## Этап 3 — Production-деплой

**Цель:** перевести на финальную инфраструктуру где продукт будет жить.

**Куда:**
- **Лендинги, статика, Mini App без бэкенда** → Beget shared (через скилл `/deploy`)
- **Боты 24/7, бэкенд, серверная логика** → Beget Cloud VPS (Cheerful Marik) или Model Zarek (после перехвата у Димы)
- **Если у клиента своя инфра** → его сервер (договариваемся отдельно, обвязка через [agents/devops-infra.md](../../agents/devops-infra.md))

**Что делаем:**
- Деплой через [agents/deployer-beget.md](../../agents/deployer-beget.md) или скилл `/deploy`
- Подключение домена клиента + SSL (Let's Encrypt)
- Продакшн-настройки `.env` с реальными ключами (по правилам из [agents/director.md](../../agents/director.md), раздел «Работа с секретами»)
- Мониторинг через скилл `/monitoring-check`
- Аналитика, SEO, всё остальное

---

## Когда Vercel-этап пропускаем

**По умолчанию — всегда двухступенчатый деплой (Vercel demo → Beget prod).**

Обходим только в этих случаях:

1. **Прямое указание Инны** «пропускаем Vercel» / «сразу на Beget» — главный путь обхода для клиентских продуктов
2. **Внутренние инструменты команды Инны** (агенты, скиллы, методички в `my-architecture/`) — клиента нет, demo-этап не применим
3. **Срочный hotfix уже работающего продукта** — точечная правка, не новая фича
4. **Telegram Mini App** — Vercel обязателен как промежуточный HTTPS-шаг для @BotFather, не как demo для клиента. Финальный prod всё равно Beget Cloud VPS
