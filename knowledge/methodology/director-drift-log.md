# Журнал сходов Директора с методологии

> **Назначение:** Каждый раз, когда Инна ловит Директора на «делаешь не по методологии» — запись сюда. Это база для drift detector (см. OPEN.md → раздел «Связанная задача»), а сейчас — текстовый журнал-самопроверка.

## Как заполнять

Каждая запись — отдельный блок:

```
### YYYY-MM-DD · короткое имя факапа

**Что просили:** что сказала Инна
**Что сделал не так:** конкретное нарушение методологии (со ссылкой на правильную)
**Где правильная методология:** файл и якорь
**Как должен был сделать:** короткий правильный ход
**Сигнал (триггер) на будущее:** по какому слову/задаче ловить себя в этой ловушке
```

Старые записи не редактируем — только дописываем сверху.

## Базовые проверки перед стартом любой задачи (обязательный чек-лист)

Прежде чем что-то делать с **архитектурой / новым продуктом / стратегией клиента** — пройди эти 5 шагов в порядке (из [product-creation.md](../playbooks/product-creation.md) раздел 0.2):

1. **Идея** — сформулировано «Я создаю ЧТО для КОГО чтобы решить КАКУЮ проблему»?
2. **Исследование** — есть Research по 5 направлениям (конкуренты, функции, UX, технологии, монетизация)?
3. **Критика** — прогнал через 4 роли (пользователь / UX-дизайнер / конкурент / тестировщик) + «Ты уверен?»
4. **План** — есть brief.md / план реализации / приоритеты?
5. **Реализация** — только сейчас передаю исполнителям

**Если пункт 2 или 3 пропущен — стоп. Возврат назад. Любая работа департаментов (юр / тех / финансы / QA) делается ТОЛЬКО на согласованной идее после Research и Critique.**

Дополнительный фильтр (из [feedback_audit_order_business_first.md](../../memory/feedback_audit_order_business_first.md)): «реализуемо / нужно / будет продавать» — это часть шага 3 (Критика «глазами конкурента» + «Ты уверен?»).

## Журнал

### 2026-05-09 · Прогон доменной карты — прыгнул в реализацию

**Что просили:** «Прогон архитектуры через команду — Business-Architect координирует, Yuridika+PD-Lawyer проверяют ОКВЭД, DevOps — реализуемость, Financial — стоимость, QA — финальный чек».

**Что сделал не так:**
- Расценил это как «беги через всех агентов и собирай отчёт».
- Запустил Business-Architect в режиме «соответствует ли карта 7-слойной модели» — но это уже шаг 4 (План), а не шаг 1-3.
- Шаги 1-3 пропустил полностью: формулировки идеи в правильном виде нет, Research («какие доменные карты делают похожие проекты — экосистемы экспертов с подписной платформой») нет, Critique по 4 ролям нет.
- В первом ответе сразу дал «вердикт Business-Architect» с правками — то есть выдал РЕШЕНИЕ без анализа, на котором это решение должно стоять.

**Где правильная методология:** [knowledge/playbooks/product-creation.md:28](../playbooks/product-creation.md) раздел 0.2 «5 шагов» + Фаза 2 «Research» + Фаза 2.5 «Critique».

**Как должен был сделать:**
1. Переформулировать карту как идею: «Я создаю экосистему доменов для КОГО (Инна-эксперт под ОКВЭД 70.22 + её клиенты) чтобы решить КАКУЮ проблему (одна точка входа для всех болей + витрина продуктов + закрытая платформа подписчиков, без юридических рисков)».
2. Заказать Research у Analyst: «найди 5-7 похожих архитектур (например, экосистемы Алёны Венум, Михаила Дашкиева, Аяза Шабутдинова — экспертов с воронкой, платформой, демо-витриной); по каждой — какие поддомены, что на корне, как продаёт, где деньги, что копировать / чего избегать».
3. Прогнать карту через Critique 4 ролями (пользователь / UX / конкурент / тестировщик).
4. Только после согласия Инны на скорректированную карту — план реализации через департаменты.
5. Юр / тех / финансы — это шаг 5 (Реализация), а не самостоятельный анализ.

**Сигнал (триггер) на будущее:** триггер «прогон через команду» / «через всех агентов» / «через департаменты» — это НЕ команда «беги в шаг 5». Это команда «доведи до шага 5 правильно». Если предыдущих шагов нет — сначала восполнить их.
