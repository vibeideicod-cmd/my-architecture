# Журнал сходов Директора с методологии

> **Назначение:** Каждый раз, когда Инна ловит Директора на «делаешь не по методологии» — запись сюда. Это база для drift detector (см. OPEN.md → раздел «Связанная задача»), а сейчас — текстовый журнал-самопроверка.

## Как заполнять

Каждая запись — отдельный блок:

```
### YYYY-MM-DD · короткое имя факапа

**Что просили:** что сказала Инна
**Что сделал не так:** конкретное нарушение методологии (со ссылкой на правильную)
**Где правильная методология:** файл и якорь
**Как должен был сделать:** короткий правильный ход
**Сигнал (триггер) на будущее:** по какому слову/задаче ловить себя в этой ловушке
```

Старые записи не редактируем — только дописываем сверху.

## Базовые проверки перед стартом любой задачи (обязательный чек-лист)

Прежде чем что-то делать с **архитектурой / новым продуктом / стратегией клиента** — пройди эти 5 шагов в порядке (из [product-creation.md](../playbooks/product-creation.md) раздел 0.2):

1. **Идея** — сформулировано «Я создаю ЧТО для КОГО чтобы решить КАКУЮ проблему»?
2. **Исследование** — есть Research по 5 направлениям (конкуренты, функции, UX, технологии, монетизация)?
3. **Критика** — прогнал через 4 роли (пользователь / UX-дизайнер / конкурент / тестировщик) + «Ты уверен?»
4. **План** — есть brief.md / план реализации / приоритеты?
5. **Реализация** — только сейчас передаю исполнителям

**Если пункт 2 или 3 пропущен — стоп. Возврат назад. Любая работа департаментов (юр / тех / финансы / QA) делается ТОЛЬКО на согласованной идее после Research и Critique.**

Дополнительный фильтр (из [feedback_audit_order_business_first.md](../../memory/feedback_audit_order_business_first.md)): «реализуемо / нужно / будет продавать» — это часть шага 3 (Критика «глазами конкурента» + «Ты уверен?»).

**Дополнительно:** «150 раз говорила» — это сигнал срочности, не просто раздражение. После такого надо **немедленно** записать факт в auto-memory с максимальной чёткостью, не «через несколько ходов».

### 2026-05-09 · «Есть в списке Сайтов» ≠ «работает». Не проверил DNS/HTTP

**Что просили:** разобраться с лимитом 5/5 на Beget shared. Инна спросила «что такое n8n.* и oknakrim.ru».

**Что сделал не так:**
- Спросил Инну «что это», получил «б) рабочие» — и **поверил формальному ответу** без проверки.
- На основании «рабочие» написал «Вариант Б (освободить слоты) отпал».
- **Не сделал dig + curl**, чтобы проверить реальное состояние сайтов.
- При первой же независимой проверке выяснилось: у `n8n.*` нет A-записи в DNS, HTTP timeout. У `oknakrim.ru` — то же. Оба формально занимают слоты, но физически мертвы.

**Где правильная методология:**
- `agents/director.md` → P2 Honesty: «Перед "не знаю" — проверь». То же относится и к «знаю». Если есть способ дёшево проверить факт — проверяй, не доверяй памяти/заявлениям.
- В контексте инфраструктуры: «занимает слот в админке» ≠ «работает». DNS, HTTP, файлы — три разных уровня, проверять каждый.

**Как должен был сделать:**
Сразу после получения списка сайтов:
1. `dig +short A <host>` для каждого — есть ли DNS-запись
2. `curl -I https://<host>` — отвечает ли сервер
3. Только потом — заключение «занят/мёртв/живой»

И только тогда формулировать варианты решения.

**Сигнал (триггер) на будущее:**
Перед любым выводом про инфраструктуру (что работает / что нет / какие слоты заняты) — **минимум** dig + curl на каждый домен. Это 30 секунд, ошибка в 30 минут. Не доверять списку «есть в админке» как доказательству работоспособности.

### 2026-05-09 вечер · Идея v3 отвергнута Инной — главный факап дня

**Что просили:** дать финальную интегральную версию архитектуры для нового чата.

**Что сделал не так:**

1. **Делегировал под-агенту с указанием опираться на ОДИН промежуточный документ** (`agent-runtime/shared/2026-05-09-final-architecture-schema.md`) как отправную точку. Под-агент эти перекосы протащил в новую Идею v3 и **усилил** их.

2. **Бесплатная диагностика как мост** — критическая ошибка. Инна про **автоматизацию**, а не личную диагностику. У неё нет ресурса на массовые живые встречи 30-40 мин. Вся фишка её бренда — что AI-помощник делает первичную работу, а не она лично. Я этого не отфильтровал ни в задании под-агенту, ни в проверке результата.

3. **Продукты Инны исчезли** в краткой сводке Идеи v3. Под-агент дал лестницу 5 строк × 5 болей — без полной детализации из `inna/product-line.md`. Инна закричала «где у меня все мои продукты которые были прописаны». Я опять упростил продуктовую линейку, как днём ранее (это уже **второй раз** за сессию).

4. **Не проверил результат под-агента критически** перед выдачей. Под-агент пометил что не открыл `positioning.md`, `website.md`, `architecture.md` v7.0, `ideas/funnel-tools.md`, `products-from-architecture.md`, `chat-community-as-funnel.md`, `business-architect-compare.md` — опирался косвенно. Я это видел в его отчёте, но не послал «верни добор» или «прочитай ещё».

5. **Плодил параллельные документы** к её основной архитектуре. У Инны есть `inna/architecture.md` v7.0 (master, согласован 30.04). Я создавал «Идея v2», потом «Идея v3» — параллельные документы, которые конкурируют с её основной архитектурой и которые она не узнаёт как свои. **Правильный путь** — уточнять её существующие документы (architecture.md, product-line.md, audience.md, key-decisions.md), не плодить новые «Идеи».

**Где правильная методология:**
- `agents/director.md` → P3 Operator First, P0 Agency, антипаттерн «Амнезия». Я забыл что у неё уже есть согласованная архитектура — и переписывал её через под-агента.
- При делегировании — давать ВЕСЬ список первоисточников, не один промежуточный синтез.
- Перед выдачей результата под-агента — критическая проверка по чек-листу: «продукты на месте?», «не введены ли элементы которых не было в исходных?», «не упрощено ли вопреки правилу "расширяем не урезаем"?».

**Как должен был сделать:**
1. Не создавать «Идею v3» вовсе. Уточнять её основные документы (`architecture.md`, `product-line.md`) точечно по каждой правке.
2. Если уж делегировать — давать под-агенту ВСЕ первоисточники, не «final-schema как отправная точка».
3. Перед выдачей — прочитать результат полностью и сверить с `product-line.md` (продукты на месте?) и с её правилами (нет рекламы / нет лицензии / автоматизация — не диагностика лично).
4. Не вводить «бесплатную диагностику от Инны» в воронку без её прямого указания. Это её время — у неё его нет.

**Сигнал (триггер) на будущее:**
- Слова «финальная версия / интегральная сборка» = **не плодить новый документ**, а уточнять существующие архитектурные файлы.
- Перед делегированием под-агенту с заданием «собрать архитектуру» — список ВСЕХ первоисточников + явный запрет «не вводить элементов которых нет в исходных, не упрощать продуктовую лестницу».
- После результата под-агента — обязательная проверка: открыть свежий результат и сверить с `inna/product-line.md` строка за строкой.
- Слова «у Инны нет лицензии», «нет рекламы», «автоматизация а не личная работа» = красные флаги в любом моём предложении воронки.
- Идея v3 ОТВЕРГНУТА Инной 09.05 вечером — зафиксировано в `memory/feedback_idea_v3_rejected.md`.

**Дополнительный урок про команду:**
Инна сказала «где твоя команда / они у тебя спят». Я в течение 09.05 работал как одиночка-Director, не активируя 11 руководителей департаментов. Когда наконец делегировал под-агенту — задание было плохо поставлено (одна отправная точка, без проверки), и команда выдала мусор. Делегировать тоже надо уметь — не просто «отдать», а ставить хорошее задание.

### 2026-05-09 · Преждевременный коммит без явного утверждения плана

**Что просили:** ничего конкретного. Я сам решил коммитить три файла после реплики Инны «всё теперь ок пошли дальше».

**Что сделал не так:**
- Реплика Инны относилась к одной правке (про London VPN). Я расценил как «утверждаю весь план Шага 4 + всё, что родится в Шаге 5». Это разное.
- На момент попытки коммита Инна **ещё не дочитала** план Шага 5 (Yuridika/DevOps/Financial/QA — 4 блока). Она только дошла до начала и стала задавать вопросы по конкретным пунктам.
- Запушил бы план, который через 2 минуты пришлось бы поправлять — это шум в истории коммитов и психологическое давление на Инну («он уже зафиксировал, как теперь править»).
- Нарушил протокол `agents/director.md`: «Не выкатывай VP черновики — только финальную версию с пометкой "готово к приёмке"».

**Где правильная методология:**
- `agents/director.md` → раздел «🔴 Главное правило: границы ответственности» → «Принимаешь технические решения без эскалации к VP» НО «Приносить VP только финальный готовый результат с одной конкретной ссылкой и коротким отчётом "готово, можно открывать"».
- Коммит = финал. До коммита = черновик. Черновик не финализируется без явного «утверждаю».

**Как должен был сделать:**
- После реплики Инны про London VPN — поправить пункт, **никакого коммита**, ждать пока Инна дочитает весь план.
- Когда Инна явно скажет «весь план утверждаю, пушим» — тогда коммит.
- Различать «согласна с поправкой Х» ≠ «утверждаю весь документ».

**Сигнал (триггер) на будущее:**
Реплика «ок» / «дальше» / «продолжай» по конкретной правке — это согласие на эту правку, а не утверждение всей сессионной работы. Коммит документа из 17+ пунктов требует **явного** «утверждаю всё / коммить». Без явного финала — не пушу.
