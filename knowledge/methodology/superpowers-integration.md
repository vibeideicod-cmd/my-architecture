# Superpowers — инженерный слой дисциплины

Дата внедрения: 2026-05-15

Источник: `obra/superpowers` — внешний фреймворк агентских скиллов для разработки: brainstorming, planning, TDD, debugging, code review, worktrees, finish branch.

В нашей архитектуре Superpowers не заменяет Director, Product Builder, `/brief`, `/discovery`, Playbook и QA. Он добавляет жёсткий инженерный контур для задач, где есть код, баги, техническая архитектура, деплой или риск поломать рабочий продукт.

## Где живёт в проекте

| Уровень | Файл | Зачем |
|---|---|---|
| Канон методологии | `knowledge/methodology/superpowers-integration.md` | Источник правил: когда применять и как адаптировать под нас |
| Локальный скилл | `.claude/skills/superpowers/SKILL.md` | Быстрый запуск методологии внутри Claude Code |
| Маршрутизация | `CLAUDE.md`, `DISPATCH-README.md`, `agents/routing-table.md` | Чтобы Director вспоминал методологию автоматически |
| Исполнители | `agents/product-builder.md`, `agents/websites.md`, `agents/systems.md`, `agents/ai-builder.md` | Чтобы технические задачи не начинались с кода |
| Контроль | `agents/qa.md`, `agents/devops-infra.md`, `agents/git-manager.md` | Чтобы результат проверялся фактами, тестами, diff и деплоем |

## Когда включать обязательно

Superpowers обязателен, если задача содержит хотя бы один пункт:

- новый цифровой продукт: сайт, лендинг, квиз, калькулятор, бот, Mini App, веб-приложение, AI-консультант;
- изменение поведения существующего продукта;
- баг, ошибка теста, сбой деплоя, странное поведение сервера;
- техническая архитектура, интеграции, API, база данных, авторизация, роли, платежи;
- клиентский продукт, который будет опубликован или передан клиенту;
- изменения больше чем в 3 файлах кода или рефакторинг;
- работа, где результат должен быть воспроизводимым, проверяемым и поддерживаемым.

## Когда не включать как жёсткий процесс

Не включать полный Superpowers-процесс для:

- быстрых текстов, постов, сценариев, офферов;
- сортировки входящих и раскладки знаний;
- исследовательских заметок без реализации;
- простых правок документации;
- личной операционной переписки с Инной.

Для этих задач остаётся обычный флоу Director-а и профильных агентов.

## Адаптация Superpowers под нашу систему

Оригинальный Superpowers предлагает цепочку:

`brainstorming → writing-plans → implementation → TDD → code review → finish branch`

В нашей архитектуре это мапится так:

| Superpowers | У нас |
|---|---|
| brainstorming | `/brief`, `/discovery`, `product-builder` Фаза 1 |
| writing-plans | `brief.md`, `BACKEND-PLAN.md`, `PLAN.md`, при необходимости отдельный implementation plan |
| test-driven-development | для `systems`, `ai-builder`, backend, JS-логики, ботов, API; для чистого HTML — проверочные чек-листы и ручной smoke test |
| systematic-debugging | `systems`, `devops-infra`, `qa`: сначала воспроизведение и корень, потом фикс |
| requesting-code-review | `qa` + при техническом коде code-review |
| finishing-development-branch | `git-manager` + `deployer-beget` + запись в `project-log.md` |

## Правило для Director

При любой инженерной задаче Director сначала спрашивает себя:

1. Это код / продукт / баг / деплой / интеграция?
2. Есть ли уже `brief.md`, `idea.md`, `BACKEND-PLAN.md` или `PLAN.md`?
3. Если нет — нельзя начинать реализацию. Сначала Discovery или короткая спецификация.
4. Если это баг — нельзя чинить до воспроизведения и гипотезы корня.
5. Если это новая функция — сначала критерии готовности и план проверки.
6. После реализации — QA, тесты/smoke test, git diff, коммит, project-log.

## Минимальный режим для маленьких задач

Если задача техническая, но маленькая, не раздувать процесс. Достаточно 5 строк перед реализацией:

```markdown
## Мини-спека
Цель:
Файлы:
Критерий готовности:
Проверка:
Риск:
```

После этого можно делать.

## Полный режим для крупных задач

Если задача затрагивает клиентский продукт, бэкенд, бота, Mini App, AI-консультанта или 3+ файла:

1. Сформировать или обновить `idea.md` / `brief.md` / `BACKEND-PLAN.md`.
2. Разбить работу в `PLAN.md` на чекбоксы.
3. Для логики написать failing test или минимальный воспроизводимый сценарий до фикса.
4. Реализовать маленькими блоками.
5. После каждого блока проверять результат.
6. Перед сдачей запустить `qa`.
7. Перед деплоем проверить `git status`, `git diff`, отсутствие секретов.
8. После завершения обновить `project-log.md` или дневник.

## Кто обязан применять

| Агент | Когда |
|---|---|
| Director | Распознаёт инженерную задачу и включает режим |
| Product Builder | Новые продукты, планирование, `brief.md`, `BACKEND-PLAN.md` |
| Websites | Лендинги, сайты, TMA, JS-логика, правки поведения |
| Systems | Боты, CRM, API, Supabase, интеграции, баги |
| AI Builder | AI-консультанты, системные промпты, база знаний, сценарии |
| QA | Проверка результата, smoke test, code review |
| DevOps/Infra | Деплой, серверы, диагностика, окружение |
| Git Manager | Чистый diff, осмысленный commit, push, finish branch |
| Skill-Auditor | Проверяет, не конфликтует ли новый скилл/агент с методологией |

## Главный принцип

Superpowers нужен не ради бюрократии. Он включается там, где цена ошибки выше цены короткого плана.

Для Инны это означает: меньше переделок, меньше случайного кода, больше проверяемых результатов, которые можно передавать клиенту или тиражировать как продукт.
