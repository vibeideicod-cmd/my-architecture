# Superpowers — инженерный слой дисциплины

Дата внедрения: 2026-05-15

Источник: `obra/superpowers` — внешний фреймворк агентских скиллов для разработки: brainstorming, planning, TDD, debugging, code review, worktrees, finish branch.

В нашей архитектуре Superpowers не заменяет Director, Product Builder, `/brief`, `/discovery`, Playbook и QA. Он добавляет жёсткий инженерный контур для задач, где есть код, баги, техническая архитектура, деплой или риск поломать рабочий продукт.

## Где живёт в проекте

| Уровень | Файл | Зачем |
|---|---|---|
| Канон методологии | `knowledge/methodology/superpowers-integration.md` | Источник правил: когда применять и как адаптировать под нас |
| Локальный скилл | `.claude/skills/superpowers/SKILL.md` | Быстрый запуск методологии внутри Claude Code |
| Маршрутизация | `CLAUDE.md`, `DISPATCH-README.md`, `agents/routing-table.md` | Чтобы Director вспоминал методологию автоматически |
| Исполнители | `agents/product-builder.md`, `agents/websites.md`, `agents/systems.md`, `agents/ai-builder.md` | Чтобы технические задачи не начинались с кода |
| Контроль | `agents/qa.md`, `agents/devops-infra.md`, `agents/git-manager.md` | Чтобы результат проверялся фактами, тестами, diff и деплоем |

## Когда включать обязательно

Superpowers обязателен, если задача содержит хотя бы один пункт:

- новый цифровой продукт: сайт, лендинг, квиз, калькулятор, бот, Mini App, веб-приложение, AI-консультант;
- изменение поведения существующего продукта;
- баг, ошибка теста, сбой деплоя, странное поведение сервера;
- техническая архитектура, интеграции, API, база данных, авторизация, роли, платежи;
- клиентский продукт, который будет опубликован или передан клиенту;
- изменения больше чем в 3 файлах кода или рефакторинг;
- работа, где результат должен быть воспроизводимым, проверяемым и поддерживаемым.

## Когда не включать как жёсткий процесс

Не включать полный Superpowers-процесс для:

- быстрых текстов, постов, сценариев, офферов;
- сортировки входящих и раскладки знаний;
- исследовательских заметок без реализации;
- простых правок документации;
- личной операционной переписки с Инной.

Для этих задач остаётся обычный флоу Director-а и профильных агентов.

## Адаптация Superpowers под нашу систему

Оригинальный Superpowers предлагает цепочку:

`brainstorming → writing-plans → implementation → TDD → code review → finish branch`

В нашей архитектуре это мапится так:

| Superpowers | У нас |
|---|---|
| brainstorming | `/brief`, `/discovery`, `product-builder` Фаза 1 |
| writing-plans | `brief.md`, `BACKEND-PLAN.md`, `PLAN.md`, при необходимости отдельный implementation plan |
| test-driven-development | для `systems`, `ai-builder`, backend, JS-логики, ботов, API; для чистого HTML — проверочные чек-листы и ручной smoke test |
| systematic-debugging | `systems`, `devops-infra`, `qa`: сначала воспроизведение и корень, потом фикс |
| requesting-code-review | `qa` + при техническом коде code-review |
| finishing-development-branch | `git-manager` + `deployer-beget` + запись в `project-log.md` |

## Правило для Director

При любой инженерной задаче Director сначала спрашивает себя:

1. Это код / продукт / баг / деплой / интеграция?
2. Есть ли уже `brief.md`, `idea.md`, `BACKEND-PLAN.md` или `PLAN.md`?
3. Если нет — нельзя начинать реализацию. Сначала Discovery или короткая спецификация.
4. Если это баг — нельзя чинить до воспроизведения и гипотезы корня.
5. Если это новая функция — сначала критерии готовности и план проверки.
6. После реализации — QA, тесты/smoke test, git diff, коммит, project-log.

## Минимальный режим для маленьких задач

Если задача техническая, но маленькая, не раздувать процесс. Достаточно 5 строк перед реализацией:

```markdown
## Мини-спека
Цель:
Файлы:
Критерий готовности:
Проверка:
Риск:
```

После этого можно делать.

## Полный режим для крупных задач

Если задача затрагивает клиентский продукт, бэкенд, бота, Mini App, AI-консультанта или 3+ файла:

1. Сформировать или обновить `idea.md` / `brief.md` / `BACKEND-PLAN.md`.
2. Разбить работу в `PLAN.md` на чекбоксы.
3. Для логики написать failing test или минимальный воспроизводимый сценарий до фикса.
4. Реализовать маленькими блоками.
5. После каждого блока проверять результат.
6. Перед сдачей запустить `qa`.
7. Перед деплоем проверить `git status`, `git diff`, отсутствие секретов.
8. После завершения обновить `project-log.md` или дневник.

## Когда отключить Superpowers, если он мешает

Superpowers — инструмент, а не догма. Если он замедляет работу больше, чем защищает — отключаем.

Признаки, что Superpowers стал тормозом:

- мини-спека занимает больше времени, чем сама правка;
- Инна вынуждена три раза согласовывать мелочь, которую можно было сделать за минуту;
- процесс заставляет писать спеку на задачу из 1-2 строк кода;
- задача уже сделана аналогичным способом 3+ раз, повторять спеку — это уже ритуал, а не дисциплина.

Что делать в таких случаях:

1. Director переходит в режим vibe (без мини-спеки) и делает задачу.
2. Фиксирует в `project-log.md` или в дневнике сессии: «Superpowers был лишним здесь, потому что X».
3. Раз в неделю Planner/Director собирает такие пометки и решает, нужно ли подкорректировать триггеры в этом каноне.

Это не оправдание пропустить дисциплину на серьёзном продукте. Это защита от того, чтобы методология не превратилась в ритуал ради ритуала на простых задачах.

## Тесты и воспроизводимый сценарий

Для логики (боты, API, AI-консультанты, Mini App, бэкенд) — нужно стремиться к тому, чтобы тесты писались. Это правильная инженерная практика и сейчас у нас её нет — но именно через это правило она формируется.

Правило для логики:

1. **Желательный максимум** — failing test, который ловит проблему до фикса. Запускается одной командой, проверяет, что бот/API/функция работает на типовых и граничных сценариях.
2. **Минимум, который должен быть всегда** — воспроизводимый сценарий: написан текстом, что вводим/нажимаем/отправляем и что должно получиться. Это не заменяет тесты, но позволяет повторно проверить вручную.

Для статической вёрстки (HTML/CSS, лендинги, визитки) — тесты не пишутся, обязателен smoke test: открыть, проверить адаптив, ссылки, CTA, отсутствие битых якорей.

Если на проекте тесты вообще не настроены — это технический долг, который нужно зафиксировать в `project-log.md` клиента и постепенно закрывать.

## Ревизия методологии (sunset-аудит)

**Запланировано на 2026-06-15.** Через месяц после внедрения проверяем:

- сколько раз реально применялся Superpowers (по записям в `project-log.md`, сессиях, коммитах);
- помог ли он избежать ошибок или стал тормозом;
- какие триггеры срабатывают правильно, какие — лишние;
- нужно ли изменить границы мини/полного режима;
- начала ли формироваться привычка писать тесты.

Ответственные: Product Builder + QA. Черновик отчёта пишется в `agent-runtime/shared/2026-06-15-superpowers-retro.md`. **Финальная версия с решением переносится в `knowledge/methodology/2026-06-15-superpowers-retro.md`** (потому что `agent-runtime/` — рабочая зона и не архив; важные ретроспективы должны жить в knowledge).

Возможные решения по итогам: оставляем как есть, корректируем триггеры, понижаем порог для какого-то режима, сворачиваем методологию.

## QA в Superpowers — это тот же QA-агент

Когда канон или скилл говорит «передай в QA», «после реализации запусти QA», «перед готово — QA» — это **один и тот же вызов** `agents/qa.md`, не отдельный шаг и не параллельный процесс.

В `agents/qa.md` уже есть «Шаг 3.5 — Superpowers-контур» с расширенным чек-листом для инженерных задач: спека/brief, воспроизведение бага, тесты или сценарий, проверка фактически выполнена, `git diff` без секретов, путь к результату.

Director вызывает QA **один раз** после реализации — этот вызов закрывает и обычный QA, и Superpowers-проверку. Двойные вызовы QA — это шум, а не дисциплина.

## Как фиксировать применение Superpowers

После каждой инженерной задачи Director добавляет запись:

- в `clients/<name>/project-log.md` — если задача была по конкретному клиенту;
- в `memory/sessions/YYYY-MM-DD.md` — если задача была по архитектуре, агентам, knowledge или другой внутренней работе.

Шаблон записи:

```markdown
#superpowers-applied
Режим: mini / full / off
Задача:
Помог:
Мешал:
Проверка:
```

`off` означает, что Superpowers был лишним для этой задачи и Director перешёл в режим vibe. **Запись всё равно нужна** — `off`-кейсы тоже материал для sunset-аудита: они показывают, где границы триггеров проходят неправильно.

Без этого правила sunset 2026-06-15 будет проведён "по ощущениям", а не на фактах.

## Калибровка первой недели

**С 2026-05-15 по 2026-05-22 (первые 7 дней)** Director включает Superpowers на каждой инженерной задаче без исключения. Инна даёт обратную связь:

- «правильно подключился» — триггер сработал верно;
- «здесь Superpowers лишний» — триггер сработал ложно, методология тормозит больше, чем защищает;
- «здесь надо было полный режим, а ты включил мини» — порог режима неправильный.

Через неделю Director собирает обратную связь и фиксирует калибровку в этом каноне (раздел «Когда включать обязательно» и «Когда не включать как жёсткий процесс»).

Без этой недели методология работает вслепую — у нас не будет данных, где границы реально проходят.

## Глоссарий — простыми словами

Чтобы Инна понимала канон без технической каши.

| Термин | Что это | Пример из жизни |
|---|---|---|
| **Тест** | Маленькая программа, которая автоматически проверяет, что другая программа работает правильно. Запускается одной командой, проверяет много сценариев за секунды | Тест для бота записи в Beauty: «если клиент вводит правильную дату → запись создаётся; если дата в прошлом → выдаётся ошибка» |
| **Failing test** | Тест, который сначала пишут так, чтобы он не прошёл (то есть «ловил» ошибку). Потом пишут код, чтобы тест прошёл. Это TDD-подход | Перед фиксом бага «бот падает на пустом сообщении» сначала пишем тест «отправь боту пустую строку → ожидаем понятную ошибку, а не падение» |
| **TDD** (test-driven development) | Разработка через тесты: сначала пишем тест → потом код → потом тест проходит | Стандарт в зрелых командах; у нас в vibe coding пока редко |
| **Smoke test** | Ручная быстрая проверка «работает ли в принципе»: открыть страницу, нажать кнопки, посмотреть глазами. Не автоматический | Для лендинга: открыть на телефоне, проверить адаптив, ссылки, CTA-кнопку — это smoke test |
| **Воспроизводимый сценарий** | Текстом описанные шаги: «введи это, нажми сюда, должно получиться вот это». Не код, а инструкция, по которой любой может повторить | «Открой бота, отправь /start, введи имя 'Анна', выбери дату 25.05 — должна появиться запись в CRM» |
| **Edge case** (крайний случай) | Необычная ситуация на границе допустимого: пустой ввод, очень длинный текст, неверный токен, отсутствие интернета, чужой пользователь | Для формы записи: что если ввести 100 пробелов? Что если выбрать дату 31 февраля? |
| **Гипотеза корня** | Предположение, почему баг происходит. Формулируется до фикса: «думаю, причина в X, потому что Y» | «Бот падает на эмодзи в имени, потому что мы не обрабатываем юникод в функции X» |
| **Code review** | Чтение кода другим человеком/агентом перед сдачей: «нет ли утечки секретов? уязвимостей? нелогичных мест?» | QA-агент с скиллом `code-review` смотрит на коммит до деплоя |
| **Refactor** | Переписать код, не меняя его поведения — чтобы было чище/быстрее/понятнее | Большая функция на 200 строк разбита на 5 маленьких — снаружи работает так же, внутри понятнее |
| **Diff** (`git diff`) | Различие между текущими файлами и предыдущей версией: «что именно изменилось» | Перед коммитом смотрим diff — нет ли там случайно `.env`, токенов, мусора |
| **Мини-режим** (в Superpowers) | Минимальная дисциплина для маленькой задачи: 5 строк спеки перед правкой | Поправить хедер на лендинге Светы — заполнить мини-спеку и сделать |
| **Полный режим** (в Superpowers) | Полная дисциплина для серьёзного продукта: brief → план → тесты → реализация → QA → деплой | Новый бот лидогенерации для Южнобережного — полный цикл |

## Главный принцип

Superpowers нужен не ради бюрократии. Он включается там, где цена ошибки выше цены короткого плана.

Для Инны это означает: меньше переделок, меньше случайного кода, больше проверяемых результатов, которые можно передавать клиенту или тиражировать как продукт.
