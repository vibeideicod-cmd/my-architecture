# Playbook: Как мы создаём цифровой продукт

**Версия:** 0.1 (2026-04-14)
**Статус:** первая рабочая версия, обкатывается на МГ-платформе задним числом.
**Источники:** уроки Нейроуниверситета (5 шагов, CRAFT+3 файла, Урок 11, Урок 12) + брендбук Инны + аудит текущих агентов.

Этот документ — **единственный источник правды о том, как команда `my-architecture` создаёт любой цифровой продукт**. Если агенты и другие документы противоречат Playbook — исправляем их, а не Playbook. Если Playbook противоречит уроку Инны — см. [feedback-lessons-strengthen-not-replace.md](../../.claude/projects/-Users-innaandreychenko-Documents-Projects-my-architecture/memory/feedback-lessons-strengthen-not-replace.md) (уроки усиливают, не заменяют).

## 0. Три рамки, которые работают во всех проектах

### 0.1. Шесть фаз жизненного цикла (из `agents/director.md`)

| # | Фаза | Что происходит |
|---|---|---|
| 1 | **Discovery** | Понять, что именно строим и зачем |
| 2 | **Research** | Собрать данные: рынок, конкуренты, аудитория, референсы |
| 3 | **Design / Plan** | Чертёж: экраны, архитектура, brief.md, BACKEND-PLAN.md |
| 4 | **Build** | Реализация на mock или тестовых данных |
| 5 | **Content** | Замена заглушек на реальный контент |
| 6 | **QA + Deploy** | Проверка, TESTING.md, DEPLOYMENT.md, публикация |

### 0.2. Пять шагов product-builder (из урока «GPS-навигатор»)

Вложены внутрь Фаз 1–3:

1. **Идея** — формулировка «Я создаю ЧТО для КОГО чтобы решить КАКУЮ проблему»
2. **Исследование** — 5 направлений: конкуренты, функции, UX, технологии, монетизация
3. **Критика** — «Раскритикуй» глазами пользователя, UX, конкурента + «Ты уверен?»
4. **План** — brief.md с экранами, BACKEND-PLAN.md если нужен бэкенд
5. **Реализация** — передача Кодычу/Системщику в Фазу 4

### 0.3. Универсальные правила (применяются на любом шаге)

- **CRAFT-формула** — каждый промпт должен содержать Context, Role, Action, Format, Tone. См. [craft-formula.md](../prompting/craft-formula.md).
- **Один промпт — одна задача** — для точечных правок. Формула: «Что именно / Где именно / Что не трогать».
- **Техника «Раскритикуй»** — после любого готового куска работы. См. [critique-techniques.md](../prompting/critique-techniques.md).
- **«Ты уверен?»** — на ключевых утверждениях Клода. Не на каждом слове, а на решениях.
- **Брендбук Инны** — для внутренних продуктов бренда «Нейро Бабки СССР» — только светлая нейтральная палитра. См. [design-system.md](../standards/design-system.md) и [feedback-nb-light-palette.md](../../.claude/projects/-Users-innaandreychenko-Documents-Projects-my-architecture/memory/feedback-nb-light-palette.md).
- **Хостинг** — Beget целевой, Vercel только для тестов. В начале каждого проекта Директор задаёт вопрос. См. [feedback-deploy-vercel-or-beget.md](../../.claude/projects/-Users-innaandreychenko-Documents-Projects-my-architecture/memory/feedback-deploy-vercel-or-beget.md).

## 2. Универсальный процесс по фазам

Это путь, который проходит **любой** продукт. Специфику по типу смотри в Разделе 3.

### Фаза 1 — Discovery (Идея)

**Кто ведёт:** Директор → product-builder
**Документ на выходе:** `clients/<project>/idea.md`

**Шаги:**

1. **Директор активирует product-builder** — не пропускает эту фазу, даже если «всё очевидно»
2. **product-builder задаёт Инне вопросы — один заход, не дробить:**
   - Тип продукта? (лендинг / Mini App / калькулятор / квиз / веб-приложение / бот / другое)
   - Формулировка: «Я создаю ЧТО для КОГО чтобы решить КАКУЮ проблему»
   - Целевая аудитория: кто эти люди, чего они хотят, чего боятся
   - Сразу на Beget или через Vercel для тестов?
   - Есть ли дедлайн / срочность?
   - Что есть из готового: тексты, логотип, фото, референсы?
3. **product-builder собирает `idea.md`** со структурой:
   ```
   # Идея: [название]

   ## Формулировка
   Я создаю [ЧТО] для [КОГО] чтобы решить [КАКУЮ проблему].

   ## Для кого (ЦА)
   ...

   ## Какую проблему решаем
   ...

   ## Главный результат для пользователя
   Что человек получит, когда продукт сработает.

   ## Тип продукта и стек
   Из матрицы Раздела 1.

   ## Хостинг
   Beget / Vercel (временно) — и почему.

   ## Чего НЕ будет в MVP
   Осознанно выкинутое — самый важный раздел, без него проект не заканчивается.
   ```
4. **Показать Инне формулировку. Дождаться подтверждения.**
5. **Обновить `clients/<project>/PLAN.md` — Фаза 1 ✅**

**Критерий выхода:** формулировка идеи подтверждена Инной.

**Красный флаг:** если идея звучит как «сделать X» вместо «помочь Y решить Z» — остановиться и переформулировать.

**Для технических продуктов (бот, веб-приложение с БД, Mini App с бэкендом, интеграция):** после базовых 6 вопросов product-builder обязан пройти прочёс по 8 доменам из [knowledge/methodology/discovery-signals.md](../methodology/discovery-signals.md) и убедиться, что в ответах нет 🚩-сигналов пробелов (hedging-язык, размытая ЦА, конфликт требований). Для простых лендингов и визиток — этот шаг пропускается.

### Фаза 2 — Research (Исследование)

**Кто ведёт:** analyst (под Директором)
**Документ на выходе:** `clients/<project>/research.md`

**Пятинаправленный промпт** (из урока и из `agents/analyst.md`):

```
Ты — маркетинговый аналитик и UX-исследователь.
Проведи исследование для проекта: [вставить формулировку идеи из idea.md].

1. КОНКУРЕНТЫ — найди 5-7 похожих продуктов.
   Для каждого: название, что умеет, плюсы, минусы, что стоит брать/избегать.

2. ФУНКЦИИ — какие функции обычно есть в подобных продуктах?
   Список из 15-20. Отметь: что для MVP, что потом.

3. UX И СЦЕНАРИИ — как пользователь обычно взаимодействует?
   Опиши 3 типичных сценария: «человек пришёл → ... → получил результат».
   Что удобно, что бесит.

4. ТЕХНОЛОГИИ — какие технологии обычно используют?
   Сравни 2-3 варианта для нашего случая: плюсы, минусы, стоимость.

5. МОНЕТИЗАЦИЯ — как подобные проекты зарабатывают?
   Подписка, комиссия, реклама, партнёрства, freemium.

Результат — Markdown-файл research.md с пятью разделами.
Реальные ссылки на конкурентов, реальные примеры. Никакой отсебятины.
```

**Критерий выхода:** пять разделов в `research.md`, Инна прочитала и согласилась двигаться дальше.

### Фаза 2.5 — Critique (Критика)

**Кто ведёт:** product-builder + техника «Раскритикуй»
**Документ на выходе:** `clients/<project>/critique.md`

**Шаги:**

1. **Промпт «Глазами пользователя»** — см. [critique-techniques.md](../prompting/critique-techniques.md)
2. **Промпт «Глазами UX-дизайнера»**
3. **Промпт «Глазами конкурента»** — «что бы ты сделал лучше, чтобы переиграть наш продукт»
4. **Вопрос «Ты уверен?»** на ключевых утверждениях из research.md
5. **Собрать список правок** в `critique.md`:
   - Что добавить
   - Что убрать
   - Что поменять в формулировке
   - Что заложить в MVP, что отложить

**Критерий выхода:** список правок есть, внесены в idea.md/research.md, Инна согласна.

### Фаза 3 — Design / Plan (Чертёж)

**Кто ведёт:** product-builder → передаёт websites или systems
**Документы на выходе:** `clients/<project>/brief.md` + опционально `BACKEND-PLAN.md`

#### brief.md — чертёж фронтенда

Структура:
```
# Brief: [название]

## Экраны / страницы
1. [Название экрана] — цель, что на нём, куда ведут кнопки
2. ...

## Навигация
Схема переходов между экранами.

## Дизайн-система
Ссылка на design-system.md + уточнения под проект.

## Контент
Что реальное, что пока заглушка, откуда брать реальное.

## Функции
Список того, что должно работать в MVP.

## Чего НЕ делаем в MVP
Повторяем из idea.md для фокуса.
```

#### BACKEND-PLAN.md — только если нужен бэкенд

Структура (из Урока 12):
```
# Backend Plan: [название]

## Ключевая задача
Описание бизнес-цели бэкенда.

## Данные
Какие сущности хранить, какие поля, какие связи.

## Роли и доступ
Кто что видит, кто что редактирует.
(Для мульти-тенанси — как разделяем данные между тенантами.)

## API
Какие эндпоинты нужны, что принимают, что возвращают.

## RLS / безопасность
Политики Supabase — кто что может читать/писать.

## Realtime
Какие таблицы подключаем к supabase_realtime.

## Чего НЕ делаем в MVP
```

**Критерий выхода:** Кодыч (websites) или Системщик (systems) читает brief.md + BACKEND-PLAN.md и может стартовать без уточняющих вопросов. Если у него есть вопросы — Фаза 3 не закончена, возвращаемся.

### Фаза 4 — Build (Реализация)

**Кто ведёт:** websites (фронт) или systems (бэкенд)
**Документы на выходе:** код + `CLAUDE.md` в корне проекта = карта проекта

**Правила:**

- **CRAFT на каждом промпте** — Context, Role, Action, Format, Tone
- **One-shot промпт старта** — взять из библиотеки:
  - Лендинг → [templates-landing.md](../prompting/templates-landing.md)
  - Mini App → [templates-mini-app.md](../prompting/templates-mini-app.md)
  - Бэкенд/API/Supabase → [templates-backend.md](../prompting/templates-backend.md)
- **После стартового one-shot — только мелкие итерации.** Один промпт — одна правка. Формула: «Что именно / Где именно / Что не трогать».
- **Проверка в браузере** — Chrome DevTools Device Toolbar, экраны 375/390/768/1440.
- **Для Mini App** — мобильный 320-430px, кнопки ≥44px, шрифт ≥14px, без горизонтального скролла.
- **Брендбук Инны** для внутренних продуктов. Для клиентских — палитра клиента.
- **Никаких внешних фреймворков** без явного согласия (react, tailwind, bootstrap — нет).
- **CLAUDE.md** обязателен: карта «какой файл за что отвечает, где менять данные, как связаны экраны».

**Критерий выхода:** MVP работает локально + в браузере с тестовыми данными. Все экраны из brief.md реализованы. CLAUDE.md готов.

---

### Фаза 5 — Content (Наполнение)

**Кто ведёт:** content (копирайтер) или Инна сама вставляет
**Документ на выходе:** реальный контент в коде/базе

**Шаги:**
1. Заменить все заглушки («Lorem ipsum», «Мастер Анна», «услуга 1») на **реальные данные** из брифа/от клиента.
2. Проверить тексты: простой язык, без канцелярита, на языке аудитории.
3. Применить правила из `agents/content.md` и `knowledge/standards/post-types.md` если это контент-продукт.
4. Если нет реальных данных — **не придумывать**. Оставить TODO-placeholder и пометить в PLAN.md «ждём контент от Инны/клиента».

**Критерий выхода:** заглушек нет, контент прошёл самопроверку.

### Фаза 6 — QA + Deploy

**Кто ведёт:** qa → deployer-beget
**Документы на выходе:** `TESTING.md`, `DEPLOYMENT.md`

#### 6.1. QA — техническая проверка

- Чеклист по типу продукта (из `agents/qa.md`)
- Автоматические проверки: `curl` на деплой, `grep` палитры, smoke-тесты форм
- Критика «Раскритикуй» ещё раз — глазами пользователя

#### 6.2. TESTING.md — для внешнего тестировщика

Промпт (из Урока 12):
```
Ты — старший QA-инженер.
Изучи весь код и документацию проекта.
Создай TESTING.md — руководство для внешнего тестировщика.
Незнакомый разработчик должен прочитать его и без вопросов
запустить, протестировать и исправить проект.
Только реальные данные из кода — ничего не придумывай.
```

TESTING.md существует, чтобы Инна могла передать продукт внешнему человеку (фрилансеру, коллеге, тестировщику) без часа объяснений по переписке.

#### 6.3. Тест на живых людях (из Урока 12)

1. Отправить ссылку **не другу, а потенциальному клиенту из ЦА**
2. Попросить пройти весь путь: открыть → посмотреть → попробовать сделать целевое действие
3. Два вопроса: **«Что непонятно?»** и **«Где застряли?»**
4. Ответы — в `clients/<project>/feedback-<дата>.md`

Каждый ответ — данные, а не критика. Три человека сказали одно и то же → исправить в первую очередь.

#### 6.4. DEPLOYMENT.md — памятка для себя

Промпт:
```
Ты — DevOps-инженер.
Прочитай код и документацию проекта.
Создай DEPLOYMENT.md — пошаговую инструкцию по деплою для новичка.
Только то, что нужно для этого проекта. Реальные команды, реальные пути.
```

Пригодится через полгода, когда Инна или коллега вернутся к проекту.

#### 6.5. Публикация

- **Beget (целевой):** через `deploy-<project>.sh` (по образцу `deploy-yub.sh`, `deploy-mg.sh`)
- **Vercel (временный):** Root Directory = нужная подпапка, HTTPS ссылка автоматически
- **Beget Cloud VPS (для ботов 24/7):** SSH по IP, systemd-сервис, настройка webhook
- **Домен (если нужен):** `beget.com` → Домены → Регистрация → подключение через DNS

**Критерий выхода:** живая ссылка открывается, все три теста проходят (см. Урок 12, раздел «Финальная проверка»).

## 3. Специфические шаги по типам продуктов

Расширения поверх универсальных фаз.

### 3.1. Mini App Telegram

**Дополнительно в Фазе 3 (Plan):**
- В brief.md указать: мобильные размеры (320-430px), тема Telegram, акцент `#2AABEE` если стиль нативный Telegram, наш брендбук если стиль фирменный.
- Решить: нужен ли `screen-offer` для `/start`-капчи (обычно да).

**Дополнительно в Фазе 4 (Build):**
- Папка `tg-app/`, точка входа `tg-app/index.html`
- Подключить Telegram Web App SDK: `<script src="https://telegram.org/js/telegram-web-app.js"></script>` первым скриптом в `<head>`
- Использовать `tg.initData` для получения пользователя, `tg.themeParams` для темы
- `MainButton`, `BackButton`, `HapticFeedback` через `window.Telegram.WebApp`
- Адаптация: `clamp()`, мобильный first

**Дополнительно в Фазе 5 (Content):**
- Имя пользователя через `tg.initDataUnsafe.user.first_name` — но **обрезать после `|`** (BUG-002 Beauty, см. session-log)

**Дополнительно в Фазе 6 (QA + Deploy):**

1. **BotFather flow:**
   - @BotFather → `/newbot` → имя → username (заканчивается на `_bot`) → получить токен
   - `.env` + `.env.example` + `.gitignore` — обязательная тройка (см. `templates-mini-app.md`)
   - Токен **никогда** в код, только в `.env`
2. **Деплой:**
   - Тест: Vercel, Root Directory = `tg-app/`, HTTPS автоматически
   - Прод: Beget через `deploy-<project>.sh`
3. **Подключение приложения к боту — оба способа:**
   - **Menu Button** (внутри бота): BotFather → `/mybots` → бот → Bot Settings → Menu Button → Configure → вставить HTTPS → текст кнопки
   - **Mini App link** (прямая ссылка): BotFather → Bot Settings → Mini Apps → Enable → вставить HTTPS
   - Результат: `t.me/<bot_username>/app` — прямая ссылка для распространения
4. **Screen-offer для `/start`-капчи** (см. `templates-mini-app.md`):
   - Модалка на первом открытии, `localStorage` «показать один раз»
   - Кнопка ведёт на `https://t.me/<bot_username>?start=from_app`
   - Зачем: чтобы бот получил право писать первым (без `/start` права нет)
5. **One-shot настройка бота** через Bot API (см. `templates-mini-app.md`):
   - `setMyDescription` — 2-3 предложения: что / зачем / куда нажать
   - `setMyShortDescription` — одно предложение
   - `setMyCommands` — `/start`, `/help`, контакт автора

## 5. Логирование прогресса в каждом проекте

Каждый `clients/<project>/PLAN.md` содержит блок «Текущая фаза» и чеклист по всем фазам Playbook. Директор обновляет его при переходах.

Образец:

```markdown
# PLAN: <название проекта>

## Текущая фаза: **2 — Research**
**Что сейчас:** собираем research.md, проанализировано 3 из 7 конкурентов.
**Блокеры:** нет.
**Следующее действие:** закончить список функций, перейти к UX-сценариям.

## Чеклист

- [x] **Фаза 1 — Discovery**
  - [x] idea.md готов и подтверждён Инной
  - [x] Тип продукта: [Mini App Telegram]
  - [x] Хостинг: Vercel (тест) → Beget (прод)
- [>] **Фаза 2 — Research**
  - [>] research.md — в работе
  - [x] Конкуренты (7)
  - [ ] Функции
  - [ ] UX-сценарии
  - [ ] Технологии
  - [ ] Монетизация
- [ ] **Фаза 2.5 — Critique**
  - [ ] critique.md
- [ ] **Фаза 3 — Design/Plan**
  - [ ] brief.md
  - [ ] BACKEND-PLAN.md (если нужен бэкенд)
- [ ] **Фаза 4 — Build**
  - [ ] Код MVP
  - [ ] CLAUDE.md
- [ ] **Фаза 5 — Content**
  - [ ] Реальные данные вместо заглушек
- [ ] **Фаза 6 — QA + Deploy**
  - [ ] TESTING.md

  - [ ] DEPLOYMENT.md
  - [ ] Живая ссылка
```

Легенда: `[ ]` не начато · `[>]` в работе · `[x]` готово · `[!]` блокер · `[~]` отложено.

## 6. Правила гибкости и эволюции Playbook

### Когда обновляем Playbook

1. **Новый тип продукта** — остановились, добавили ветку в Раздел 3, потом делаем.
2. **Нашли что-то лучше** — обновили соответствующий раздел, добавили запись в Changelog.
3. **Противоречие с уроком Инны** — применяем правило «усиливаем, не копируем» (`feedback-lessons-strengthen-not-replace.md`). Берём из урока лучшее, встраиваем в наш процесс, не ломаем то, что уже работает.
4. **Провал на реальном проекте** — это первопричина большинства обновлений. После каждого провала (и после каждой победы) — короткое ревью: что не работало в Playbook.

### Правила хождения по шагам

- **Нельзя пропускать фазы.** Если очень хочется — значит, фаза недоописана. Фиксим описание, не пропускаем.
- **Каждый переход между фазами залогирован в PLAN.md** — чтобы Инна и Директор видели «где мы».
- **Один промпт — одна задача** применяется ко всему: и к коду, и к правкам документов. Не хватай три фазы одним махом.
- **Директор обязан называть фазу** в начале каждого ответа по клиенту: «МГ — Фаза 2 (Research), работаю над research.md». Это убирает ощущение «нет плана».

### Что НИКОГДА не делаем

- Не пишем код до Фазы 4.
- Не придумываем данные о клиенте/Инне — только то, что сказали.
- Не пропускаем критику ради скорости.
- Не меняем хостинг на Vercel инициативно — только с явным ответом Инны.
- Не добавляем фичи из «а давайте ещё вот это» без обсуждения с Инной — это раздувает MVP.
